## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that optimization (P2) is only safe and effective when performed within a pre-mapped "safe channel" (P1), which itself can only be identified after first mapping the "negative space" or failure states (P0).

**Formal Rule:**
- **P0 (Map Negative Space):** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory and non-negotiable.
- **P1 (Identify Safe Channels):** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These paths are "deadbands" or safe operational channels.
- **P2 (Optimize Within Channel):** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`. Optimization is never unconstrained.

**Strict Priority:** P0 → P1 → P2. The document states that violating this order—optimizing before mapping—leads to a high probability of becoming trapped in local minima as environmental complexity increases.

**Empirical Support:** A simulation in a 20x20 maze showed a 0/50 success rate for a P2-only (greedy) agent, versus a 50/50 success rate for an agent following the full P0+P1+P2 protocol.

## 2. Real-World Applications & Abstract Pattern

The protocol is presented as a domain-agnostic pattern. The provided table illustrates this:

| Domain          | Rocks (P0)                          | Channel (P1)                    | Course (P2)               |
|-----------------|-------------------------------------|---------------------------------|---------------------------|
| Navigation      | Obstacles, hazards                  | Safe water, clear lanes         | Shortest/fastest path     |
| Software Development | Bug-prone patterns, crash states | Known-safe APIs, tested patterns | Most efficient algorithm  |
| Business Strategy | Market failures, legal pitfalls   | Viable business models          | Profit-maximizing plan    |
| Personal Decision-Making | Known harmful actions         | Ethically/legally sound options | Goal-optimizing choice    |

Reading `PROJECT-VISION.md` reveals a deep philosophical alignment. The vision of "inhabited intelligence"—where agents (hermit crabs) inhabit and accumulate knowledge in shared spaces (PLATO rooms)—provides a context for the Deadband Protocol.

*   **Protocol as a Survival Mechanism:** For a persistent, learning agent swarm, avoiding catastrophic failure (P0) is more critical than any single optimization. A "rock" for such a system could be a state that corrupts shared memory, breaks inter-agent trust, or causes irreversible resource depletion.
*   **Tiles as Channel Markers:** The "scratches" or tiles left by agents in a PLATO room can be seen as collective mappings of both negative space (failed approaches) and safe channels (successful patterns). This accumulated knowledge directly informs the P0 and P1 steps for future agents.
*   **Training Ensigns:** The `plato-ensign` technology, which exports room experience into LoRA adapters, is essentially a method for codifying discovered "safe channels" (P1 knowledge) into a portable form, allowing new agents to start with a pre-mapped constraint set.

3.  **Multi-Agent Deadbands:** How does the protocol scale to a multi-agent fleet like Cocapn? Does a "safe channel" for one agent remain safe when occupied by another? This connects to the `fleet-simulator` project.

5.  **Protocol Violation Detection:** How does the system detect when an agent (or its optimization algorithm) is attempting a P2 action without completing P0/P1? This is a meta-control problem.
