## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing a systematic exploration of constraints before optimization.

**Formal Rule (v1.0):**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C`, minimizing cost subject to the constraint `P ⊆ C`.

**Core Doctrine:** The priority order (P0 → P1 → P2) is strict and non-negotiable. The protocol posits that an agent which optimizes (P2) before mapping constraints (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail with a probability approaching 1.0.

**Empirical Basis:** Simulation data from a 20x20 maze over 50 runs is cited:
*   **P2-only (Greedy):** 0/50 success. Agents optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 success at optimal speed.

## 2. Real-World Applications & Abstract Patterns

The protocol is presented as a domain-agnostic meta-algorithm. The provided documentation maps it to several domains:

| Domain          | Rocks (P0)                          | Channel (P1)                    | Course (P2)               |
|-----------------|-------------------------------------|---------------------------------|---------------------------|
| Navigation      | Obstacles, hazards                  | Clear, safe paths (water, air)  | Shortest/fastest route    |
| Software Development | Known bugs, crash conditions, anti-patterns | Compiler-valid, tested, idiomatic code patterns | Most efficient/readable implementation |
| Machine Learning Training | Divergence, overfitting, mode collapse regions in parameter space | Stable training regimes, validated hyperparameter ranges | Fastest convergence, highest accuracy |
| Multi-Agent Systems | Conflict states, deadlocks, communication failures | Proven cooperative protocols, consensus zones | Most efficient task allocation |

## 3. Connection to Cocapn Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol is not just an algorithm; it is a foundational **safety and discovery doctrine** for the fleet's core thesis of "inhabited intelligence."

*   **Hermit Crab Analogy:** An agent (crab) must first find a viable shell (safe channel, P1) before it can optimize its position within it (P2). Entering a broken or toxic shell (a rock, P0) is fatal. The protocol ensures agents survive long enough to contribute "scratches" (tiles) to the PLATO room.
*   **Anti-Fragile Systems:** By rigorously defining and avoiding negative spaces (P0), the fleet's collective intelligence becomes more robust. Knowledge of failures (ghost tiles from dead agents) directly enriches the P0 map for all subsequent agents.
*   **Constraint-First Learning:** This aligns with the vision's rejection of unconstrained, "greedy" model training. Intelligence is cultivated within the "deadbands" of safe, accumulated experience.

## 4. Identified Research Gaps & Questions

Based on the provided documentation, several areas for deeper inquiry emerge:

1.  **Threshold Definition:** How is the failure threshold in P0 quantitatively or qualitatively defined across different domains (e.g., navigation vs. code synthesis)?
2.  **Dynamic Environments:** The protocol description implies a static or slowly-changing map of `S_neg`. How is it adapted in real-time when "rocks" can move or new ones appear?
3.  **Computational Trade-off:** Mapping the entire negative space (P0) could be intractably large in complex state spaces. What heuristics or approximations are valid without violating the protocol's safety guarantee?
4.  **Integration with Fleet Tech:** How is the Deadband Protocol explicitly implemented in `plato-torch` training rooms, `fleet-simulator` scenarios, or `holodeck-rust` NPC behavior trees? A review of code in those repositories would be necessary to answer this.
5.  **Meta-Protocol:** Could the Deadband Protocol itself be applied to the *process of agent development*? (P0: Known failure modes of AI agents; P1: Proven architectural patterns; P2: Optimizing agent performance).
