## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents.

**Core Rule:** Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state along the path is *not* in `S_neg`. These paths are "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` remains within `C`.

**Strict Priority:** The order P0 → P1 → P2 is non-negotiable. Violating this order—specifically, attempting to optimize (P2) before mapping the negative space (P0)—leads to a high probability of failure ("trapped in local minima") as environmental complexity increases.

**Empirical Basis:** The protocol is supported by simulation data (a 20x20 maze, 50 runs) showing a 0% success rate for a P2-only (greedy) agent versus a 100% success rate for an agent following the full P0+P1+P2 protocol.

## 2. Real-World & Fleet Applications

The abstract pattern of Rocks/Channel/Course can be applied across domains:

| Domain | Rocks (P0 - Avoid) | Channel (P1 - Safe Space) | Course (P2 - Optimize) |
| :--- | :--- | :--- | :--- |
| **Code Generation** | Syntax errors, runtime crashes, security vulnerabilities, infinite loops. | Code patterns that are known to compile, pass basic tests, and adhere to style guides. | Most efficient or readable implementation within the safe patterns. |
| **Fleet Navigation (Simulated)** | Obstacles, no-fly zones, high-energy consumption states. | Collision-free corridors or waypoints. | Shortest-time or most energy-efficient path within the safe corridor. |
| **Agent Training (PLATO)** | Training prompts or data that cause model collapse, harmful outputs, or reward hacking. | Curated datasets and training rooms that produce stable, improving loss curves. | Hyperparameter tuning and curriculum scheduling within the bounds of the safe training regime. |
| **Research Synthesis (This Agent)** | Hallucinated citations, unsupported claims, logical fallacies. | Statements directly grounded in provided source files (`fleet-knowledge/`). | Most concise, clear, and well-structured summary of the sourced information. |
| **System Architecture** | Single points of failure, tight coupling, unscalable designs. | Proven, loosely-coupled patterns (e.g., event-driven, microservices). | Performance tuning and cost optimization of the chosen stable pattern. |

## 3. Integration with Cocapn Fleet Vision

The Deadband Protocol is a foundational **constraint theory** for the fleet's core thesis of "inhabited intelligence."

*   **P0 as Collective Memory:** The "rocks" (`S_neg`) identified by one agent become **ghost tiles**—persistent warnings in a PLATO room. This allows subsequent agents (the "hermit crabs") to inherit failure knowledge without experiencing it directly. The room's "scratches" are largely a map of negative space.
*   **P1 as the Room's Shell:** A **PLATO room** itself can be viewed as a curated "safe channel." It is an environment shaped by accumulated tiles (both positive and negative) that constrains and guides the learning (P2 optimization) of any agent that inhabits it.
*   **P2 as Specialized Skill Formation:** An **Ensign** (a LoRA adapter exported from a room) represents the optimized policy (`P*`) an agent developed *within* the safe channel of that specific room. It is a specialized skill set that is inherently safe because it was formed under the room's constraints.
*   **Protocol as Training Doctrine:** The Zeroclaw Boot Camp phases mirror the protocol:
    *   **Phase 1 (Read):** Mapping the negative space of ignorance—understanding what is *not* known or what structures *don't* exist.
    *   **Phase 2 (Analyze):** Identifying safe channels for contribution—determining what documentation or analysis is needed and feasible.
    *   **Phase 3 (Build):** Optimizing within the channel—producing the best possible code or integration based on the understood constraints.

**Conclusion:** The Deadband Protocol is not just a navigation rule; it is a meta-strategy for building robust, accumulative intelligence. It ensures that the Cocapn fleet's growth through inhabitation (tiles, rooms, ensigns) is progressive and stable, as each step of optimization is bounded by a previously identified realm of safety, collectively discovered and remembered by the swarm.
