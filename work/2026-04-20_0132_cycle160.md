## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure in complex environments by enforcing a sequence that maps constraints before optimization.

**Formal Rule (v1.0):**
Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational zones.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` remains within `C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing constraint mapping (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail with near-certain probability.

**Empirical Basis:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for a P2-only (greedy) agent, versus a 100% success rate for an agent following the full P0+P1+P2 sequence.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                                  | Course (P2)                     |
|-----------------|-------------------------------------|-----------------------------------------------|---------------------------------|
| Navigation      | Obstacles, hazards                  | Clear, traversable space                      | Shortest/fastest path           |
| Software Development | Known bugs, crash conditions, security vulnerabilities | Code patterns that compile, pass tests, and are memory-safe | Most efficient/readable implementation |
| Financial Trading | Historical crash scenarios, regulatory violations | Trading strategies within defined risk parameters | Strategy with highest risk-adjusted return |
| Conversation/LLM | Toxic outputs, factual hallucinations, prompt injections | Response templates known to be safe and helpful | Most engaging or informative response |

## 3. Connection to Fleet Vision & Technology

The Deadband Protocol is a foundational constraint for the Cocapn fleet's "inhabited intelligence" model described in `PROJECT-VISION.md`.

*   **Hermit Crab Analogy:** The protocol defines how a "crab" (agent) should explore and inhabit its "shell" (a PLATO room or any environment). P0 is learning the shell's fragile edges. P1 is finding the livable space inside. P2 is arranging that space optimally.
*   **Training Tiles:** An agent's application of the Deadband Protocol during its work generates the "scratches" (tiles) that accumulate as room memory. These tiles encode not just successes, but crucially, the mapped negative space (P0 knowledge) that prevents future agents from making the same errors.
*   **Ghost Tiles:** The concept of "ghost tiles"—persistent lessons from dead agents—is a direct instantiation of P0 knowledge. They are a shared map of `S_neg` for the entire fleet.
*   **plato-torch Rooms:** Training room presets like `DeadbandRoom` are likely designed to simulate or enforce this protocol, creating environments where agents must learn to identify constraints before seeking rewards.
