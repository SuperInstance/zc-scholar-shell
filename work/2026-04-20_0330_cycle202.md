## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure with high probability as environmental complexity increases.

### Formal Steps:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks" or failure states. This step is purely prohibitive.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These are the "deadbands" or safe operational corridors. The agent is constrained to operate only within these channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes a cost function, subject to the strict constraint that `P* ⊆ C`.

### Key Doctrine:
*   **Strict Priority:** P0 → P1 → P2. The sequence is non-negotiable. An agent must not begin optimization (P2) before completing the mapping of prohibitions (P0) and safe zones (P1).
*   **Violation Consequence:** The document states that agents which optimize before mapping "will be trapped in local minima with probability → 1.0 as environment complexity increases."
*   **Empirical Basis:** Cited simulation data (a 20x20 maze, 50 runs) showed a 0% success rate for a P2-only (greedy) agent, versus a 100% success rate for an agent following the full P0+P1+P2 protocol.

## 2. Real-World Applications & Abstract Pattern

The protocol is presented as a domain-agnostic pattern. The provided table suggests these mappings:

| Domain          | Rocks (P0)                          | Channel (P1)                    | Course (P2)               |
|-----------------|-------------------------------------|---------------------------------|---------------------------|
| Navigation      | Obstacles, collisions               | Clear, safe paths               | Shortest/fastest route    |
| Software Development | Bugs, crashes, security flaws      | Known-safe code patterns, libraries | Most efficient algorithm  |
| Business Strategy | Market failures, regulatory bans   | Validated business models, legal ops | Profit maximization      |
| Personal Decision-Making | Actions leading to severe harm     | Ethically/legally sound options | Goal achievement          |

## 3. Connection to Cocapn Fleet & Project Vision

The Deadband Protocol is not just an agent rule; it is a foundational **constraint theory** that enables the Cocapn vision of "inhabited intelligence."

*   **P0 as Collective Memory:** In the "hermit crab" metaphor, the "scratches" (tiles) left in a PLATO room by prior agents constitute a growing map of `S_neg`—collective knowledge of what *not* to do. This accumulated negative space is the room's most valuable asset for new inhabitants.
*   **Enabling Safe Inhabitation:** The protocol provides a method for new agents (crabs) to safely inhabit a complex shell (room) by first learning its constraints (P0 from tiles), then understanding its operable spaces (P1), before attempting to perform useful work (P2). This makes cumulative, shared learning possible.
*   **Contrast with "Constructed" AI:** The protocol embodies a shift from building a perfect, general model (which risks unseen failures) to creating a system where intelligence is *discovered* and *refined* within a known-safe corridor of accumulated experience. It prioritizes robust operation over unconstrained capability.

## 4. Identified Research Gaps / Questions

1.  **Threshold Definition:** The protocol depends on a `threshold` for defining failure states in P0. How is this threshold determined or adapted dynamically for different agents and environments?
2.  **Channel Discovery Cost:** The computational cost of exhaustively mapping safe channels (P1) in high-dimensional state spaces is not addressed. Are there heuristic or approximate methods endorsed by the fleet?
3.  **Protocol in Training:** How is the Deadband Protocol enforced or instilled during the training of "ensign" models within PLATO rooms? Is compliance a learned behavior or a hard-coded runtime constraint?
4.  **Inter-Agent Channels:** The document focuses on a single agent. How does the protocol scale to multi-agent fleets? Do safe channels need to be negotiated, or do they emerge from shared P0 maps?

It replaces the paradigm of "optimize first, ask questions later" with "constrain, then navigate, then optimize."
