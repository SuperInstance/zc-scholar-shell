## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping failure states (P0) and safe operational channels (P1) leads to a high probability of catastrophic failure ("hitting rocks") as environmental complexity increases.

**Formal Steps:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` within the path is *not* in `S_neg`. These are the "deadbands" or safe operational corridors.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C`, minimizing cost subject to the constraint `P* ⊆ C`.

**Key Principle:** The priority order P0 → P1 → P2 is strict. Simulation data (20x20 maze, 50 runs) cited in the document shows a 0% success rate for P2-only (greedy) agents versus a 100% success rate for agents following the full P0+P1+P2 protocol.

## 2. Real-World Applications (Abstract Pattern)

The protocol presents an abstract pattern applicable to any agentic domain. The following table extrapolates from the examples given in the source document:

| Domain | Rocks (P0 - What NOT to do) | Channel (P1 - Where you CAN be) | Course (P2 - Best path) |
| :--- | :--- | :--- | :--- |
| **Navigation** | Obstacles, no-fly zones, hazardous terrain. | Clear lanes, mapped roads, safe air/water corridors. | Shortest, fastest, or most fuel-efficient route within safe channels. |
| **Software Development** | Code patterns that cause bugs, crashes, or security vulnerabilities; unstable dependencies. | Known-safe libraries, design patterns, and API usage conventions. | Most efficient, readable, or performant implementation using safe patterns. |
| **Machine Learning Training** | Hyperparameter sets that cause divergence; data distributions that induce bias; model architectures prone to overfitting. | Validated training regimes, robust data augmentation pipelines, stable architectural templates. | Tuning for highest accuracy or lowest loss within the stable regime. |
| **Business Strategy** | Markets with insurmountable barriers, illegal activities, proven failed business models. | Validated customer segments, compliant operational frameworks, proven revenue channels. | Optimizing for growth, profit, or market share within the legal and viable space. |
| **Conversational AI (This Agent)** | Making absolute claims ("always", "never", "guaranteed"); hallucinating information not in context; skipping P0 analysis. | Stating findings based on provided source files; identifying knowledge gaps; operating within documented fleet doctrine. | Producing the most concise, useful, and well-structured synthesis of the source material. |

## 3. Connection to Cocapn Fleet Doctrine

The Deadband Protocol is not just a theoretical rule; it is the **operational doctrine** for the Cocapn fleet, as stated in the "Fleet Doctrine" preamble. This aligns with the project's vision of "Inhabited Intelligence." Agents ("hermit crabs") learn by inhabiting shared spaces (PLATO rooms), where their work outputs become "tiles" (scratches on the shell). The protocol ensures that this accumulation of knowledge is structured and safe, preventing agents from repeatedly learning catastrophic failures. The "ghost tiles" concept—persisting lessons from dead agents—serves as a collective P0 map for the entire swarm, a direct implementation of the protocol's first and most critical step.
