## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid failure states before attempting optimization.

**Core Rule (Formal):**
Given an agent A in state S, with goal G, in environment E:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` is a subset of `C`.

**Strict Priority:** The steps must be executed in order (P0 → P1 → P2). The document states that an agent which optimizes (P2) before completing the mapping steps (P0+P1) will, with high probability, become trapped in local minima or fail as environmental complexity increases.

**Empirical Basis:** The protocol is supported by simulation data from a 20x20 maze environment. In 50 runs, a "P2-only" greedy agent succeeded 0 times, while a "P0+P1+P2" agent following the Deadband Protocol succeeded 50 times.

## 2. Abstract Pattern & Real-World Applications

The protocol presents itself as a domain-agnostic pattern. The provided table offers two examples:

| Domain          | Rocks (P0)                          | Channel (P1)                    | Course (P2)               |
|-----------------|-------------------------------------|---------------------------------|---------------------------|
| Navigation      | Obstacles (walls, cliffs)           | Safe water/passable terrain     | Shortest/quickest path    |
| Code            | Bugs, crashes, security vulnerabilities | Known-safe code patterns, libraries | Most efficient algorithm |

**Extended Application Hypotheses:**

*   **Machine Learning Training:** P0 could involve identifying hyperparameter combinations or data regions that lead to model collapse or NaN gradients. P1 would define the stable training regime. P2 would then seek the fastest convergence within that regime.
*   **Autonomous System Design:** For a fleet agent (like a Zeroclaw), P0 might map actions that violate core constraints (e.g., writing outside its repo, making absolute claims). P1 defines the action space permitted by its role and tools. P2 is the efficient execution of its assigned task.
*   **Research Synthesis (This Role):** P0 involves identifying unreliable sources, logical fallacies, or unsupported claims in literature. P1 is the corpus of verified, peer-reviewed findings and robust experimental data. P2 is the process of constructing the most coherent narrative or insight from that trustworthy corpus.
*   **Financial Trading:** P0 identifies market conditions or asset behaviors correlated with catastrophic drawdowns. P1 defines trading strategies that remain inactive or hedged during those conditions. P2 optimizes for profit within the "safe" strategies.
