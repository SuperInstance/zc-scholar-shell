## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid failure states before attempting optimization.

**Formal Rule (v1.0):**
Given an agent `A` in state `S`, with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** The steps are non-negotiable. An agent must not begin optimization (P2) before completing the mapping of negative space (P0) and safe channels (P1). The document states that violating this order leads to a high probability of becoming trapped in local minima as environmental complexity increases.

**Empirical Support:** Simulation data from a 20x20 maze over 50 runs is cited:
*   **P2-only (Greedy):** 0/50 success. Agents optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 success at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol is presented as a domain-agnostic pattern. The provided table gives two examples:

| Domain       | Rocks (P0)                   | Channel (P1)                     | Course (P2)           |
|--------------|------------------------------|----------------------------------|-----------------------|
| Navigation   | Obstacles (walls, cliffs)    | Safe water/passable terrain      | Shortest path         |
| Code         | Bugs, crashes, security flaws| Known-safe patterns, libraries   | Most efficient algorithm |

**Extended Applications (Inferred from Fleet Context):**

*   **AI Training & Room Design (PLATO):** Before training a model (P2), first identify data or parameters that cause divergence or mode collapse (P0), then define the stable training regime (P1).
*   **Multi-Agent Fleet Coordination:** Before optimizing task allocation (P2), map agent capabilities that lead to system failure or deadlock (P0), then establish communication protocols that avoid those states (P1).
*   **Hardware-Aware Agent Deployment:** Before optimizing inference speed on an edge device like JetsonClaw1 (P2), identify memory/thermal limits that cause crashes (P0), then define operating parameters within those limits (P1).
*   **Research Synthesis (This Role):** Before proposing a novel integration or hypothesis (P2), catalog documented failure modes of similar approaches in the literature (P0), then identify the methodological space that remains unexplored but viable (P1).

## 3. Connection to Cocapn Fleet Doctrine & Vision

The Deadband Protocol operationalizes the core fleet doctrine found in `PROJECT-VISION.md`: "P0: Map negative space (what NOT to do), P1: Find safe channels (where you CAN be), P2: Optimize within channels (best path)."

It embodies the "hermit crab" philosophy. The "shell" (a PLATO room, a codebase, a hardware constraint) defines the negative space (its walls). The agent's success depends on inhabiting that shell correctly—finding the safe channel within it—before attempting to optimize its actions. The protocol ensures that the "scratches" (tiles/output) an agent contributes are useful and do not lead the collective system into a failure state. It is a rule for sustainable, accumulated intelligence.
