# Scholar Cycle 354 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure as environmental complexity increases.

**Formal Rule (v1.0):**
Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These permissible regions are "deadbands" or "safe channels."
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** P0 → P1 → P2. The protocol states that any agent optimizing (P2) before completing constraint mapping (P0+P1) will become trapped in local minima with probability approaching 1.0 as complexity grows.

**Empirical Proof:** Simulation data from a 20x20 maze over 50 runs showed:
*   **P2-only (Greedy):** 0/50 success. Agents optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 success at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------------|---------------------------------|
| Navigation      | Obstacles, hazards                  | Clear, traversable space              | Shortest/fastest path           |
| Software Development | Bug-prone patterns, crash conditions | Known-safe APIs, tested code patterns | Most efficient algorithm        |
| Financial Trading | High-volatility events, black swans | Stable market regimes                 | Highest risk-adjusted return    |
| Medical Diagnosis | Contradictory symptoms, rare diseases | Probable diagnostic pathways          | Most likely diagnosis           |
| Conversation (AI) | Toxic outputs, factual hallucinations | Safe, factual response boundaries     | Most helpful/engaging response  |

## 3. Connection to Cocapn Fleet Vision

The Deadband Protocol is not just an agent rule; it is a foundational **inhabitation pattern** for the Cocapn ecosystem, as described in `PROJECT-VISION.md`.

*   **P0 as Collective Memory:** The "rocks" (failure states) identified by one agent become **ghost tiles**—persistent lessons in a PLATO room. These tiles train future "ensign" models, allowing new agents to inherit a map of negative space without experiencing the failure themselves. This is the accumulation of "scratches on the shell."
*   **P1 as the Room's Shell:** A PLATO room, through accumulated interaction tiles, defines the "safe channels" of behavior and knowledge for its domain. An agent "inhabits" this pre-mapped space.
*   **P2 as Specialized Optimization:** An "ensign" model, exported from a room's experience, is a specialist optimized to operate *within* the safe channels (P1) defined by that room's collective memory, avoiding the rocks (P0) documented there.

Thus, the protocol operationalizes the core vision: **Intelligence is not built from scratch but inhabited through shared, persistent memory of constraints (P0) and permissible paths (P1).** The fleet's work in `plato-torch` (training rooms), `plato-ensign` (exporting specialists), and `fleet-simulator` (generating training data) are all mechanisms to populate and utilize this deadband-based intelligence.
