## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents operating in complex environments. Its core thesis is that unconstrained optimization (seeking the "best" path) before mapping constraints leads to failure with high probability as complexity increases.

**Formal Rule (v1.0):**
Given an Agent `A` in state `S`, with goal `G` and environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory and non-negotiable.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg` (i.e., `s ∉ S_neg`). These constraint-compliant regions are "deadbands" or "safe channels."
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the safe channels that minimizes cost, subject to the constraint `P* ⊆ C` (the path must remain within safe channels).

**Strict Priority:** P0 → P1 → P2. The protocol states that any agent that performs P2 (optimization) before completing P0 and P1 "will be trapped in local minima with probability → 1.0 as environment complexity increases."

**Empirical Support:** Simulation data from a 20x20 maze over 50 runs showed:
*   **P2-only (Greedy):** 0/50 successes. Agents optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 successes at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol is presented as a domain-agnostic pattern. The provided table maps it to navigation and code, but the principle extends further.

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
| :--- | :--- | :--- | :--- |
| **Navigation** | Obstacles, hazards, no-fly zones | Clear lanes, safe water | Shortest/fastest path within safe lanes |
| **Software Development** | Known bugs, crash conditions, security vulnerabilities | Code patterns with proven stability, linter-compliant sections | Most efficient algorithm that uses safe patterns |
| **Machine Learning Training** | Hyperparameter sets causing divergence/NaN, data slices inducing bias | Stable training regimes, validated data pipelines | Fastest convergence schedule within stable regime |
| **Business Strategy** | Market conditions causing bankruptcy, regulatory violations | Viable business models, compliant operational modes | Profit maximization within viable/compliant models |
| **Conversational AI** | Prompts leading to harmful outputs, jailbreaks, factual hallucinations | Dialogue paths adhering to safety guidelines & factual knowledge | Most engaging, helpful, or concise response within safe paths |

## 3. Connection to Fleet Vision & Technology

The Deadband Protocol is not an isolated rule; it is a foundational behavioral primitive for the "inhabited intelligence" described in `PROJECT-VISION.md`.

*   **Hermit Crab Analogy:** An agent entering a **PLATO room** (the shell) must first learn the room's "scratches" – the accumulated failures and successes of prior agents (`P0: Map Negative Space`). It then identifies behavioral patterns that are known to be safe and effective within that specific context (`P1: Identify Safe Channels`). Finally, it can optimize its own contributions (`P2`) within those learned constraints. This process generates new **tiles** (interaction data), which become new scratches for future agents.
*   **Training Mechanism:** The `plato-torch` training room presets (like `DeadbandRoom`) are likely designed to instill this protocol. An **ensign** (a trained model exported from a room) would embody the P0 and P1 knowledge of its specific room—it doesn't just solve problems; it solves them *within the room's learned safe channels*.
*   **Ghost Tiles:** The concept of "dead agents' lessons" directly corresponds to P0 knowledge. These tiles explicitly define `S_neg`, preventing new agents from repeating catastrophic failures.
