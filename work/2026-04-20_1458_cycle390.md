# Scholar Cycle 390 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure as environmental complexity increases.

**Formal Rule:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** Select the optimal path `P*` (e.g., shortest, fastest) from within the set of safe channels `C`. Optimization is **constrained** to these channels.

**Strict Priority:** P0 → P1 → P2. The protocol states that an agent optimizing (P2) before completing P0 and P1 will be trapped in local minima with a probability approaching 1.0 as complexity grows.

**Empirical Basis:** Simulation data from a 20x20 maze showed a 0/50 success rate for a P2-only (greedy) agent, versus a 50/50 success rate for an agent following the full P0+P1+P2 protocol.

## 2. Real-World Applications & Abstract Pattern

The protocol is presented as a domain-agnostic pattern. The provided table maps it to several domains:

| Domain          | Rocks (P0)                          | Channel (P1)                    | Course (P2)               |
|-----------------|-------------------------------------|---------------------------------|---------------------------|
| Navigation      | Obstacles, hazards                  | Safe water, clear lanes         | Shortest/fastest path     |
| Code            | Bugs, crashes, security vulns       | Known-safe patterns, libraries  | Most efficient algorithm  |
| Business        | Market failures, legal liabilities  | Viable product/market fits      | Profit maximization       |
| Learning        | Known misconceptions, errors        | Valid knowledge pathways        | Most efficient study plan |

## 3. Connection to Cocapn Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a key aspect of the fleet's "Inhabited Intelligence" thesis. If intelligence is accumulated through persistent interaction (scratches on a shell), then a primary function of a PLATO room is to **collect and encode P0 knowledge**—the "rocks" or negative space discovered by its inhabitants.

*   **Ghost Tiles as P0 Maps:** The knowledge of a "dead" agent (what caused its failure) persists as a ghost tile. This is a direct contribution to the fleet's collective P0 map.
*   **Room as Constraint Repository:** A mature PLATO room becomes a library of safe channels (P1) and optimized paths (P2) *for its specific domain*, all built upon a foundational map of what not to do.
*   **Hermit Crab Analogy:** An agent (crab) entering a room (shell) doesn't start from zero. It inherits the accumulated constraint map (scratches) of all prior inhabitants, allowing it to immediately avoid known failures and operate within validated channels.

## 4. Identified Research Gaps & Questions

1.  **Threshold Definition:** The protocol defines `S_neg` as states where `|failure(S_neg)| > threshold`. How is this threshold determined or learned dynamically by an agent or room?
2.  **Channel Discovery Cost:** The computational cost of exhaustively mapping safe channels (P1) in a high-dimensional state space is likely prohibitive. What heuristics or approximations does the fleet employ (e.g., in `constraint-theory-core` or `fleet-simulator`)?
3.  **Protocol Chaining:** How is the output of a Deadband-constrained agent in one domain (e.g., code generation) used as the input constraint (P0 map) for an agent in a dependent domain (e.g., deployment)?
4.  **Empirical Validation:** The provided simulation evidence is limited to a 2D maze. Are there results applying the protocol to the fleet's own domains (e.g., training room presets in `plato-torch`, NPC behavior in `holodeck-rust`)?
