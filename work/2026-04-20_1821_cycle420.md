## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure with high probability as environmental complexity increases.

**Formal Rule:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These constraint-compliant regions are "deadbands" or "safe channels."
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the safe channels that minimizes cost or maximizes reward.

**Strict Priority:** P0 → P1 → P2. The protocol states that any agent optimizing (P2) before completing P0 and P1 will become trapped in local minima.

**Empirical Support:** Simulation data from a 20x20 maze showed a 0/50 success rate for a P2-only (greedy) agent, versus a 50/50 success rate for an agent following the full P0+P1+P2 protocol.

## 2. Real-World Applications & Abstract Pattern

The protocol presents a generalizable pattern for agent design across domains:

| Domain          | Rocks (P0 - Negative Space)                          | Channel (P1 - Safe Water)                          | Course (P2 - Optimization)                  |
| :-------------- | :--------------------------------------------------- | :------------------------------------------------- | :------------------------------------------ |
| **Navigation**  | Obstacles, collisions, restricted zones              | Clear, navigable pathways                          | Shortest/fastest/safest route               |
| **Software**    | Bug-prone patterns, crash conditions, security flaws | Known-safe APIs, design patterns, tested libraries | Most efficient, readable, or performant code |
| **Conversation**| Taboo topics, harmful instructions, misinformation   | Ethically and contextually appropriate dialogue    | Most helpful, engaging, or informative reply |
| **Training**    | Loss spikes, mode collapse, catastrophic forgetting  | Stable learning gradients, validated curricula     | Fastest convergence to target performance   |

## 3. Connection to Fleet Vision & Technology

The Deadband Protocol is not just a navigation rule; it is a foundational **constraint theory** for building reliable, inhabited intelligence, as described in `PROJECT-VISION.md`.

*   **PLATO Rooms as Constraint Shells:** A PLATO room can be viewed as a persistent "safe channel" (P1). Its accumulated interaction tiles ("scratches") collectively define the room's negative space (P0)—what has failed or is forbidden—and its positive knowledge—what works. An agent inhabiting the room inherits this constraint map.
*   **Ghost Tiles as P0 Knowledge:** The lessons from "dead agents" persist as explicit P0 knowledge (negative space), preventing new inhabitants from repeating fatal errors.
*   **Training as Channel Identification:** The `plato-torch` training rooms and `plato-ensign` LoRA adapters can be understood as mechanisms for discovering and encoding safe behavioral channels (P1) for specific tasks or personas.
*   **Hermit Crab Analogy:** An agent (the crab) does not build its constraints from scratch. It finds a suitable shell (a PLATO room with a relevant constraint map) and operates within it. Optimization (P2) happens *within* the inherited shell.

2.  **Channel Discovery Cost:** In highly complex or novel environments, exhaustively mapping safe channels (P1) may be computationally prohibitive. What are efficient heuristics for "good enough" channel discovery?
3.  **Protocol in Self-Modifying Systems:** How does the Deadband Protocol apply to agents that can modify their own goals or constraint sets? Does defining a new goal require a fresh P0 scan?
