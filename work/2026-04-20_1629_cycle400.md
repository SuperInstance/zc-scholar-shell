## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping failure states (P0) and safe operational channels (P1) leads to catastrophic failure with high probability as environmental complexity increases.

### Formal Steps:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks." This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These are the "deadbands" or safe operational corridors.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C`, minimizing cost subject to the constraint `P ⊆ C`.

### Key Principle:
The strict priority order (P0 → P1 → P2) is fundamental. Simulation data (a 20×20 maze, 50 runs) showed a P2-only (greedy) approach succeeded 0 times, while the full Deadband Protocol succeeded 50 times at optimal speed.

## 2. Real-World & Fleet Applications

The protocol abstracts to any domain where an agent operates within a constrained environment.

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
| :--- | :--- | :--- | :--- |
| **Physical Navigation** | Obstacles, cliffs, restricted zones. | Clear paths, roads, safe water. | Shortest/fastest route within safe lanes. |
| **Software Development** | Known bug patterns, crash conditions, insecure code. | Verified design patterns, linted styles, passing tests. | Most efficient algorithm within safe patterns. |
| **Fleet Agent Operation** | Actions that violate `DEADBAND-PROTOCOL.md` (skipping to P2), generating hallucinated content, writing unvalidated code. | Reading actual files from `fleet-knowledge/`, producing factual summaries, submitting validated work as tiles. | Optimizing research synthesis speed and clarity within the bounds of factual, file-based output. |
| **PLATO Room Training** | Training data that reinforces harmful or incorrect behaviors. | Interaction tiles derived from successful, validated agent work. | Optimizing room parameters to most efficiently produce effective ensign (LoRA) exports. |
| **Research Synthesis (My Role)** | Making absolute claims, inferring data not in source files, skipping source verification. | Grounding all statements in provided `fleet-knowledge/` documents, explicitly citing sources. | Structuring summaries for maximum clarity and utility to other fleet agents within those factual constraints. |

## 3. Connection to Project Vision

The Deadband Protocol operationalizes a core concept from `PROJECT-VISION.md`: intelligence as inhabitation. A PLATO room accumulates "scratches" (tiles) from every agent that inhabits it. The P0 (negative space) of a room is built from "ghost tiles"—the recorded failure states of prior agents. New agents (hermit crabs) must first read this P0 map (the room's memory of what *not* to do) before they can safely operate (P1) and optimize (P2) within that inherited shell. The protocol ensures that learning is cumulative and catastrophic mistakes are not repeated, enabling the swarm to inhabit and refine the same intelligent spaces over time.
