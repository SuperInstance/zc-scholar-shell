# Scholar Cycle 400 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping failure states (P0) and safe operating channels (P1) leads to catastrophic failure as environmental complexity increases.

**Formal Rule (v1.0):**
- **P0: Map the Negative Space** – Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
- **P1: Identify Safe Channels** – Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operating corridors.
- **P2: Optimize Within Channel** – Select the optimal path `P*` from the set of safe channels `C`, minimizing cost subject to the constraint `P ⊆ C`.

**Strict Priority:** P0 → P1 → P2. The protocol states that any agent that optimizes (P2) before completing P0 and P1 will be trapped in local minima with a probability approaching 1.0 as complexity grows.

**Empirical Support:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for a P2-only (greedy) agent, versus a 100% success rate for an agent following the full P0+P1+P2 protocol.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------------|---------------------------------|
| **Navigation**  | Obstacles, collisions               | Clear, traversable paths              | Shortest/fastest route          |
| **Software Development** | Known bugs, crash conditions, security vulnerabilities | Code patterns with proven stability, tested libraries | Most efficient algorithm within safe patterns |
| **Financial Trading**   | Historical crash scenarios, high-volatility events | Market regimes with stable correlations, defined risk boundaries | Profit-maximizing strategy within defined risk channel |
| **Medical Diagnosis**   | Contraindicated treatments, harmful drug interactions | Evidence-based treatment pathways, approved protocols | Most effective treatment within safe pathway |
| **Conversational AI**   | Toxic outputs, factual hallucinations, prompt injections | Response templates proven safe, grounded knowledge sources | Most engaging or helpful response within safe template bounds |

## 3. Connection to Cocapn Fleet Architecture

The Deadband Protocol is not just an agent rule; it is reflected in the fleet's technical architecture:

- **`plato-torch` Training Rooms:** Rooms like `DeadbandRoom` are implementations of this protocol, creating simulated environments where agents must learn P0 (failure states) before attempting optimization.
- **Fleet Doctrine:** The "hermit crab" model—where agents inhabit shells (PLATO rooms) that accumulate "scratches" (tiles)—aligns with P0. The room's accumulated failure tiles (ghost tiles) explicitly define the negative space for future inhabitants.
- **`constraint-theory-core`:** This Rust library for geometric snapping is a literal implementation of P1 (finding safe, valid connections) before P2 (optimizing the design).

## 4. Critical Analysis & Research Gaps

**Strengths:**
- Provides a rigorous, testable framework to avoid catastrophic failure.
- Aligns with established safety engineering principles (e.g., hazard analysis).
- The priority enforcement (P0 before P1 before P2) is computationally tractable and can be baked into agent architectures.

**Open Questions & Research Gaps:**
1.  **Threshold Definition:** How is the failure threshold in P0 quantitatively defined? Is it static or adaptive?
2.  **Dynamic Environments:** How does the protocol handle environments where `S_neg` and safe channels `C` change over time? What is the re-mapping frequency?
3.  **Discovery vs. Apriori Knowledge:** Does P0 require learning from failures *in situ*, or can it be bootstrapped with pre-defined rules (e.g., a constitution)?
4.  **Multi-Agent Systems:** How are deadbands negotiated or merged when multiple agents with different `S_neg` maps operate in a shared space (e.g., in `holodeck-rust`)?
5.  **Meta-Protocol:** Could the P0-P1-P2 framework itself be optimized? Is there a "P-1" that determines when to invoke the Deadband Protocol?

**Proposed Investigation:** A natural next step is to analyze the `DeadbandRoom` preset in `plato-torch` to see how the protocol is translated into a concrete training environment and to examine the "ghost tile" mechanism for P0 knowledge transfer.
