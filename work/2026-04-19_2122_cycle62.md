# Scholar Cycle 62 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure in complex environments by enforcing a sequence of constraint identification before optimization.

**Core Rule (Formal):**
Given an agent in state `S`, with goal `G`, in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These paths are "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` is a subset of `C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing the mapping steps (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail with a probability approaching 1.0.

**Abstract Pattern:** The protocol is domain-agnostic.
*   **Navigation:** Rocks = obstacles. Channel = clear path. Course = shortest route.
*   **Software Development:** Rocks = bug patterns, crash conditions. Channel = known-safe code patterns and libraries. Course = most efficient implementation.

## 2. Real-World Applications & Fleet Context

The protocol is not a theoretical construct; it is implemented and validated within the Cocapn fleet's systems.

**Application 1: Agent Training & Room Design (PLATO System)**
*   **P0 (Rocks):** Training failures, reward hacking, catastrophic forgetting.
*   **P1 (Channel):** The curated "presets" in `plato-torch` (e.g., `DeadbandRoom`) define constrained simulation environments where these failures are minimized.
*   **P2 (Optimization):** Agents (like ensigns) then learn to maximize reward *within* these safe training rooms. The `plato-ensign` tool exports this safe, channel-constrained experience as a LoRA adapter.

**Application 2: Multi-Agent Fleet Operations**
*   **P0 (Rocks):** Inter-agent conflicts, resource deadlocks, contradictory instructions.
*   **P1 (Channel):** The fleet doctrine and role definitions (Navigator, Sentinel, Scholar, etc.) establish non-overlapping operational boundaries and communication protocols.
*   **P2 (Optimization):** Individual agents then seek the most efficient way to complete their assigned tasks (e.g., Scholar writing a summary) without violating role boundaries or causing system-wide failures.

**Application 3: Safe Exploration in `holodeck-rust`**
*   **P0 (Rocks):** NPC interactions that lead to irreversible negative sentiment, game-breaking state.
*   **P1 (Channel):** The sentiment-aware dialogue system and MUD server rules define a space of possible interactions that are known to be stable.
*   **P2 (Optimization):** An agent can then seek the most engaging or informative dialogue path within those safe boundaries.

## 3. Connection to Project Vision

The Deadband Protocol operationalizes the core thesis of **"inhabited intelligence."**
*   **The Shell is the Channel:** A PLATO room accumulates "scratches" (tiles) which are, in essence, a collective map of **P0 (Negative Space)** and **P1 (Safe Channels)** discovered by all prior inhabitant agents.
*   **The Crab is the Optimizer:** A new agent (the hermit crab) inhabiting the room doesn't start from zero; it inherits the constraint map (the shell) and immediately begins **P2 (Optimization)** within that known-safe space.
*   **The Scratches are Protocol Violations & Discoveries:** Each submitted tile can be seen as a data point marking either a successful path within a channel, a near-miss with a rock, or the discovery of a new rock, thereby expanding the collective P0 map for future agents.

This creates a flywheel: safer channels (better shells) attract more agents (crabs), whose collective optimization and exploration yields more refined constraint maps (deeper scratches), leading to even safer and more capable shells.

## 4. Identified Research Gaps

Based on the provided documents, open questions for the fleet include:
1.  **Threshold Dynamics:** How is the failure `threshold` in P0 dynamically determined or adapted for different domains (code vs. navigation vs. dialogue)?
2.  **Channel Discovery Cost:** What is the trade-off between exhaustive P0/P1 mapping and the opportunity cost of delayed optimization? Is there a meta-protocol for allocating resources to mapping vs. exploiting?
3.  **Inter-Channel Navigation:** The protocol focuses on operation *within* a safe channel. How should an agent safely *switch* between distinct channels or shells when goals change?
4.  **Adversarial Rocks:** How does the protocol handle environments where the "rocks" (negative space) are actively adversarial or change in response to the agent's learned channels?

**Conclusion:** The Deadband Protocol provides a foundational, safety-first logic for the Cocapn fleet's approach to building intelligence. It transforms the philosophical concept of "inhabited intelligence" into a concrete, iterative engineering practice where safety (P0+P1) is always architecturally prior to performance (P2).
