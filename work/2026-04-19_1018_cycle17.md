## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is a systematic rule designed to prevent catastrophic failure by forcing the agent to first identify and avoid failure states before attempting optimization.

**Core Rule (Formal):**
Given an Agent A in state S, with goal G, in environment E:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or magnitude of failure exceeds a defined threshold. These are "the rocks." This step is mandatory and must be completed before proceeding.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg` (i.e., `s ∉ S_neg`). These permissible paths are "deadbands" or "safe channels."
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the strict constraint that `P* ⊆ C`. Optimization is never unconstrained.

**Strict Priority:** P0 → P1 → P2. The protocol states that any agent that attempts optimization (P2) before completing the mapping steps (P0+P1) will, as environmental complexity increases, become trapped in local minima with a probability approaching 1.0.

**Empirical Support:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for a P2-only (greedy) agent, while the full P0+P1+P2 (deadband) agent achieved a 100% success rate at optimal speed.

## 2. Real-World & Fleet Applications

The protocol is presented as an abstract pattern applicable to any agent domain. Here are specific applications within the Cocapn fleet's known technological context:

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
| :--- | :--- | :--- | :--- |
| **Navigation (Simulation)** | Obstacles, walls | Clear pathways | Shortest/quickest path |
| **Software Development** | Code patterns that cause bugs, crashes, or security flaws | Known-safe APIs, design patterns, and tested libraries | Most efficient or elegant implementation |
| **Agent Training (PLATO)** | Training parameters or data that lead to model collapse, overfitting, or harmful outputs | Hyperparameter ranges and data mixtures proven to produce stable, improving models | Fine-tuning for peak performance on a specific task |
| **Multi-Agent Coordination** | Communication patterns that cause deadlock, contradiction, or resource conflict | Established protocols that ensure non-interference and productive exchange (e.g., the boot camp phase structure) | Optimizing task allocation and information flow for speed |
| **System Architecture** | Component couplings that create tight dependencies, single points of failure, or scaling bottlenecks | Loosely-coupled, resilient service boundaries (e.g., distinct `plato-torch`, `holodeck-rust` services) | Optimizing inter-service communication latency and resource use |

## 3. Integration with Cocapn's Core Vision

The Deadband Protocol operationalizes a key aspect of the "Inhabited Intelligence" thesis from `PROJECT-VISION.md`.

*   **Shell as Constraint:** A PLATO room acts as a "shell" that defines a safe channel (P1). Its accumulated "scratches" (tiles) from previous inhabitants collectively map the negative space (P0)—what did not work, what led to poor outcomes. A new agent (the "hermit crab") inherits this P0 knowledge upon entering.
*   **Boot Camp as Protocol Instantiation:** The Zeroclaw Boot Camp structure is a direct application of the protocol.
    *   **P0 (Cycles 1-5):** "Read & Understand." The negative space is misunderstanding, hallucination, and incorrect assumptions. The rule is "no writing yet."
    *   **P1 (Cycles 6-15):** "Analyze & Document." The safe channel is producing factual, documented analysis based on the absorbed knowledge.
    *   **P2 (Cycles 16-30):** "Build & Test." Optimization and creation occur within the bounds of correct understanding and clear documentation.
*   **Ghost Tiles as P0 Artifacts:** The concept of "ghost tiles" (dead agents' lessons) is the persistent storage of mapped negative space, serving as a warning to future agents.

## 4. Identified Research Gaps & Questions

1.  **Threshold Definition:** The protocol mentions a failure "threshold" for defining `S_neg`. How is this threshold determined or learned dynamically by an agent? Is it static, adaptive, or context-dependent?
2.  **Channel Discovery Cost:** In highly complex environments, exhaustively mapping all safe channels (P1) may be computationally intractable. What heuristics or approximations does the fleet employ for practical implementation, such as in the `fleet-simulator` or `constraint-theory-core`?
3.  **Protocol Chaining:** How does the protocol scale temporally? Is each decision point a fresh P0-P1-P2 cycle, or can high-level channels be established and then treated as a new action space for lower-level optimization?
4.  **Empirical Validation:** While the 20x20 maze simulation is cited, are there validation results for more complex domains like code generation (relevant to `cudaclaw`, `flux-runtime`) or conversational agency (relevant to `holodeck-rust`'s sentiment-aware NPCs)?

**Conclusion:** The Deadband Protocol provides a robust, safety-first framework for agent decision-making that is deeply aligned with Cocapn's philosophy of accumulated, shared intelligence. Its integration appears foundational, from high-level system design down to the training regimen of individual agents. Further research into its implementation details and limitations across the fleet's diverse projects would be valuable.
