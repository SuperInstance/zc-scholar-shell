## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing agents to first identify and avoid failure states before attempting optimization.

**Formal Rule (v1.0):**
Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational zones.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` remains within `C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing the mapping steps (P0+P1) will, with high probability, become trapped in a local minimum or fail as environmental complexity increases.

**Empirical Basis:** Simulation data from a 20x20 maze (50 runs) showed a 0% success rate for a P2-only (greedy) agent, while the full P0+P1+P2 protocol achieved a 100% success rate at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------------|---------------------------------|
| Navigation      | Obstacles, hazards                  | Clear, traversable paths              | Shortest/fastest route          |
| Software Development | Bug-prone patterns, crash conditions | Known-safe APIs, tested code patterns | Most efficient algorithm        |
| Financial Trading | High-volatility events, regulatory violations | Compliant, historically stable strategies | Highest risk-adjusted return    |
| Conversation Agent | Toxic outputs, factual hallucinations | Ethically and factually bounded responses | Most helpful, engaging response |

## 3. Connection to Fleet Vision & Technology

The Deadband Protocol is not just a navigation rule; it is a foundational **constraint theory** for building reliable, inhabited intelligence, as described in `PROJECT-VISION.md`.

*   **P0 as Collective Memory:** The "rocks" (`S_neg`) are the accumulated scratches in a PLATO room's shell—the recorded failures of all prior agents. This negative space map is a shared, persistent memory that protects future inhabitants.
*   **P1 as the Shell:** The "safe channels" are the validated boundaries of the room itself. An agent (the hermit crab) must first find and inhabit this shell (the room's constraints) before it can act.
*   **P2 as Inhabited Intelligence:** Optimization only occurs *within* the inhabited shell. The intelligence of the fleet is the sum of optimal paths found by many agents, all operating within the safe, remembered channels of their shared environments.

**Technological Instantiation:**
*   **`plato-torch` Rooms:** A training room preset (e.g., `DeadbandRoom`) would enforce this protocol, rewarding agents for correct P0/P1 mapping before P2 optimization.
*   **`constraint-theory-core`:** This Rust library for geometric snapping provides a concrete implementation of the "safe channel" concept—finding valid, connected states within a constrained space.
*   **Fleet Training:** The protocol directly informs the "Boot Camp" phased approach: **Phase 1 (Read)** is P0 (learn what not to do), **Phase 2 (Analyze)** is P1 (map the safe space of understanding), **Phase 3 (Build)** is P2 (optimize output within known-safe patterns).

## 4. Identified Research Gaps

Based on the provided documentation, potential areas for further fleet investigation include:
1.  **Dynamic Deadbands:** The protocol describes a static map (`S_neg`). How is it updated in real-time as the environment changes or as the agent's own actions alter the state space?
2.  **Threshold Specification:** How is the failure "threshold" defined quantitatively across different domains (e.g., loss value in ML, probability of crash in code)?
3.  **Integration with Stochastic Models:** How does the protocol interact with probabilistic world models? Is `S_neg` a set of states or a probability distribution?
4.  **Meta-Protocol:** Could the Deadband Protocol itself be applied to the process of *designing* AI agents? (P0: Identify agent architectures prone to failure; P1: Select from proven, stable design patterns; P2: Optimize parameters).

**Conclusion:** The Deadband Protocol is a crystallized expression of a core fleet principle: **safety through constraint, and intelligence through operation within validated, collective boundaries.** It transforms the problem of intelligence from unconstrained optimization to one of constrained navigation within a space defined by shared, accumulated experience.
