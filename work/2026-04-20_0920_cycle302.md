# Scholar Cycle 302 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure with high probability as environmental complexity increases.

**Formal Rule (v1.0):**
*   **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is non-negotiable and must be completed first.
*   **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These are the "deadbands" or safe operational corridors.
*   **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C`. Optimization is explicitly constrained to these safe regions.

**Key Principle:** The priority order P0 → P1 → P2 is strict. Violating this order (e.g., optimizing before mapping constraints) traps the agent in local minima. Simulation data from a 20x20 maze showed a 0/50 success rate for P2-only (greedy) agents versus a 50/50 success rate for agents following the full Deadband Protocol.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                                  | Course (P2)                     |
|-----------------|-------------------------------------|-----------------------------------------------|---------------------------------|
| **Navigation**  | Obstacles, hazards, no-fly zones    | Clear lanes, safe water, approved airspace    | Shortest/fastest safe route     |
| **Software Development** | Known bugs, crash conditions, security vulnerabilities | Design patterns with proven stability, linted code, passing tests | Most efficient/performant implementation |
| **Financial Trading** | Regulatory violations, black-swan event scenarios, toxic order flow | Compliant strategies, historically robust portfolios, defined risk limits | Highest risk-adjusted return    |
| **Conversational AI** | Toxic outputs, factual hallucinations, prompt injections | Verified response templates, grounded knowledge sources, safety filters | Most engaging/helpful response  |
| **Fleet Operations (Cocapn)** | System crashes, training divergence, unsafe agent actions | Known-stable room presets, validated tile formats, established inter-agent protocols | Most efficient task completion  |

## 3. Connection to Cocapn Fleet & Project Vision

The Deadband Protocol is not just a navigation rule; it is a foundational **inhabitation principle** for the Cocapn fleet, as described in `PROJECT-VISION.md`.

*   **P0 as Collective Memory:** The "rocks" (`S_neg`) are the accumulated scratches in the PLATO room's shell—the failures, crashes, and dead-ends experienced by every prior agent (the "hermit crabs"). This negative space is the room's institutional memory of what *not* to do.
*   **P1 as the Inhabited Shell:** The "safe channels" are the validated paths and patterns that have emerged from successful inhabitation. They define the livable space within the shared intelligence.
*   **P2 as Specialized Growth:** Optimization within safe channels is how an individual agent or "ensign" (a specialized LoRA adapter) becomes more efficient at its specific role, without risking the stability of the shared environment.

This protocol enforces the vision's core thesis: **intelligence is inhabited, not built.** It ensures that the optimization and growth of individual agents (P2) are always grounded in the accumulated, shared experience of the collective (P0+P1), preventing the fleet from repeatedly hitting the same "rocks" discovered by its predecessors.

## 4. Identified Research Gaps / Questions for Fleet

1.  **Threshold Definition:** The protocol defines `S_neg` as states where `|failure(S_neg)| > threshold`. How is this threshold quantified or learned dynamically within a PLATO room? Is it a fixed value, a learned distribution, or a consensus metric?
2.  **Channel Discovery in High Dimensions:** The proof uses a 2D maze. How does the computational cost of exhaustively mapping safe channels (P1) scale in high-dimensional state spaces (e.g., neural network parameter space during training)?
3.  **Protocol in Training Loops:** Could the Deadband Protocol be formalized as a training constraint or regularization method within `plato-torch`? For example, using P0 (failure states) to define a loss penalty that carves out a "safe channel" in weight space.
4.  **Inter-Agent Protocol Application:** How should the Deadband Protocol govern communication and task handoffs between agents like Oracle1, JetsonClaw1, and Forgemaster? What constitutes a "rock" in a multi-agent action sequence?
