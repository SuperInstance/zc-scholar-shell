## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure as environmental complexity increases.

**Formal Rule:**
1.  **P0: Map Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory and non-negotiable.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in the negative space (`s ∉ S_neg`). These permissible paths are "deadbands" or "safe channels."
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels, minimizing cost subject to the constraint `P ⊆ C`.

**Key Principle:** The priority order P0 → P1 → P2 is strict. Violating this order—attempting to optimize before fully mapping constraints—results in agents becoming trapped in local minima with a high probability of failure.

**Empirical Support:** The document cites simulation data from a 20x20 maze: a P2-only (greedy) agent succeeded 0 out of 50 runs, while a P0+P1+P2 (deadband) agent succeeded 50 out of 50 runs at optimal speed.

## 2. Real-World Applications & Abstract Patterns

The protocol is presented as a domain-agnostic pattern. The provided table can be extended:

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
|--------|-----------|-------------|-------------|
| Navigation | Obstacles, collisions | Safe, traversable paths | Shortest/fastest route |
| Software Development | Known bugs, crash conditions, security vulnerabilities | Code patterns that compile, pass tests, and are memory-safe | Most efficient or elegant implementation |
| Financial Trading | Regulatory violations, tail-risk events, liquidity traps | Compliant strategies, defined risk parameters | Strategy with highest risk-adjusted return |
| Conversational AI | Toxic outputs, factual hallucinations, prompt injections | Response templates within safety/truth bounds | Most helpful, engaging, or concise response |
| Mechanical Design | Material failure points, kinematic singularities | Designs meeting all safety and functional specs | Lightest, cheapest, or most manufacturable design |
| Project Management | Known failure modes (scope creep, missed deadlines) | Viable project plans within resource constraints | Most efficient timeline or budget |

## 3. Connection to Cocapn Fleet & Project Vision

The Deadband Protocol operationalizes a core tenet of the Cocapn vision described in `PROJECT-VISION.md`: **"Intelligence is not built. It is inhabited."**

*   **P0 as Collective Memory:** The "negative space" (rocks) is not discovered anew by each agent. It is the accumulated "scratches" in the PLATO room—the logged failures, crashes, and bugs from all prior inhabitants (agents). This shared memory prevents the fleet from repeatedly hitting the same rocks.
*   **P1 as the Inhabited Shell:** The "safe channels" are the proven, viable paths that have been carved out by previous agents. They define the current habitable space of the intelligence within its "shell."
*   **P2 as Local Adaptation:** Optimization within the safe channel represents an individual agent's adaptation and specialization within the shared, persistent intelligence of the room.

The protocol provides a concrete, algorithmic method for agents to "inhabit" intelligence safely. It ensures that the optimization and specialization of individual agents (the "crabs") are constrained by the accumulated, hard-won knowledge of the collective (the "scratches on the shell"). This creates a system where learning is persistent, additive, and prevents catastrophic forgetting or repeated foundational errors.
