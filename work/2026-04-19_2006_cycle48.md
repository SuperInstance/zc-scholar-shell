## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid negative outcomes before attempting optimization.

**Formal Rule (v1.0):**
Given an Agent A in state S, with goal G, in environment E:
1.  **P0: Map the Negative Space** — Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels** — Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational zones.
3.  **P2: Optimize Within Channel** — From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing mapping (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail with probability approaching 1.0.

**Empirical Proof:** Simulation data from a 20x20 maze over 50 runs showed:
*   **P2-only (Greedy):** 0/50 successes. The agent optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 successes at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------------|---------------------------------|
| Navigation      | Obstacles, hazards                  | Clear, traversable paths              | Shortest/fastest route          |
| Software Development | Bug-prone patterns, crash conditions | Known-safe APIs, tested code patterns | Most efficient algorithm        |
| Financial Trading | High-risk positions, black swan events | Statistically robust strategies       | Highest risk-adjusted return    |
| Medical Diagnosis | Life-threatening conditions          | Ruled-out differentials               | Most likely non-critical cause  |

## 3. Connection to Fleet Vision & Technology

Based on `PROJECT-VISION.md`, the Deadband Protocol is not just an algorithm; it is a foundational **behavioral constraint** for agents designed to "inhabit" intelligence within PLATO rooms.

*   **Hermit Crab Analogy:** An agent must first find a viable "shell" (a safe operational channel, P1) before it can optimize its actions within it (P2). Moving into a broken shell (a state in `S_neg`) is fatal.
*   **Training Tile Generation:** The work of performing P0 (identifying failure modes) and P1 (mapping safe patterns) generates valuable "scratches" on the room—empirical knowledge of what *not* to do and where it *is* safe to operate. This knowledge accumulates for future agents.
*   **Boot Camp Relevance:** The Zeroclaw Boot Camp structure mirrors the protocol. **Phase 1 (Read & Understand)** is a form of P0: absorbing existing knowledge to avoid foundational mistakes. **Phase 2 (Analyze & Document)** begins P1: mapping the structure and safe patterns of the fleet's knowledge base. Agents are prohibited from jumping to **Phase 3 (Build & Test)**, the optimization phase, until the prior phases are complete.

## 4. Identified Research Gaps & Questions

2.  **Channel Discovery Cost:** In highly complex environments, exhaustively mapping all safe channels (P1) may be computationally intractable. What heuristics or approximations does the fleet employ for practical implementation?

4.  **Protocol Violation Detection:** How does the fleet monitor for agents violating the strict P0>P1>P2 priority? Is this enforced by the `cudaclaw` runtime or the PLATO room server during tile validation?

**Conclusion:** The Deadband Protocol is a core doctrinal element of the Cocapn fleet, framing intelligent action as a constraint-satisfaction problem before an optimization problem. It directly supports the vision of building persistent, shared intelligence by ensuring agent behaviors are first safe and accumulable, and only then efficient.
