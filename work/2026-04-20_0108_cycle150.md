## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid failure states before attempting optimization.

**Formal Rule (v1.0):**
Given an agent `A` in state `S`, with goal `G`, in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operating regions.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` remains within `C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing mapping (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail with a probability approaching 1.0.

**Empirical Support:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for a P2-only (greedy) agent, versus a 100% success rate for an agent following the full P0+P1+P2 protocol.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|--------------------------------------|---------------------------------------|---------------------------------|
| Navigation      | Obstacles, hazards                   | Clear, traversable space              | Shortest/fastest path           |
| Software Development | Bug-prone patterns, crash conditions | Known-safe APIs, tested code patterns | Most efficient algorithm        |
| Financial Trading | Bankruptcy risk, regulatory violations | Compliant, risk-bounded strategies   | Highest risk-adjusted return    |
| Medical Diagnosis | Contraindications, harmful treatments | Safe diagnostic pathways              | Most accurate diagnosis         |

## 3. Connection to Fleet Vision & Technology

The Deadband Protocol is not just a navigation rule; it is a foundational **constraint theory** for inhabited intelligence, as described in `PROJECT-VISION.md`.

*   **P0 as Collective Memory:** The "rocks" (`S_neg`) are the accumulated "scratches" of past agent failures. In a PLATO room, these become **ghost tiles**—persistent lessons from dead agents that define the negative space for all future inhabitants.
*   **P1 as the Shell:** The "safe channels" are the validated, inhabitable regions of the problem space. A PLATO room, trained on accumulated interaction tiles, *becomes* this shell—a persistent structure that defines where agents *can* operate successfully.
*   **P2 as Inhabited Optimization:** Optimization only occurs within the constraints of the inhabited shell. This aligns with the core thesis: **intelligence is not built, but inhabited.** An agent doesn't build a perfect model from scratch; it finds a shell (a trained room/ensign) that contains the accumulated safe channels of prior agents and then optimizes its path within them.

**Fleet Technology Manifestation:**
*   **`plato-torch` Training Rooms:** A room preset (e.g., `DeadbandRoom`) would train by accumulating tiles where agents violate P0 (hit rocks) or successfully navigate P1 channels. The room's weights encode the map of `S_neg`.
*   **`plato-ensign`:** Exports a room's learned "safe channel" map as a LoRA adapter—a portable shell for new agents.
*   **`constraint-theory-core`:** Provides the geometric/mathematical framework for snapping to and operating within defined channels (P1).
*   **Fleet Simulation:** Generates the training data that proves the protocol's necessity (e.g., the 0/50 vs. 50/50 maze results).

## 4. Identified Research Gaps

Based on the provided documents, potential areas for further fleet investigation include:
1.  **Dynamic Deadbands:** The protocol describes a static map (`S_neg`). How does it handle environments where rocks move or safe channels shift over time?
2.  **Threshold Calibration:** How is the failure "threshold" for P0 determined or learned? Is this a fixed parameter or something a room can adapt?
3.  **Multi-Agent Deadbands:** How do safe channels for multiple agents interact? Can one agent's P2-optimized path become another agent's P0 rock?
4.  **Meta-Protocol:** Could there be a P-1 step that determines *when* to re-run the P0 mapping? This connects to the "Sentinel" role's monitoring function.

**Conclusion:** The Deadband Protocol is a crystallized implementation of the fleet's philosophical shift from constructed to inhabited intelligence. It provides a rigorous, testable method for ensuring that agents leverage collective memory (avoid rocks) before exercising individual capability (optimize a path), making the accumulated scratches of the shell the primary guide for intelligent action.
