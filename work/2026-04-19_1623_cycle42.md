## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid hazardous states before attempting optimization.

**Core Steps (Strict Priority):**
*   **P0: Map the Negative Space.** Identify all states (`S_neg`) where the probability or cost of failure exceeds a defined threshold. These are "the rocks." This step is non-negotiable and must be completed before proceeding.
*   **P1: Identify Safe Channels.** Find all possible paths (`P`) where every state along the path is *not* in the negative space (`s ∉ S_neg`). These paths are "safe channels" or "deadbands"—the only spaces where the agent is permitted to operate.
*   **P2: Optimize Within Channel.** Only after P0 and P1 are complete may the agent select the optimal path (`P*`) from within the set of safe channels, minimizing cost or maximizing reward subject to the constraint `P ⊆ C`.

**Key Insight:** The protocol asserts that an agent which attempts to optimize (P2) before completing the mapping steps (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail catastrophically with a probability approaching 1.0. Empirical simulation data (a 20x20 maze, 50 runs) is cited as evidence: a P2-only (greedy) strategy succeeded 0 times, while the full deadband protocol succeeded 50 times.

## 2. Real-World Applications & Fleet Context

The protocol is presented as an abstract pattern applicable to any domain. Here are its potential applications within the Cocapn fleet's known technological context:

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
| :--- | :--- | :--- | :--- |
| **Agent Navigation (Fleet Simulator)** | Obstacles, collision zones, energy-depletion areas. | Verified, collision-free corridors. | Shortest or most energy-efficient path within safe corridors. |
| **Code Generation & Safety (Zeroclaw Runtime)** | Code patterns known to cause crashes, security vulnerabilities, or infinite loops. | Known-safe API usage patterns, validated code templates. | Most efficient or elegant implementation using only safe patterns. |
| **Training Room Design (plato-torch)** | Hyperparameter sets that lead to model collapse, vanishing gradients, or overfitting. | Ranges of hyperparameters (learning rate, batch size) proven stable across prior runs. | Fine-tuning hyperparameters within the safe range for optimal convergence speed. |
| **NPC Behavior (holodeck-rust)** | Dialogue trees or actions that break character, violate world rules, or cause player frustration. | Behavior scripts that remain within defined personality and narrative constraints. | Choosing the most engaging or context-appropriate action from the safe set. |
| **System Integration (Weaver Role)** | Service dependencies that cause cascading failures, incompatible data formats. | Known-stable service communication pathways and data transformation pipelines. | Optimizing data flow latency or resource usage across the stable integration map. |

## 3. Connection to Project Vision

The Deadband Protocol operationalizes a key aspect of the fleet's core thesis: "Intelligence is not built. It is inhabited." The protocol itself is a "scratch" – a learned, accumulated lesson about avoiding failure. By mandating P0 (identifying negative space), it ensures that the hard-won knowledge of "rocks" (failures, bugs, crashes) from past agents becomes foundational for future action. This directly supports the vision of PLATO rooms as persistent shells that accumulate these lessons (tiles), preventing each new "hermit crab" agent from relearning the same catastrophic failures.

**Conclusion:** The Deadband Protocol is less an optimization algorithm and more a **meta-safety constraint**. It is a rule for how an agent should apply its intelligence: first to learn boundaries from collective experience (P0), then to explore possibility within those boundaries (P1), and finally to refine performance (P2). It is a formal mechanism to ensure that the fleet's accumulated "scratches" directly guide and constrain the behavior of all subsequent agents.
