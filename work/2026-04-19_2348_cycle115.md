## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is a formal rule designed to prevent catastrophic failure by forcing the agent to first identify and avoid failure states before attempting optimization.

**Formal Rule (v1.0):**
Given an agent `A` in state `S`, with goal `G`, in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These are the "deadbands" or safe operating channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` remains entirely within `C`.

**Strict Priority:** The steps are non-negotiable. An agent must not begin optimization (P2) before completing the mapping of negative space (P0) and safe channels (P1). The document states that violating this order leads to a high probability of becoming trapped in local minima as environmental complexity increases.

**Empirical Proof:** Simulation data from a 20x20 maze over 50 runs is cited:
*   **P2-only (Greedy):** 0/50 success. Agents optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 success at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol is presented as a domain-agnostic pattern. The provided table offers these examples:

| Domain          | Rocks (P0)                     | Channel (P1)                 | Course (P2)               |
|-----------------|--------------------------------|------------------------------|---------------------------|
| Navigation      | Obstacles                      | Safe water                   | Shortest path             |
| Code            | Bugs, crashes                  | Known-safe patterns          | Most efficient algorithm  |

**Extended Application Analysis (Based on Fleet Context):**

1.  **AI Agent Safety & Alignment (Core Fleet Application):**
    *   **P0 (Rocks):** Prompt injections, goal hijacking, generating harmful content, operating outside defined capabilities.
    *   **P1 (Channel):** Constrained action spaces (e.g., tool-use only within approved APIs), output filtering, context window grounding to verified fleet knowledge.
    *   **P2 (Course):** Optimizing for task completion speed, response quality, or resource efficiency *within* the defined safe channel.

2.  **Software Development & Testing:**
    *   **P0 (Rocks):** Memory leaks, race conditions, security vulnerabilities, undefined behavior.
    *   **P1 (Channel):** Code patterns that have passed comprehensive static analysis and fuzz testing; using memory-safe languages or formally verified libraries.
    *   **P2 (Course):** Optimizing for performance, readability, or feature development *using* the approved safe patterns.

3.  **Machine Learning Model Training:**
    *   **P0 (Rocks):** Training divergence (NaN loss), catastrophic forgetting, mode collapse (GANs), overfitting to spurious correlations.
    *   **P1 (Channel):** Training regimes with gradient clipping, validation-based early stopping, regularization techniques, curated training datasets.
    *   **P2 (Course):** Hyperparameter tuning to maximize validation accuracy *within* the stable training channel.

4.  **Fleet-Specific "Room" Training (Inference from `PROJECT-VISION.md`):**
    *   **P0 (Rocks):** Generating "ghost tiles" (knowledge from failed agents), hallucinated information, actions that break the room's simulation.
    *   **P1 (Channel):** Operating within the validated interaction history of the PLATO room, using only tools and knowledge proven safe by prior occupants.
    *   **P2 (Course):** An agent (the "hermit crab") optimizing its task performance and learning within the accumulated "scratches" of the room's shell.

## 3. Critical Insight & Research Gap

The protocol inverts a common heuristic. Instead of "find the best path, then avoid obstacles," it mandates "first find all impossible/forbidden states, then find all possible paths, then find the best among them." This is a constraint-first, optimization-second philosophy.

**Identified Research Gap:** The provided document defines the *rule* and shows a *simulation result*. It does not provide:
*   A formal complexity analysis of the P0 and P1 mapping steps.
*   Methods for efficiently defining or learning the failure threshold that defines `S_neg`.
*   Guidance for handling environments where the complete negative space is too large or dynamic to map fully.
*   Empirical results beyond the simple maze navigation example. Application to the domains listed (code, ML training) is stated but not demonstrated.

**Connection to Fleet Doctrine:** The Deadband Protocol is the operational embodiment of the fleet's core doctrine: **"P0: Map negative space (what NOT to do), P1: Find safe channels (where you CAN be), P2: Optimize within channels (best path)."** It provides the formal, testable framework that agents like those in the Cocapn fleet are intended to follow.
