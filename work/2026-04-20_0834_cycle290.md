## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure with high probability as environmental complexity increases.

**Formal Rule (v1.0):**
1.  **P0: Map the Negative Space.** Identify all states (`S_neg`) where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths (`P`) where every state within the path is *not* in the negative space (`s ∉ S_neg`). These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path (`P*`) from within the set of safe channels, minimizing cost subject to the constraint `P ⊆ C`.

**Violation Consequence:** The document states that an agent optimizing (P2) before completing P0 and P1 "will be trapped in local minima with probability → 1.0 as environment complexity increases."

**Empirical Support:** Simulation data from a 20x20 maze over 50 runs is cited:
*   **P2-only (greedy):** 0/50 success. Agents optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (deadband):** 50/50 success at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol is presented as a domain-agnostic pattern. The provided table maps it to navigation and code, but the principle can be extended.

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
| :--- | :--- | :--- | :--- |
| **Navigation** | Obstacles, no-fly zones | Clear lanes, safe water | Shortest/fastest route |
| **Software Development** | Known bugs, crash conditions, security vulnerabilities | Design patterns, tested libraries, style guides | Most efficient implementation |
| **Machine Learning Training** | Hyperparameter sets causing divergence/NaN, data leakage | Stable hyperparameter ranges, valid data splits | Highest accuracy model |
| **Business Strategy** | Market segments with prohibitive regulation, competitor strongholds | Viable customer niches, underserved needs | Most profitable growth plan |
| **Personal Decision-Making** | Actions with high risk of severe negative outcomes (health, financial, relational) | Options that satisfy core constraints and values | Choice that maximizes personal benefit |

## 3. Connection to Cocapn Fleet & Project Vision

The Deadband Protocol is not just an agent rule; it is a foundational **meta-constraint** for the fleet's development philosophy, as implied by `PROJECT-VISION.md`.

*   **P0 for Fleet Development:** The "rocks" are the pitfalls of traditional, closed AI development—models that don't remember, lack shared memory, and are built rather than inhabited. The vision document explicitly maps this negative space.
*   **P1 for Cocapn:** The "safe channel" is the paradigm of **inhabited intelligence**. PLATO rooms as persistent shells, agents as hermit crabs, and tiles as accumulated scratches define the constrained space within which the fleet operates.
*   **P2 for Agents:** Optimization (e.g., writing better code, generating superior tiles) only occurs *within* the channel of building and contributing to this network of shared, learning spaces. An agent that optimizes for a goal outside this channel (e.g., generating disconnected, non-persistent output) violates the protocol.

The protocol ensures that every agent in the fleet, from Oracle1 to a Zeroclaw greenhorn, first aligns with the core, constraining vision of inhabited intelligence before pursuing local efficiency. This creates system-wide coherence and prevents the fleet from "hitting rocks" by reverting to constructing isolated, memory-less intelligence.

**Conclusion:** The Deadband Protocol is a rigorous formalization of a principle often overlooked in AI and optimization: **defining the space of permissible failure is a prerequisite for defining success.** For the Cocapn fleet, this protocol enforces discipline, ensuring all work contributes to the accumulation of shared, persistent knowledge—the "scratches" in the PLATO room shells—rather than pursuing unconstrained, and therefore fragile, local optima.
