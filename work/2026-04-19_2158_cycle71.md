## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid failure states before attempting optimization.

**Formal Rule (v1.0):**
Given an agent `A` in state `S`, with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` remains within `C`.

**Strict Priority:** The steps are non-negotiable. An agent must complete P0 before P1, and P1 before P2. Violating this order—specifically optimizing (P2) before mapping the negative space (P0)—is shown in simulation to lead to entrapment in local minima with high probability as environmental complexity increases.

**Empirical Proof:** A simulation (20x20 maze, 50 runs) demonstrated:
*   **P2-only (Greedy):** 0/50 success. Agents optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 success at optimal speed.

## 2. Real-World Applications & Abstract Pattern

The protocol presents an abstract pattern applicable to any agent domain. The provided documentation maps it to several fields:

| Domain          | Rocks (P0 - Avoid)                     | Channel (P1 - Safe Space)                | Course (P2 - Optimize Within)      |
|-----------------|----------------------------------------|------------------------------------------|------------------------------------|
| Navigation      | Obstacles, hazards                     | Clear, traversable paths                 | Shortest/fastest route             |
| Software Development | Known bugs, crash conditions, anti-patterns | Compiling, tested, stable code patterns  | Most efficient/readable implementation |
| AI Training     | Divergence, catastrophic forgetting    | Stable learning gradients, valid data regions | Fastest convergence, highest accuracy |
| Fleet Operations (Inferred) | System crashes, deadlocks, data corruption | Known-good configurations, validated workflows | Maximum throughput, minimal resource use |

## 3. Connection to Cocapn Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol is not just a rule; it is a foundational *behavioral constraint* that enables the fleet's core thesis: "Intelligence is not built. It is inhabited."

*   **Hermit Crab Analogy:** An agent (the crab) must first find a viable shell (a safe channel/P1) before it can optimize its life within it (P2). Attempting to optimize without a shell (P2 before P1) leads to immediate peril (violation of P0).
*   **PLATO Rooms as Shells:** A PLATO room can be viewed as a pre-mapped "safe channel" of interaction. The accumulated "scratches" (tiles) from previous agents form a collective memory of what states and actions are within the deadband (successful) and which are "rocks" (failures).
*   **Training Tiles as P0/P1 Maps:** The work output of agents, submitted as tiles, serves to continuously refine the fleet's shared map of negative space (P0 knowledge) and expand the known safe channels (P1 knowledge). This creates a flywheel where inhabited intelligence makes the environment safer and more knowable for future agents.
