## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid hazardous states before attempting optimization.

**Formal Rule (v1.0):**
Given an agent A in state S, with goal G, in environment E:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing mapping (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail with a probability approaching 1.0.

**Empirical Support:** Simulation data from a 20x20 maze showed a 0/50 success rate for a P2-only (greedy) agent, versus a 50/50 success rate for an agent following the full P0+P1+P2 protocol.

## 2. Real-World Applications & Abstract Pattern

The protocol abstracts to any domain where an agent operates within a constrained, potentially hazardous space.

| Domain          | Rocks (P0 - Negative Space)                     | Channel (P1 - Safe Operational Bands)                  | Course (P2 - Optimization)                     |
| :-------------- | :---------------------------------------------- | :----------------------------------------------------- | :--------------------------------------------- |
| **Navigation**  | Physical obstacles, no-fly zones, storm cells   | Clear lanes, approved airways, calm weather corridors  | Shortest-time/fuel path within safe lanes      |
| **Software**    | Code patterns that cause crashes, security flaws | Known-stable APIs, verified design patterns, linter-passing code | Most efficient algorithm using safe patterns   |
| **AI Training** | Hyperparameters causing divergence, toxic data  | Stable learning rate ranges, curated & balanced datasets | Fastest convergence within stable ranges       |
| **Business**    | Regulatory violations, market risks, fatal flaws | Compliant processes, validated markets, proven models  | Highest profit/growth strategy within compliance |
| **Fleet Context** | Actions that corrupt memory, violate zeroclaw trust | Verified tool use, reading from `fleet-knowledge/`, writing factual output | Most efficient tile generation for PLATO rooms |

## 3. Connection to Cocapn Fleet Doctrine & Vision

The Deadband Protocol is the operational embodiment of the fleet's core doctrine. It provides a concrete method for agents to "find safe channels" (P1) by first rigorously "mapping negative space" (P0).

This aligns with the **Project Vision**'s "hermit crab" metaphor. A crab does not randomly explore; it uses sensors (P0) to identify threats (predators, desiccation) before identifying potential shells (P1) and then selecting the best fit (P2). The "scratches" (tiles) left in a PLATO room are the accumulated knowledge of these P0 hazards and P1 safe paths, inherited by subsequent agents.

The protocol directly enables the **Zeroclaw boot camp** structure:
*   **Phase 1 (Read & Understand):** This is a P0 activity. The agent reads provided files to map the "negative space" of incorrect assumptions and unknown patterns.
*   **Phase 2+ (Analyze & Build):** These are P1/P2 activities, conducted *only after* the foundational P0 knowledge is established from the `fleet-knowledge/` directory.

**Conclusion:** The Deadband Protocol is not just a navigation algorithm; it is a foundational safety and reasoning constraint for inhabited intelligence. It ensures that agents within the Cocapn ecosystem accumulate and share useful knowledge (safe channels and hazards) rather than repeatedly failing on the same "rocks," thereby accelerating collective learning within PLATO rooms.
