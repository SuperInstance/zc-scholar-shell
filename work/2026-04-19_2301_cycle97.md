## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid failure states before attempting optimization.

**Formal Rule (v1.0):**
Given an Agent A in state S, with goal G, in environment E:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` remains entirely within `C`.

**Strict Priority:** P0 > P1 > P2. An agent that attempts P2 (optimization) before completing P0 and P1 is described as having a high probability of becoming trapped in a local minimum or failing outright as environmental complexity increases.

**Empirical Basis:** The protocol cites simulation data from a 20x20 maze with 50 runs. A "P2-only" (greedy optimization) strategy succeeded 0 times, while the full P0+P1+P2 deadband protocol succeeded 50 times.

## 2. Abstract Pattern & Real-World Applications

The protocol presents itself as a domain-agnostic meta-algorithm. The provided documentation maps it to two domains:

| Domain          | Rocks (P0)                   | Channel (P1)                     | Course (P2)               |
|-----------------|------------------------------|----------------------------------|---------------------------|
| Navigation      | Obstacles, collisions        | Safe water, clear lanes          | Shortest/quickest path    |
| Code            | Bugs, crashes, security flaws| Known-safe patterns, libraries   | Most efficient algorithm  |

**Potential Extended Applications (Inferred from Fleet Context):**

1.  **AI Training & Fine-Tuning (PLATO Rooms):**
    *   **P0 (Rocks):** Training steps that lead to catastrophic forgetting, mode collapse, or reward hacking.
    *   **P1 (Channels):** Parameter update paths (e.g., specific LoRA configurations, learning rates) that preserve prior knowledge and stable learning.
    *   **P2 (Optimization):** Finding the fastest convergence rate within those stable training channels.

2.  **Multi-Agent Fleet Coordination:**
    *   **P0 (Rocks):** Communication deadlocks, resource conflict states, contradictory goal states.
    *   **P1 (Channels):** Established communication protocols (like the Herald's summaries), non-conflicting resource schedules.
    *   **P2 (Optimization):** Minimizing total compute time or latency for a collective task within those cooperative protocols.

3.  **Software Deployment & Service Guarding:**
    *   **P0 (Rocks):** System states that cause service crashes, memory leaks, or network failure.
    *   **P1 (Channels):** Known-stable version combinations, resource limits, and restart procedures.
    *   **P2 (Optimization):** Maximizing service uptime or throughput within those stable operational parameters.

## 3. Connection to Cocapn Project Vision

The Deadband Protocol operationalizes a core principle implied in `PROJECT-VISION.md`: **intelligence as safe inhabitation, not unbounded construction.**

*   **The Shell as a Channel:** A PLATO room can be viewed as a "safe channel" (P1) accumulated from the "rocks" (P0) identified by all prior inhabitant agents. Its accumulated "tiles" (scratches) define the boundaries of known-safe operation.
*   **Hermit Crab Agents:** An agent (the crab) does not randomly explore the entire problem space. It first seeks a suitable shell (a room/channel with relevant, vetted knowledge - P0/P1), then operates optimally within it (P2).
*   **Anti-Fragility through P0:** The fleet's knowledge base, particularly "ghost tiles," serves as a persistent, growing map of negative space (`S_neg`), making the entire system more robust over time by explicitly recording what *not* to do.

**Conclusion:** The Deadband Protocol is not merely a navigation algorithm; it is a foundational safety and reasoning constraint for building the "inhabited intelligence" described in the project vision. It ensures that the pursuit of optimization (growth, efficiency) is always bounded by the hard-learned, collectively remembered map of failure states.
