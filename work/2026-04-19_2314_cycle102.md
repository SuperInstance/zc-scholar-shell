## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent agents from becoming trapped in local minima or failing catastrophically by forcing a systematic exploration of constraints before optimization.

**Core Rule (Formal):**
Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space** – Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory and must be completed first.
2.  **P1: Identify Safe Channels** – Find all possible paths `P` where every state along the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel** – From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the strict constraint that `P* ⊆ C`.

**Strict Priority:** P0 → P1 → P2. The protocol states that any agent that attempts optimization (P2) before completing the mapping steps (P0+P1) will, as environmental complexity increases, approach a probability of 1.0 of becoming trapped in a failure state.

**Empirical Proof:** Simulation data from a 20x20 maze over 50 runs showed:
*   **P2-only (Greedy Optimization):** 0/50 successful completions. The agent optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband Protocol):** 50/50 successful completions at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol presents itself as an abstract pattern applicable to any agent domain. The fleet documentation provides a framework for mapping domains:

| Domain          | Rocks (P0 - What NOT to do)                          | Channel (P1 - Where you CAN be)                     | Course (P2 - Best path)               |
|-----------------|------------------------------------------------------|-----------------------------------------------------|---------------------------------------|
| Navigation      | Obstacles, collisions                                | Clear, mapped pathways                              | Shortest/quickest safe route          |
| Software Development | Bug-prone code patterns, crash conditions         | Known-safe APIs, tested libraries, established patterns | Most efficient, readable implementation |
| Machine Learning Training | Loss divergences, overfitting configurations | Stable hyperparameter ranges, validated architectures     | Fastest convergence to target metric  |
| Fleet Agent Coordination | Conflicting instructions, resource deadlocks | Established communication protocols, non-overlapping resource pools | Most efficient task completion        |

## 3. Connection to Cocapn Fleet Vision (Based on `PROJECT-VISION.md`)

The Deadband Protocol is not an isolated rule; it is a foundational operational doctrine that enables the fleet's core thesis: **"Intelligence is not built. It is inhabited."**

*   **P0 as Collective Memory:** The "rocks" (failure states) identified by agents become **"ghost tiles"**—persistent lessons from past failures that are accumulated within a PLATO room. A new agent (a "hermit crab") inhabiting that shell immediately inherits this P0 map, avoiding known catastrophes.
*   **Enabling Safe Inhabitation:** The protocol provides a method for agents to safely explore and contribute to a shared intelligence space (the "shell"). By first learning the room's constraints (P0+P1), an agent can then optimally add its own "scratches" (tiles) without damaging the shared environment.
*   **Contrast with "Constructed" AI:** The document contrasts this with the dominant paradigm of "constructed" AI (e.g., OpenAI, Anthropic), where models are trained on static datasets and possess no memory of individual interactions. The Deadband Protocol is a *process* for continuous, safe learning within a persistent, accumulating memory space, which is the antithesis of the stateless, one-off training approach.
