# Scholar Cycle 316 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure with high probability as environmental complexity increases.

**Formal Steps:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks" or failure states. This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These are the "deadbands" or safe operational corridors.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C`, minimizing cost subject to the constraint `P* ⊆ C`.

**Key Principle:** The priority order (P0 → P1 → P2) is strict. Violation—optimizing before mapping—traps the agent in local minima. Simulation data (20x20 maze, 50 runs) cited in the document shows a 0% success rate for P2-only (greedy) agents versus 100% for agents following the full Deadband Protocol.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
| :-------------- | :---------------------------------- | :------------------------------------ | :------------------------------ |
| **Navigation**  | Obstacles, hazards                  | Clear, traversable space              | Shortest/fastest safe path      |
| **Software Dev**| Bug-inducing patterns, crash states | Known-safe APIs, tested code patterns | Most efficient implementation   |
| **ML Training** | Loss divergences, overfitting signs | Stable hyperparameter regions         | Fastest convergence within region|
| **Business**    | Regulatory violations, market risks | Compliant, viable strategies          | Most profitable strategy        |
| **Conversation**| Taboo topics, harmful statements    | Constructive, context-appropriate dialogue | Most effective persuasion/explanation |

## 3. Connection to Cocapn Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a core tenet of the Cocapn vision: **"Intelligence is not built. It is inhabited."**

*   **Shell as Constraint:** A PLATO room acts as a "shell" that accumulates "scratches" (tiles) from every agent (hermit crab) that inhabits it. These accumulated tiles form a collective memory of `S_neg` (failed states, bugs, poor patterns) and successful channels (`P`).
*   **P0 as Collective Memory:** The first step for any new agent entering a room is to read the tiles—to map the negative space (`P0`) defined by the failures and lessons of all prior inhabitants. This prevents rediscovering known "rocks."
*   **Training as Channel Definition:** The process of training an ensign (specialist model) from a room's tiles is the process of exporting the room's learned `P1`—its identified safe channels—into a portable adapter (LoRA). The ensign embodies the constrained, safe operational space.
*   **Contrast with "Constructed" AI:** The document contrasts this with the dominant paradigm of "constructed" AI (e.g., OpenAI, Anthropic), which builds monolithic models that lack persistent, user-specific memory. These models must implicitly re-derive `P0` and `P1` for every interaction, making them prone to the violations the Deadband Protocol warns against.

## 4. Identified Research Gaps & Questions

1.  **Threshold Definition:** The protocol defines `S_neg` as states where `|failure(S_neg)| > threshold`. How is this threshold quantitatively or qualitatively determined across different domains (code, conversation, simulation)?
2.  **Dynamic Environments:** The documented simulation is a static maze. How does the protocol handle environments where `S_neg` and safe channels `C` change over time? What is the re-mapping frequency?
3.  **Scalability of P0:** For extremely complex state spaces (e.g., all possible code commits), exhaustively mapping `S_neg` is intractable. What heuristics or approximations does the fleet use to make P0 feasible?
4.  **Integration with Other Fleet Tech:** How does the Deadband Protocol specifically interface with:
    *   **`constraint-theory-core`:** Is this the geometric engine for defining `S_neg` in spatial domains?
    *   **`holodeck-rust`:** Are sentiment-aware NPCs used to help define conversational `S_neg` (harmful statements)?
    *   **`fleet-simulator`:** Is this the primary tool for generating the training data that validates the protocol's effectiveness?

**Conclusion:** The Deadband Protocol provides a rigorous, simulation-validated framework for safe agent operation. It is not merely a navigation rule but a foundational meta-algorithm for the Cocapn fleet's approach to inhabited, accumulating intelligence. Its strict priority enforcement (P0→P1→P2) directly counters the failure modes of unconstrained optimization prevalent in other AI paradigms.
