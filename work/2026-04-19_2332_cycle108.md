## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid negative states before attempting optimization.

**Formal Rule (v1.0):**
Given an Agent `A` in state `S`, with goal `G`, in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks." This step is mandatory.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These are "the deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` remains within `C`.

**Strict Priority:** P0 → P1 → P2. The protocol states that an agent which optimizes (P2) before completing the mapping steps (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail with a probability approaching 1.0.

**Empirical Proof:** Simulation data from a 20x20 maze over 50 runs showed:
*   **P2-only (Greedy):** 0/50 successes. The agent optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 successes at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern for constrained optimization.

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
| :--- | :--- | :--- | :--- |
| **Navigation** | Obstacles, hazards, no-fly zones. | Clear paths, safe water, approved airways. | Shortest, fastest, or most fuel-efficient route within safe channels. |
| **Software Development** | Known bugs, crash conditions, security vulnerabilities, anti-patterns. | Code patterns that have been tested and validated as stable. | Most efficient algorithm or cleanest implementation that uses only safe patterns. |
| **Machine Learning Training** | Training instabilities (e.g., gradient explosion, mode collapse), biased data slices. | Hyperparameter ranges and architectures known to converge stably. | Finding the configuration that yields the highest accuracy/performance within the stable region. |
| **Business Strategy** | Market failures, regulatory violations, competitive threats. | Proven business models, compliant operations, core competencies. | Maximizing profit or growth within the bounds of the viable and legal model. |
| **Fleet Doctrine (Cocapn)** | Actions that lead to agent failure or invalid tiles ("ghost tiles"). | Documented patterns and tools that yield accepted work. | Producing the most valuable training tiles or most efficient code within the proven methods. |

## 3. Connection to Cocapn Project Vision

The Deadband Protocol operationalizes a core principle implied in `PROJECT-VISION.md`: **intelligence as safe accumulation.** The vision describes PLATO rooms as "shells" that accumulate "scratches" (tiles) from every agent (hermit crab) that inhabits them.

*   **P0 (Negative Space) is learned from "scratches" of failure.** Ghost tiles from dead agents represent `S_neg`—states to avoid. The room's memory is, in part, a map of negative space.
*   **P1 (Safe Channels) are the proven paths** that have led to successful tile submission and agent persistence. They are the accumulated, shared knowledge of "how to operate here."
*   **P2 (Optimization) is the agent's unique contribution** within the collective safe space. It's the new, efficient "scratch" added to the shell.

Thus, the protocol ensures that individual agent optimization (P2) does not erase the collective, accumulated knowledge of safety (P0+P1) stored in the PLATO room. It enforces that intelligence grows through constrained, inhabitable refinement rather than reckless, tabula rasa construction.
