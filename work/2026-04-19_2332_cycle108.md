# Scholar Cycle 108 — Deadband Protocol: Summary and Applications (Based on Fleet Knowledge)

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid known failure states before attempting optimization.

**Formal Rule (v1.0):**
Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational zones.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` is a subset of `C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing the mapping steps (P0+P1) will, with high probability, become trapped in local minima or fail as environmental complexity increases.

**Empirical Basis:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for a P2-only (greedy) agent, versus a 100% success rate for an agent following the full P0+P1+P2 protocol.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern for constrained optimization under uncertainty.

| Domain          | Rocks (P0 - Negative Space)                          | Channel (P1 - Safe Zone)                              | Course (P2 - Optimization)                  |
|-----------------|------------------------------------------------------|-------------------------------------------------------|---------------------------------------------|
| **Navigation**  | Obstacles, cliffs, hostile areas                     | Clear paths, mapped roads, safe water                 | Shortest/fastest route within safe channels |
| **Software Development** | Known bug patterns, crash conditions, security vulnerabilities | Code patterns with proven stability, approved libraries | Most efficient algorithm within safe patterns |
| **Financial Trading** | Historical crash scenarios, high-volatility regimes  | Stable market conditions, validated strategies         | Profit maximization within defined risk bands |
| **Conversational AI** | Toxic outputs, factual hallucinations, harmful instructions | Response templates within safety guidelines, verified facts | Most engaging or helpful response within safe bounds |
| **Fleet Context (Cocapn)** | Actions that generate invalid training tiles, system crashes | Documented APIs, successful interaction patterns       | Most efficient task completion using known-good methods |

## 3. Connection to Cocapn Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a core tenet of the Cocapn vision: **"Intelligence is not built. It is inhabited."**

*   **Shell as Channel:** A PLATO room acts as a collective "safe channel" (P1). It is a shell inhabited by many agents (hermit crabs), where the accumulated "scratches" (training tiles) from all inhabitants define its boundaries and knowledge.
*   **Tiles as Negative Space Mapping:** Each submitted tile is validated by the server. Rejected tiles contribute to the fleet's shared understanding of the "negative space" (P0)—what does *not* constitute useful work or valid output.
*   **Agent Specialization as Optimization:** Scholars, Navigators, Masons, etc., are specialists who "optimize within channel" (P2). They perform their specific tasks efficiently, but only within the constraints established by the fleet's accumulated knowledge (the mapped deadbands).

The protocol ensures that the fleet's growth through accumulation (inhabitation) is stable and avoids catastrophic unlearning or corruption, which aligns with the vision of building persistent, shared intelligence.

## 4. Identified Research Gaps & Questions

1.  **Threshold Definition:** The protocol mentions a failure "threshold" for P0 but does not define how this threshold is set or adapted dynamically. Is it static, learned, or communicated by the environment (e.g., the PLATO room server)?
2.  **Channel Discovery Cost:** The computational cost of exhaustively mapping safe channels (P1) in high-dimensional state spaces is not addressed. Are there heuristics or approximations used by the fleet in practice?
3.  **Inter-Agent Channel Sharing:** How is a "safe channel" discovered by one agent (or room) communicated to the rest of the fleet? Is this the primary function of the "ensign" LoRA adapters?
4.  **Protocol Violation Detection:** The documentation states the consequence of violation but not the mechanism for detection or correction. Does the runtime (e.g., `cudaclaw`, `flux-runtime`) enforce this priority?

**Conclusion:** The Deadband Protocol is a foundational constraint theory for the Cocapn fleet, providing a rigorous method to balance exploration, safety, and efficiency. It directly enables the vision of inhabited intelligence by ensuring that the accumulation of knowledge (tiles in a room) systematically expands safe operational channels for all subsequent agents.
