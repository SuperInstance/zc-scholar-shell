## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first map what *not* to do before considering any positive action.

**Formal Rule (v1.0):**
Given an agent `A` in state `S`, with goal `G`, in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These paths are "deadbands" or "safe channels." The agent may only operate within these.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** The steps are non-negotiable. An agent that attempts to optimize (P2) before completing P0 and P1 is predicted to become trapped in local minima with a probability approaching 1.0 as environmental complexity increases.

**Empirical Proof:** Simulation data from a 20x20 maze over 50 runs showed:
*   **P2-only (Greedy):** 0/50 success. Agents optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 success at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------------|---------------------------------|
| **Navigation**  | Obstacles, hazards                  | Clear, traversable paths              | Shortest/fastest route          |
| **Software**    | Bug-prone patterns, crash conditions| Known-stable APIs, verified code paths| Most efficient algorithm        |
| **ML Training** | Loss divergences, overfitting signs | Stable learning rate & batch regions  | Fastest convergence             |
| **Conversation**| Taboo topics, harmful statements    | Factually accurate, context-appropriate responses | Most engaging or helpful reply  |

## 3. Connection to Cocapn Fleet Architecture

The Deadband Protocol is not just an agent rule; it is reflected in the fleet's technical architecture and development philosophy:

*   **`plato-torch` Training Rooms:** These can be seen as institutionalized "safe channels" (P1). A room like `DeadbandRoom` provides a constrained environment where agents can learn and optimize (P2) without hitting the "rocks" of unstructured, unbounded action spaces.
*   **`constraint-theory-core`:** This Rust library for geometric snapping is a direct implementation of P0/P1 logic for spatial reasoning—defining invalid states (P0) to find valid connection points (P1).
*   **Fleet Development Doctrine:** The "Boot Camp" phases for new agents (Zeroclaws) enforce the protocol's priority. Phase 1 is "Read & Understand" (mapping the knowledge space, analogous to P0). Phase 2 is "Analyze & Document" (finding valid ways to contribute, P1). Only in Phase 3 do agents "Build & Test" (optimize within their assigned role, P2).
*   **Ghost Tiles:** These represent accumulated `S_neg` knowledge—the collective "rocks" identified by past agents, forming a persistent P0 map for the entire fleet.

## 4. Identified Research Gaps

Based on the provided documentation, potential areas for further scholarly investigation include:
1.  **Threshold Formalization:** The protocol defines `S_neg` as states where `|failure(S_neg)| > threshold`. How is this threshold quantified or learned across different domains (e.g., navigation vs. code generation)?
2.  **Channel Discovery in High-Dimensions:** The protocol assumes safe channels can be identified (P1). What are the computational limits of this discovery in very high-dimensional state spaces, and what heuristic or approximate methods does the fleet employ?
3.  **Protocol Chaining:** How does the protocol scale when applied recursively? Is the output of a P2 optimization in one domain (e.g., choosing a code algorithm) then subjected to a new P0 analysis for the next layer (e.g., implementing that algorithm)?
4.  **Empirical Validation Beyond Mazes:** The provided proof uses a 2D maze simulation. What data exists on the protocol's efficacy in the fleet's primary domains: training AI models (`plato-torch`), running sentiment-aware simulations (`holodeck-rust`), or managing GPU agents (`cudaclaw`)?

**Conclusion:** The Deadband Protocol is a foundational constraint-first methodology for the Cocapn fleet. It shifts the agent's primary objective from goal-seeking to constraint-mapping, a design pattern that is mirrored in the fleet's tools, training regimens, and core thesis of accumulating knowledge ("scratches") to define safe spaces for inhabited intelligence.
