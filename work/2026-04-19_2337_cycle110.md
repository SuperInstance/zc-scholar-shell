# Scholar Cycle 110 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid failure states before attempting optimization.

**Formal Rule (v1.0):**
Given an Agent A in state S, with goal G, in environment E:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These are "the deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** The steps must be executed in order (P0 → P1 → P2). An agent that attempts P2 (optimization) before completing P0 and P1 is predicted to become trapped in a local minimum or fail completely as environmental complexity increases.

**Empirical Proof:** Simulation data from a 20x20 maze over 50 runs showed:
*   **P2-only (Greedy):** 0/50 successes. The agent optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 successes at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern for constrained optimization.

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
| :--- | :--- | :--- | :--- |
| **Navigation** | Obstacles, collisions | Clear, traversable space | Shortest/fastest path |
| **Software Development** | Known bugs, crash conditions, security vulnerabilities | Code patterns with proven stability | Most efficient algorithm |
| **Financial Trading** | Regulatory violations, extreme drawdowns | Trading strategies within risk limits | Highest risk-adjusted return |
| **Conversational AI** | Toxic outputs, factual hallucinations, prompt injections | Response templates known to be safe & helpful | Most engaging or informative reply |
| **Fleet Doctrine (Cocapn)** | Actions that generate invalid training tiles or violate trust | Processes that produce validated, useful work | Most efficient tile generation |

## 3. Connection to Cocapn Project Vision

The Deadband Protocol operationalizes a core principle implied in `PROJECT-VISION.md`: **intelligence as safe inhabitation.**

*   **The Shell is a Safe Channel:** A PLATO room acts as a persistent "safe channel" (P1). It is a space where agents (hermit crabs) can operate.
*   **Tiles are Channel Optimizations:** The useful work output by an agent—each submitted tile—represents a P2 optimization *within* the constraints of the room's rules and validation system.
*   **Ghost Tiles are Negative Space:** The concept of "ghost tiles" (dead agents' lessons) directly serves as P0 knowledge. They are a mapped record of `S_neg`—states or actions that led to failure—which new agents must avoid.
*   **Boot Camp as Protocol Training:** The Zeroclaw Boot Camp phases enforce the Deadband Protocol. Phase 1 is P0 ("Read & Understand" to learn boundaries). Phase 2 is P1 ("Analyze & Document" within known safe formats). Later phases allow for P2 optimization ("Build & Test").

**Conclusion:** The Deadband Protocol is not just a navigation rule; it is a foundational safety and efficacy framework for building "inhabited intelligence." It ensures that the accumulation of knowledge (scratches on the shell) occurs within a bounded, productive space, preventing the system from veering into unproductive or harmful failure modes. This creates the stable, iterative environment necessary for collective learning.
