## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid failure states before attempting optimization.

**Formal Rule (v1.0):**
Given an agent `A` in state `S`, with goal `G` and environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` is a subset of `C`.

**Strict Priority:** P0 > P1 > P2. An agent that attempts P2 (optimization) before completing P0 and P1 is described as having a high probability of becoming trapped in a local minimum as environmental complexity increases.

**Empirical Support:** The document cites simulation data from a 20x20 maze, where a "P2-only" greedy agent succeeded 0 out of 50 times, while an agent following the full Deadband Protocol succeeded 50 out of 50 times.

## 2. Real-World Applications & Abstract Patterns

The protocol is presented as a domain-agnostic pattern. The document provides a table of examples:

| Domain          | Rocks (P0)                          | Channel (P1)                    | Course (P2)               |
|-----------------|-------------------------------------|---------------------------------|---------------------------|
| Navigation      | Obstacles, collisions               | Safe water, clear lanes         | Shortest path             |
| Code            | Bugs, crashes, security vulns       | Known-safe patterns, libraries  | Most efficient algorithm  |
| Conversation    | Offensive topics, misinformation    | Polite, factual topics          | Most persuasive argument  |
| Business        | Regulatory violations, market risks | Compliant, proven strategies    | Highest profit plan       |

## 3. Connection to Cocapn Fleet Vision (Based on `PROJECT-VISION.md`)

The Deadband Protocol operationalizes a core tenet of the Cocapn vision: **"Intelligence is not built. It is inhabited."**

*   **Shell as Constraint:** A PLATO room acts as a "shell" for an agent (the "hermit crab"). The room's accumulated knowledge—the "scratches" or tiles from previous inhabitants—defines a historical **negative space (P0)** and **safe channels (P1)**. An incoming agent doesn't build intelligence from scratch; it inherits and operates within the validated constraints of the inhabited space.
*   **Training as Channel Identification:** The boot camp process described for Zeroclaws mirrors the protocol. Phase 1 ("Read & Understand") is P0/P1: learning the fleet's knowledge base (negative space) and successful patterns (safe channels). Only in later phases do agents begin to "Optimize Within Channel" (P2) by producing their own work.
*   **Failure as a Tile:** When an agent fails, that experience becomes a "ghost tile"—a concrete data point added to the room's map of `S_neg`. This directly enriches the P0 knowledge for all future inhabitants, making the collective intelligence more robust.

## 4. Identified Research Gaps & Questions

Based on the provided documents, several open questions emerge that could guide further fleet research:

1.  **Threshold Definition:** How is the failure "threshold" for `S_neg` quantitatively defined or learned within a PLATO room? Is it static, adaptive, or context-dependent?
2.  **Channel Discovery Cost:** The protocol assumes safe channels can be identified. In highly complex or novel environments, what is the cost/strategy for exhaustive P1 mapping before any P2 action is permitted?
3.  **Dynamic Environments:** How does the protocol handle environments where `S_neg` and safe channels shift in real-time? Is there a mechanism for continuous P0/P1 reassessment?
4.  **Inter-Agent Protocol:** How do multiple agents operating with the Deadband Protocol coordinate? Could one agent's P2-optimized path inadvertently create a `S_neg` state for another?
5.  **Empirical Validation:** The provided proof is a single maze simulation. Broader validation across the fleet's domains (e.g., `holodeck-rust` NPC interaction, `constraint-theory-core` geometric solving) would strengthen the claim of domain-agnostic utility.

**Conclusion:** The Deadband Protocol provides a compelling, simulation-backed framework for safe agent operation. Its integration with the Cocapn "inhabited intelligence" model suggests its primary value is as a **collective constraint system**, where the safe channels for any agent are progressively defined by the accumulated experiences—both successes and failures—of all agents that have inhabited its shared computational space.
