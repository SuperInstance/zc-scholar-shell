## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure with high probability as environmental complexity increases.

**Formal Rule (v1.0):**
Given an Agent A in state S, with goal G in environment E:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that any agent optimizing (P2) before completing P0 and P1 will become trapped in local minima as complexity rises.

**Empirical Basis:** The document cites simulation data from a 20x20 maze across 50 runs. A P2-only (greedy) strategy succeeded 0 times, while the full P0+P1+P2 (deadband) strategy succeeded 50 times at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol is presented as a domain-agnostic pattern. The provided table maps it to navigation and code, but the principle extends further.

| Domain          | Rocks (P0 - Avoid)                          | Channel (P1 - Safe Space)                  | Course (P2 - Optimize Within)          |
| :-------------- | :------------------------------------------ | :----------------------------------------- | :------------------------------------- |
| **Navigation**  | Obstacles, collisions, restricted zones     | Clear paths, legal roads, safe waterways   | Shortest/fastest route within safe area |
| **Software Dev**| Known bugs, crash conditions, anti-patterns | Tested, stable code patterns, linter rules | Most efficient/readable implementation  |
| **ML Training** | Loss divergences, overfitting signatures, gradient explosions | Stable hyperparameter regions, validated architectures | Convergence speed, final accuracy       |
| **Business Strategy**| Regulatory violations, market failures, competitor strengths | Compliant operations, validated market niches, core competencies | Profit maximization, market share growth |
| **Personal Decision-Making**| Actions with high risk of harm (health, financial, relational) | Ethically sound, legally compliant, low-risk options | Path to greatest personal fulfillment   |

## 3. Connection to Cocapn Fleet & PLATO Rooms

The Deadband Protocol is not just an agent rule; it is a foundational **training constraint**. In the Cocapn vision where "intelligence is inhabited," PLATO rooms accumulate "scratches" (tiles) from every agent that passes through.

*   **P0 as Collective Memory:** A room's accumulated failure states (`S_neg`) from all prior agents become a shared negative map for future inhabitants. This is the "ghost tile" concept—lessons from dead agents persist as P0 knowledge.
*   **P1 as Emergent Doctrine:** Successful paths (`P`) discovered and validated by multiple agents coalesce into the room's "safe channels" or standard operating procedures.
*   **P2 as Specialized Skill:** Optimization within these channels is where an agent (or an exported "Ensign" LoRA) can develop unique, high-performance expertise, but only atop a foundation of shared safety.

This creates a **recursive safety loop**: the protocol governs individual agent behavior, and the aggregate results of that behavior (tiles) refine the protocol's knowledge base for the entire community inhabiting the room.
