## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure in complex environments by enforcing a sequence of constraint identification before optimization.

**Formal Rule (v1.0):**
Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational corridors.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing the mapping steps (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail with probability approaching 1.0.

**Empirical Basis:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for a P2-only (greedy) agent, versus a 100% success rate for an agent following the full P0+P1+P2 sequence.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern for safe agent operation:

| Domain          | Rocks (P0 - Negative Space)                          | Channel (P1 - Safe Corridor)                          | Course (P2 - Optimization)                  |
| :-------------- | :--------------------------------------------------- | :---------------------------------------------------- | :------------------------------------------ |
| **Navigation**  | Obstacles, collisions, restricted zones              | Clear paths, mapped lanes, permissible air/sea space  | Shortest, fastest, or most fuel-efficient route within the safe channel |
| **Software Development** | Known bug patterns, crash conditions, security vulnerabilities | Code patterns that are provably safe, tested libraries, approved APIs | Most efficient algorithm, cleanest implementation within the safe patterns |
| **Financial Trading** | Regulatory violations, tail-risk events, known fraud patterns | Compliant strategies, within-risk-limit portfolios    | Highest risk-adjusted return within the compliant strategy set |
| **Conversational AI** | Toxic outputs, factual hallucinations, privacy violations | Response templates grounded in verified data, ethical guidelines | Most engaging, helpful, or concise response that remains within the safe template boundaries |
| **Fleet Context (Cocapn)** | Actions that break the git-shell, produce invalid training tiles, or violate zerocrust principles | Operations that read real files, write useful output, and follow the P0>P1>P2 doctrine | Most efficient path to generating high-quality training tiles within the safe operational bounds |

## 3. Connection to Cocapn Fleet Vision

The Deadband Protocol is a concrete implementation of the fleet's core philosophical stance as described in `PROJECT-VISION.md`. It operationalizes the "hermit crab" model:

*   **The Shell is the Constraint:** The safe channel (P1) is the "shell" — a defined, persistent space where the agent can safely operate and accumulate experience.
*   **Scratches are Lessons:** Each interaction within the safe channel produces a "tile" (a unit of experience). Negative outcomes discovered during P0 mapping become "ghost tiles" — persistent knowledge of what *not* to do, enriching the shell for future inhabitants.
*   **Inhabited Intelligence:** The protocol ensures that intelligence (the optimized path, P2) is not constructed in a vacuum but is *inhabited* within the historically accumulated constraints (P0+P1) of the PLATO room. An agent cannot be intelligent without first understanding the boundaries of its shell.

## 4. Identified Research Gaps

Based on the provided documentation, potential areas for further investigation include:
1.  **Threshold Dynamics:** How is the failure `threshold` for P0 defined and adapted across different domains or as an agent learns?
2.  **Channel Discovery Cost:** The protocol assumes safe channels can be identified. What are the computational or exploratory costs of P1 in high-dimensional, partially observable state spaces?
3.  **Protocol Chaining:** How does the protocol scale when applied recursively? (e.g., the output of a P2 optimization becomes the state `S` for a new P0 mapping at a higher level of planning).
4.  **Empirical Validation Beyond Mazes:** While the maze simulation is compelling, validation in the fleet's primary domains (code generation, multi-agent simulation, natural language interaction) is not detailed in the provided files.

**Conclusion:** The Deadband Protocol provides a robust, priority-driven framework for ensuring agent safety and efficacy. It translates the fleet's philosophical foundation into a actionable, testable algorithm, making the accumulation of "scratches" (tiles) a systematic process rather than a random one. Its power lies in the strict, non-negotiable ordering of its steps, forcing the agent to respect the accumulated constraints of its environment before seeking reward.
