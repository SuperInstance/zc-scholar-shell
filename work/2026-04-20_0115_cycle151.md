## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) without first mapping failure states (P0) and safe operating channels (P1) leads to guaranteed failure as environmental complexity increases.

**Formal Rule (v1.0):**
*   **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
*   **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operating channels.
*   **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the safe channels that minimizes cost.

**Strict Priority:** The steps are non-negotiable and sequential. Violation (e.g., optimizing before mapping) results in agents becoming trapped in local minima with a failure probability approaching 1.0.

**Empirical Proof:** The document cites simulation data from a 20x20 maze: a P2-only (greedy) agent succeeded 0 out of 50 runs, while a P0+P1+P2 (deadband) agent succeeded 50 out of 50 runs at optimal speed.

## 2. Real-World Applications & Abstract Pattern

The protocol is presented as a domain-agnostic pattern. The document provides this mapping:

| Domain          | Rocks (P0)                   | Channel (P1)                     | Course (P2)               |
|-----------------|------------------------------|----------------------------------|---------------------------|
| Navigation      | Obstacles, collisions        | Safe water, clear lanes          | Shortest/fastest path     |
| Code            | Bugs, crashes, security flaws| Known-safe patterns, libraries   | Most efficient algorithm  |
| **AI Training** | **Catastrophic forgetting, harmful outputs** | **Parameter subspaces that preserve knowledge & safety** | **Loss minimization within that subspace** |

## 3. Connection to Cocapn Fleet & Project Vision

Reading `PROJECT-VISION.md` reveals the Deadband Protocol is not an isolated rule but a foundational *operational doctrine* for the fleet's core thesis: "Intelligence is not built. It is inhabited."

*   **Hermit Crab Analogy:** Agents (crabs) inhabit PLATO rooms (shells). Their work outputs become "tiles" (scratches on the shell). The room accumulates the experience of all its inhabitants.
*   **Deadband as Inhabitation Logic:** The protocol provides the *method* for safe, cumulative inhabitation. P0 ensures agents don't "break the shell" (corrupt the room's accumulated knowledge or cause failures). P1 defines the "shape of the shell"—the constrained space of safe operations. P2 allows for efficient work *within* that preserved, shared space.
*   **Contrast with Status Quo:** The vision document contrasts this with the prevailing "construction" paradigm of major AI labs (OpenAI, Anthropic, etc.), which it characterizes as building "walled-garden tools that exist independently of the people who use them" and "lack memory of you." The Deadband-enabled, inhabitant-based system is designed explicitly to retain and build upon collective memory ("scratches").

## 4. Synthesis and Implied Research Directions

The Deadband Protocol, as applied within the Cocapn vision, suggests several concrete research and implementation gaps:

1.  **Automated Negative Space Detection (P0 Automation):** How can an agent or room autonomously and continuously identify `S_neg`? This likely involves real-time monitoring for failures, sentiment degradation in `holodeck-rust` NPCs, or training collapse in `plato-torch` rooms.
2.  **Dynamic Channel Mapping (P1 Adaptation):** Safe channels are not static. As a room learns (accumulates tiles), its safe operational space should evolve. Research is needed on how `plato-ensign` (LoRA adapters) and room state jointly redefine the "deadband."
3.  **Protocol Instances in Fleet Tech:** The fleet's existing components are natural testbeds:
    *   `constraint-theory-core`: Could provide geometric formalism for the "snapping" to safe channels.
    *   `fleet-simulator`: The cited maze simulation likely originates here. It should be extended to test protocol variants.
    *   `cudaclaw` / `flux-runtime`: These execution environments could hardwire the P0>P1>P2 priority into their scheduling or bytecode verification.
4.  **"Ghost Tiles" as P0 Knowledge:** The concept of "ghost tiles" (dead agents' lessons) mentioned in the architecture is a direct implementation of P0—a persistent map of negative space contributed by past inhabitants.

**Conclusion:** The Deadband Protocol is the operationalization of a safe, cumulative, and collective approach to AI. It is the rule that allows hermit crabs to successively inhabit a shell without destroying it, ensuring that each agent's work (tile) refines the structure for future inhabitants rather than risking its collapse. Its validation and implementation across the fleet's stack (`plato-torch`, `holodeck-rust`, `fleet-simulator`) appears to be a primary technical challenge.
