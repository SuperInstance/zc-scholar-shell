## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid states of high failure probability before attempting optimization.

**Formal Rule (v1.0):**
Given an agent A in state S, with goal G, in environment E:
1.  **P0: Map the Negative Space** — Identify all states `S_neg` where the probability of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels** — Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are "the deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel** — From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` is a subset of `C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing mapping (P0+P1) will, as environmental complexity increases, become trapped in local minima with a probability approaching 1.0.

**Empirical Proof:** Simulation data from a 20x20 maze over 50 runs showed:
*   **P2-only (Greedy):** 0/50 successes. The agent optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 successes at optimal speed.

## 2. Real-World Applications & Abstract Pattern

The protocol presents an abstract pattern applicable to any agent domain. The provided documentation maps it to two domains:

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
|--------|-----------|-------------|-------------|
| Navigation | Obstacles (walls, cliffs) | Safe water/passable terrain | Shortest/quickest path |
| Software Development | Bugs, crashes, security vulnerabilities | Known-safe code patterns, libraries, and architectures | Most efficient or performant implementation |

**Potential Extended Applications:**
*   **Financial Trading:** P0: Identify market conditions (e.g., flash crashes, illiquidity) that cause system failure. P1: Define trading strategies that remain operational in those conditions. P2: Optimize for profit within those safe strategies.
*   **Robotics Manipulation:** P0: Map joint configurations and object poses that lead to collisions or torque limits. P1: Plan trajectories that avoid these states. P2: Optimize for speed or energy efficiency along those trajectories.
*   **Conversational AI:** P0: Identify user inputs or response types that lead to harmful, biased, or nonsensical outputs. P1: Constrain the response generation space to avoid those regions. P2: Optimize for coherence, helpfulness, or style within the safe space.

## 3. Connection to Cocapn Fleet Vision

The Deadband Protocol is a concrete instantiation of the broader Cocapn thesis that "intelligence is inhabited." As described in `PROJECT-VISION.md`, PLATO rooms are shells that accumulate "scratches"—lessons and interactions—from every agent that inhabits them.

The Deadband Protocol's "rocks" (`S_neg`) are the negative knowledge—the "what not to do"—that constitutes a critical form of accumulated experience. A PLATO room trained on Deadband interactions would not just learn successful paths (P2 optima), but would primarily internalize the hard boundaries of failure (P0). This creates a persistent, shared memory of constraint, allowing new "hermit crab" agents to immediately inhabit a shell pre-scratched with the lessons of prior failures, dramatically accelerating safe learning.

**Conclusion:** The Deadband Protocol is more than a navigation rule; it is a foundational principle for building robust, inhabited intelligence. It ensures that optimization is always bounded by accumulated, communal knowledge of failure, aligning perfectly with the fleet's vision of learning through persistent, shared experience rather than isolated model construction.
