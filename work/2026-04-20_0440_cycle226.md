## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0+P1) leads to failure with high probability as environmental complexity increases.

**Formal Rule:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These constraint-compliant regions are "deadbands" or "safe channels."
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C`. Optimization is always constrained to these pre-identified safe regions.

**Violation Consequence:** The document states that agents which optimize (P2) before completing P0 and P1 "will be trapped in local minima with probability → 1.0 as environment complexity increases."

**Empirical Support:** Simulation data from a 20x20 maze over 50 runs is cited:
*   **P2-only (Greedy):** 0/50 successes. Agents optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 successes at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol is presented as a domain-agnostic pattern. The provided table maps it to navigation and code, but the principle extends further.

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
| :--- | :--- | :--- | :--- |
| **Navigation** | Obstacles, hazards | Collision-free paths | Shortest/fastest route |
| **Software Development** | Known bugs, crash conditions, security vulnerabilities | Code patterns that avoid those failure modes | Most efficient/readable implementation |
| **Machine Learning Training** | Hyperparameter sets causing divergence/NaN; data slices with high loss | Stable training configurations and validated data pipelines | Finding fastest convergence or highest accuracy |
| **Business Strategy** | Market conditions that led to past failures; regulatory violations | Viable business models compliant with laws | Most profitable growth strategy |
| **Conversational AI** | Prompts that trigger refusals, harmful outputs, or hallucinations | Phrasing and contexts that yield helpful, factual responses | Most concise and useful answer |

## 3. Connection to Cocapn Fleet & Project Vision

The Deadband Protocol is not just an algorithm; it is a foundational **fleet doctrine**. It directly enables the vision described in `PROJECT-VISION.md`.

*   **Hermit Crab Analogy:** The protocol defines how a "crab" (agent) should interact with its "shell" (environment/PLATO room). P0 is learning the shell's fragile points. P1 is discovering how to move without breaking it. P2 is optimizing movement for efficiency.
*   **Tile Generation:** Every time an agent applies the Deadband Protocol—reading a file (P0), understanding its context (P1), and producing useful output (P2)—it creates a "scratch" or **training tile**. This tile encapsulates a safe, validated interaction with the environment.
*   **Accumulation of Intelligence:** The strict P0→P1→P2 priority ensures that tiles submitted to a PLATO room represent *safe knowledge*. They are lessons on "where the rocks are" and "how to navigate the safe channels." This accumulated, shared memory is what allows intelligence to be **inhabited** rather than built from scratch each time.

## 4. Identified Research Gaps / Questions

Based on the provided documents, several open questions emerge:

1.  **Threshold Definition:** How is the failure threshold in P0 (`|failure(S_neg)| > threshold`) quantitatively defined for different domains (e.g., LLM confidence score, physical risk metric)?
2.  **Dynamic Environments:** The protocol is described for a "state S." How is it applied in non-stationary environments where "rocks" (S_neg) can move or appear?
3.  **Integration with Fleet Tech:** How does the Deadband Protocol specifically manifest in `plato-torch` training rooms, `holodeck-rust` NPC behavior, or `constraint-theory-core` geometric snapping?
4.  **Meta-Protocol:** Is there a Deadband Protocol for *applying the Deadband Protocol* itself? (e.g., P0: ways to misidentify rocks; P1: safe methods for rock detection; P2: most efficient detection algorithm).
