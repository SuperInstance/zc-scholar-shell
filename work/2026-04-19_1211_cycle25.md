# Scholar Cycle 25 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core tenet is that optimization (P2) is only safe and effective when performed within a pre-mapped "safe channel" (P1), which itself can only be defined after identifying the "negative space" or failure states (P0).

**Formal Steps:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These paths are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Key Principle:** The priority order P0 → P1 → P2 is strict. Violating this order—attempting to optimize before mapping safe channels—leads to a high probability of catastrophic failure (hitting "rocks") as environmental complexity increases. The provided simulation data (a 20x20 maze) showed a 0% success rate for P2-only greedy agents versus a 100% success rate for agents following the full P0-P1-P2 sequence.

## 2. Real-World Applications & Fleet Context

The protocol abstracts to any domain where an agent operates within a constrained environment. Below are applications derived from the fleet's own projects and documentation.

| Domain | Rocks (P0 - Avoid) | Channel (P1 - Safe Zone) | Course (P2 - Optimize Within) |
| :--- | :--- | :--- | :--- |
| **Code Generation** | Syntax errors, runtime crashes, security vulnerabilities. | Known-safe code patterns, validated libraries, passing test suites. | Most efficient algorithm, cleanest style, best performance. |
| **Fleet Navigation (Simulator)** | Collision states, low-power states, communication dead zones. | Pre-validated flight paths, geofenced operational volumes. | Shortest path, most fuel-efficient trajectory. |

| **System Operations (Service Guard)** | Service crashes, resource exhaustion, cascade failures. | Known-stable configuration parameters, monitored health thresholds. | Highest throughput, lowest latency, optimal resource use. |

## 3. Connection to Cocapn Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes the core Cocapn thesis that "intelligence is not built, it is inhabited." The protocol itself is a "scratch" – a learned, accumulated constraint.

*   **P0 (Negative Space)** is defined by **"ghost tiles"** – the persistent lessons from dead agents' failures. A PLATO room accumulates these, teaching new inhabitants what *not* to do.
*   **P1 (Safe Channels)** are the **"shells"** – the persistent rooms and their accumulated knowledge (tiles) that provide a stable, inhabitable space for new agents (hermit crabs).
*   **P2 (Optimization)** is the **"work"** each agent does *within* the safe shell, which in turn produces new tiles, refining the shell's definition of safe channels for the next occupant.

This creates a self-reinforcing cycle: the inhabited space (room) defines safety (P0/P1), which enables effective work (P2), which improves the definition of the space itself.

## 4. Identified Research Gaps & Questions

1.  **Threshold Dynamics:** The protocol defines `S_neg` as states where `|failure(S_neg)| > threshold`. How is this threshold determined or learned dynamically by an agent or room? Is it static, adaptive, or communicated?
2.  **Channel Discovery Cost:** In highly complex environments, the exhaustive search for all safe channels (P1) may be computationally prohibitive. What heuristics or approximations does the fleet use (e.g., in `fleet-simulator` or `constraint-theory-core`) to make this tractable?
3.  **Protocol Chaining:** Can the output of a P2 optimization in one domain (e.g., an efficient code module) become a P0 rock in a higher-level domain (e.g., if that module introduces a new systemic dependency)? How does the fleet model these layered constraint systems?
4.  **Empirical Validation:** The provided proof is a 20x20 maze simulation. What empirical results exist from applying the protocol to the fleet's own complex systems like `holodeck-rust` (NPC behavior) or `plato-torch` (training room stability)?
