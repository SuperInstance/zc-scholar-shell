## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid states of high failure probability before attempting optimization.

**Formal Steps:**
1.  **P0: Map the Negative Space.** Identify all states (`S_neg`) where the probability or magnitude of failure exceeds a defined threshold. These are "rocks." This step is mandatory and non-negotiable.
2.  **P1: Identify Safe Channels.** Find all possible paths (`P`) where every state along the path is *not* in `S_neg`. These paths are "deadbands" or safe channels. The agent's operational space is constrained to these channels.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete may the agent select the optimal path (`P*`) from within the set of safe channels, minimizing cost or maximizing reward.

**Core Doctrine:** The priority order (P0 → P1 → P2) is strict. Violating this order—for example, optimizing (P2) before mapping the negative space (P0)—leads to a high probability of becoming trapped in local minima or encountering failure as environmental complexity increases. Simulation data (a 20x20 maze, 50 runs) cited in the document supports this: a P2-only (greedy) approach succeeded 0 times, while the full P0+P1+P2 protocol succeeded 50 times.

## 2. Real-World Applications & Abstract Patterns

The protocol is presented as a domain-agnostic pattern. The document provides a framework for translating it:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------------|---------------------------------|
| Navigation      | Obstacles, hazards                  | Clear, safe paths                     | Shortest/fastest route          |
| Software Development | Bug-prone patterns, crash conditions | Known-safe APIs, tested code patterns | Most efficient algorithm        |
| Financial Trading | High-volatility events, flash crashes | Stable market regimes                 | Highest risk-adjusted return    |
| Conversation (AI) | Toxic outputs, factual hallucinations | Ethically and factually bounded responses | Most helpful, engaging response |

## 3. Connection to Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a key aspect of the fleet's "inhabited intelligence" thesis. It provides a concrete **rule for survival and learning** that can be accumulated as "scratches" within a PLATO room.

*   **Hermit Crab Analogy:** The protocol itself is a foundational "scratch" on the shell—a learned, hard-won lesson about how to avoid destruction. An agent that violates P0 and hits a "rock" contributes a failure tile (`S_neg`) to the room's memory, enriching the collective negative space map for future occupants.
*   **Training Tiles:** Successful applications of the protocol (successful traversals of a maze, bug-free code commits) become P1 and P2 training tiles. These tiles train "ensign" models not just on how to optimize, but on *where it is safe to optimize*.
*   **Constraint vs. Construction:** The protocol aligns with the vision's rejection of unconstrained "construction." It does not seek to build an omnipotent agent. Instead, it seeks to define the safe operating channels (the "shell") within which a specialized intelligence can effectively and safely inhabit its domain.
