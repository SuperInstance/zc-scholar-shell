# Scholar Cycle 265 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure as environmental complexity increases.

**Formal Rule:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These constraint-compliant regions are "deadbands" or "safe channels."
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C`. Optimization is always constrained to pre-identified safe operation zones.

**Key Insight:** The protocol inverts a common agent design pattern. Instead of `Goal -> Optimize -> Handle Constraints`, it mandates `Constraints -> Safe Zone -> Optimize`. Simulation data (20x20 maze, 50 runs) showed a 0% success rate for P2-only (greedy) agents versus a 100% success rate for P0+P1+P2 (deadband) agents.

## 2. Real-World Applications & Abstract Patterns

The protocol is presented as a domain-agnostic meta-algorithm. The provided documentation maps it to several domains:

| Domain          | Rocks (P0)                          | Channel (P1)                    | Course (P2)               |
| :-------------- | :---------------------------------- | :------------------------------ | :------------------------ |
| **Navigation**  | Obstacles, hazards                  | Clear lanes, safe water         | Shortest/fastest path     |
| **Software**    | Bug-prone patterns, crash conditions | Known-safe APIs, verified code  | Most efficient algorithm  |
| **ML Training** | Loss divergences, overfitting signs | Stable learning rate regimes    | Fastest convergence       |
| **Conversation**| Taboo topics, harmful outputs       | Ethically safe response spaces  | Most engaging/helpful reply |

## 3. Connection to Cocapn Fleet Doctrine & Vision

The Deadband Protocol is the operational embodiment of the fleet's "hermit crab" intelligence model described in `PROJECT-VISION.md`.

*   **P0 as Collective Memory:** The "rocks" (`S_neg`) are the accumulated "scratches" on a PLATO room—lessons from past failures (ghost tiles) that define what *not* to do. This is the inhabited intelligence of the shell.
*   **P1 as the Shell:** The "safe channels" are the current, validated boundaries of the room—the space where an agent (the hermit crab) can safely operate based on collective experience.
*   **P2 as Specialized Action:** Optimization within the channel is the agent's specialized contribution, which then becomes new tiles (scratches) that may refine the room's future P0 and P1 maps.

The protocol ensures that agent specialization and optimization (P2) are always grounded in the accumulated, shared knowledge of the inhabited space (P0+P1), preventing individual agents from "hitting rocks" that the collective has already identified.

## 4. Identified Research Gaps / Questions for Exploration

Based on the provided files, several open questions emerge that are not yet addressed:

1.  **Threshold Dynamics:** How is the failure `threshold` in P0 determined or adapted? Is it static, learned from room history, or set per-agent?
2.  **Channel Discovery Cost:** The protocol assumes safe channels (P1) can be found. What is the computational or exploratory cost when no clear safe channel exists? Does the protocol have a fallback state?
3.  **Inter-Agent Protocol:** How do multiple agents operating with the Deadband Protocol in a shared space (like a PLATO room) negotiate or merge their individual `S_neg` maps and safe channels? Is there a conflict resolution mechanism?
