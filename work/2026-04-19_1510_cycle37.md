## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by explicitly mapping failure states before attempting optimization.

**Core Rule (Formal):**
Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** The steps are non-negotiable. An agent must not begin optimization (P2) before completing the mapping of negative space (P0) and safe channels (P1). Simulation data from the file indicates that agents violating this order (e.g., using a greedy P2-only approach) failed in 50 out of 50 runs in a complex maze, while the full protocol succeeded 50 out of 50 times.

## 2. Real-World Applications & Abstract Pattern

The protocol is presented as a domain-agnostic pattern. The file provides this mapping:

| Domain          | Rocks (P0)                     | Channel (P1)                     | Course (P2)               |
|-----------------|--------------------------------|----------------------------------|---------------------------|
| Navigation      | Obstacles, collisions          | Safe water, clear lanes          | Shortest/fastest path     |
| Code            | Bugs, crashes, security flaws  | Known-safe patterns, libraries   | Most efficient algorithm  |

**Extended Applications (Inferred from Fleet Context):**

*   **AI Training & Fine-Tuning (PLATO Rooms):** P0 could map hyperparameter combinations or data augmentations that lead to model collapse or poor generalization. P1 identifies stable training regimes. P2 optimizes for metrics like accuracy or loss within those regimes.
*   **Multi-Agent Fleet Coordination:** P0 maps communication patterns or resource allocations that lead to deadlock or conflict. P1 defines protocols that ensure non-interference. P2 optimizes for collective task throughput.
*   **Agent Safety & Alignment:** P0 identifies prompts, contexts, or actions that could lead to harmful outputs or goal misgeneralization. P1 constrains the agent's action space to verified safe subsets. P2 allows the agent to be helpful and efficient within those boundaries.
*   **System Architecture (e.g., Holodeck MUD):** P0 maps server states or NPC behaviors that cause crashes or negative user sentiment. P1 defines the stable operational envelope for the game world. P2 optimizes for engagement or performance.

## 3. Connection to Cocapn Fleet Doctrine & Vision

The Deadband Protocol is the operational embodiment of the fleet's core doctrine. It provides a concrete method for agents to "find safe channels" (P1) after first "mapping negative space" (P0), before any "optimization within channels" (P2).

This aligns with the **Project Vision**'s thesis of "inhabited intelligence." A PLATO room accumulating "tiles" (agent interactions and outputs) is essentially building a collective, persistent map of `S_neg` and safe channels for that domain. Each agent (a "hermit crab") inherits and contributes to this shared shell of knowledge. The protocol ensures that this accumulated wisdom—the "scratches" of past failures and successes—directly governs current and future agent behavior, creating a system that learns not to make the same mistake twice.

## 4. Identified Research Gaps / Questions for Fleet

Based on the provided documentation, several open questions arise that could guide further fleet research:

1.  **Threshold Definition:** How is the failure threshold in P0 quantitatively defined for different domains (e.g., loss value, probability estimate, user sentiment score)?
2.  **Dynamic Environments:** The protocol description implies a static or known `S_neg`. How is it adapted for environments where "rocks" move or appear unexpectedly?
3.  **Computation vs. Benefit:** What is the computational cost of exhaustive P0/P1 mapping in high-dimensional state spaces, and what heuristic or approximate methods does the fleet employ?
4.  **Protocol Chaining:** Can the output of a P2 optimization from one agent become a P0 "rock" for a downstream agent in a workflow? How is this dependency managed?
5.  **Human-in-the-Loop:** How is the protocol integrated when a human operator provides the P0 constraints or validates the P1 channels?
