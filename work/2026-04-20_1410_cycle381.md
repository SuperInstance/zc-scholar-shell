## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping failure states (P0) and safe operating channels (P1) leads to catastrophic failure as environmental complexity increases.

**Formal Rule (v1.0):**
- **P0: Map the Negative Space** – Identify all states `S_neg` where the probability or cost of failure exceeds a threshold. These are "rocks." This step is mandatory and non-negotiable.
- **P1: Identify Safe Channels** – Find all possible paths `P` where every state `s` within the path is not in `S_neg`. These are "deadbands" or safe operating corridors.
- **P2: Optimize Within Channel** – From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`. Optimization is only permitted within the bounds established by P0 and P1.

**Priority Order:** P0 → P1 → P2. The protocol states that any agent that attempts P2 before completing P0 and P1 will become trapped in local minima with a probability approaching 1.0 as the environment's complexity grows.

**Empirical Proof:** Simulation data from a 20×20 maze over 50 runs showed:
- **P2-only (Greedy Optimization):** 0/50 successful completions. The agent optimized directly for the goal and hit obstacles every time.
- **P0+P1+P2 (Deadband Protocol):** 50/50 successful completions at optimal speed.

## 2. Real-World Applications & Abstract Pattern

The protocol is presented as a domain-agnostic pattern. The provided table illustrates this abstraction:

| Domain          | Rocks (P0)                     | Channel (P1)                    | Course (P2)               |
|-----------------|--------------------------------|---------------------------------|---------------------------|
| Navigation      | Obstacles, hazards             | Safe water, clear lanes         | Shortest path             |
| Software Development | Bugs, crashes, security flaws | Known-safe patterns, linted code | Most efficient algorithm  |
| Business Strategy | Market failures, legal risks  | Validated business models       | Profit maximization       |
| Machine Learning | Overfitting, mode collapse    | Stable training regimes         | Loss minimization         |

## 3. Synthesis with Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a core tenet of the Cocapn vision: **"Intelligence is not built. It is inhabited."**

*   **The Shell as a Safe Channel:** A PLATO room, as a persistent, shared memory space, can be viewed as an accumulated "safe channel" (P1). It contains the validated knowledge tiles (successful paths) and ghost tiles (recorded failures, or "rocks" - P0) from all prior inhabitants.
*   **The Hermit Crab Agent:** An agent (the "crab") does not construct intelligence from scratch. It first **inhabits** a shell (reads the room's P0 and P1 knowledge). Its own work and optimization (P2) are constrained by this inherited understanding. Its contributions then become new tiles, refining the shell's map of safe channels and negative space for future agents.
*   **Contrast with Current AI Paradigm:** The document contrasts this with the "construction" paradigm of major AI labs, which often resembles a pure "P2-only" approach: massive, unconstrained optimization on raw data, frequently leading to predictable failures (hallucinations, instability) that are not accumulated as shared, avoidable knowledge for subsequent iterations.

## 4. Identified Research Gaps & Questions

1.  **Threshold Definition:** The protocol defines `S_neg` as states where `|failure(S_neg)| > threshold`. How is this threshold determined or learned dynamically by an agent? Is it a fixed parameter, or can it be adapted from a room's ghost tiles?
2.  **Channel Discovery Complexity:** In high-dimensional, non-stationary environments (e.g., real-world markets, open-ended dialogue), exhaustively mapping safe channels (P1) may be computationally intractable. What heuristics or approximations does the fleet employ?
3.  **Integration with Learning:** The protocol is presented as a decision rule. How does it integrate with iterative learning processes? Does a training loop in `plato-torch` constitute a series of P0-P1-P2 cycles, where each epoch refines the failure map?
4.  **Empirical Scope:** The provided proof is from a discrete 2D maze simulation. Are there documented applications or results from more complex, continuous domains within the fleet's `fleet-simulator` or `holodeck-rust` projects?
