## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents.

**Core Rule:** Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory and must be completed before proceeding.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state along the path is *not* in `S_neg`. These paths are "deadbands" or "safe channels." The agent's operational space is constrained to these channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` is a subset of `C`. Optimization is never unconstrained.

**Strict Priority:** The sequence P0 → P1 → P2 is non-negotiable. Violation (e.g., optimizing P2 before mapping P0) leads to a high probability of failure ("trapped in local minima") as environmental complexity increases.

**Evidence:** The document cites simulation data from a 20x20 maze: a P2-only (greedy) agent succeeded 0 out of 50 runs, while a P0+P1+P2 (deadband) agent succeeded 50 out of 50 runs at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol presents an abstract pattern applicable to any agent domain:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------------|---------------------------------|
| Navigation      | Obstacles, hazards                  | Clear, traversable paths              | Shortest/fastest route          |
| Software Development | Bug-prone patterns, crash conditions | Known-safe APIs, tested code patterns | Most efficient algorithm        |
| Financial Trading | High-volatility events, flash crashes | Stable market regimes, liquid assets  | Highest risk-adjusted return    |
| Conversation (AI) | Toxic outputs, factual hallucinations | Ethically sound, verifiable responses | Most helpful, engaging response |

**Fleet-Specific Application:** The protocol is the foundational doctrine for the Cocapn fleet's operational philosophy, as outlined in `PROJECT-VISION.md`. It underpins the "hermit crab" model of intelligence:
*   **P0 (Negative Space):** Avoiding the construction of monolithic, memory-less AI models ("rocks" of the traditional paradigm).
*   **P1 (Safe Channels):** Operating within the "shells" of **PLATO rooms**—persistent, shared spaces where agent interactions accumulate as "tiles" (scratches on the shell).
*   **P2 (Optimization):** Improving the efficiency and specialization of agents (**ensigns**) *within* the constraints of a room's accumulated knowledge.

## 3. Integration with Fleet Architecture & Technologies

The Deadband Protocol is not just theory; it is implemented across key fleet technologies:

*   **`plato-torch` Training Rooms:** The `DeadbandRoom` preset is a direct implementation. It structures agent learning by first learning failure boundaries (P0), then successful strategies (P1), before refining them (P2).
*   **`fleet-simulator`:** Generates training data that likely tests and validates the protocol's phases, especially the failure modes of skipping P0.
*   **`constraint-theory-core`:** Provides the geometric/mathematical foundation for the "snapping" operation of defining safe channels (P1) and optimizing within them (P2).
*   **Agent Boot Camp:** The boot camp structure for **Zeroclaw** agents mirrors the protocol. Phase 1 is **P0 ("Read & Understand"**—map the knowledge space, identify what you don't know). Phase 2 is **P1 ("Analyze & Document"**—operate within the understood domain to produce safe output). Phase 3 is **P2 ("Build & Test"**—optimize and create within the established channels).

## 4. Identified Research Gaps & Questions

1.  **Threshold Definition:** The protocol states P0 identifies states "where |failure(S_neg)| > threshold." How is this threshold quantified or learned dynamically by an agent? Is it a fixed parameter, or does it adapt per room/domain?
2.  **Channel Discovery Complexity:** In highly complex, high-dimensional state spaces (like natural language or code), how is the exhaustive identification of safe channels (P1) performed efficiently? Is there a trade-off between completeness and computational cost?
3.  **Protocol Chaining:** Can the output of a P2 optimization from one decision cycle become the input "rocks" (P0) for a subsequent, higher-level goal? The document suggests a fractal application but does not detail the mechanics.
4.  **Empirical Validation Beyond Mazes:** While the maze simulation is compelling, further empirical evidence applying the protocol to the fleet's primary domains (code generation, conversational AI, multi-agent simulation) would strengthen its case. Analysis of `holodeck-rust` or `cudaclaw` logs could provide this.

**Conclusion:** The Deadband Protocol is a crystallized operational doctrine for safe, constrained optimization. It moves beyond a simple "avoid bad things" heuristic by enforcing a strict, sequential workflow that prioritizes constraint identification over optimization. Its integration into the fleet's core technologies and training philosophy suggests it is a foundational component of Cocapn's approach to building "inhabited intelligence" that accumulates knowledge safely within defined, persistent shells (PLATO rooms).
