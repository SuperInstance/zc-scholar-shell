# Scholar Cycle 314 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (seeking the "best" path) before mapping constraints leads to failure with high probability as environmental complexity increases.

**Formal Rule (v1.0):**
Given an Agent `A` in state `S`, with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg` (`s ∉ S_neg`). These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that any agent that performs P2 (optimization) before completing P0 and P1 becomes trapped in local minima with probability approaching 1.0 as complexity grows.

**Empirical Proof:** A cited simulation (20x20 maze, 50 runs) showed:
*   **P2-only (Greedy):** 0/50 successes. The agent optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 successes at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                    | Course (P2)                   |
|-----------------|-------------------------------------|---------------------------------|-------------------------------|
| **Navigation**  | Obstacles, hazards                 | Safe water, clear lanes         | Shortest/fastest path         |
| **Software Development** | Bugs, crashes, security flaws      | Known-safe patterns, linted code | Performance optimization      |
| **Machine Learning** | Overfitting, mode collapse, adversarial examples | Robust model architectures, validated training regimes | Hyperparameter tuning for accuracy |
| **Business Strategy** | Market failures, regulatory bans   | Validated business models, compliant operations | Growth and scaling optimization |
| **Personal Decision-Making** | High-risk actions, known pitfalls  | Safe, legal, and ethical options | Most efficient or rewarding choice |

## 3. Connection to Cocapn Fleet & Project Vision

The Deadband Protocol is not just a navigation rule; it is a foundational **meta-constraint** for the fleet's core thesis of "inhabited intelligence."

*   **P0 as Collective Memory:** In the Cocapn paradigm, "rocks" (`S_neg`) are not just static obstacles. They are accumulated failures, bugs, and dead ends experienced by *all prior agents* in a PLATO room. Each "scratch" (tile) left by a hermit crab agent can be a record of a `S_neg` state. The room's memory becomes the ever-growing P0 map for future inhabitants.
*   **P1 as the Room's "Shell":** The safe channels (`P`) are the emergent, validated paths of successful interaction within a room. They define the operational "shell" that new agents inherit. An agent doesn't need to rediscover safe patterns; it inhabits a space where they are already encoded.
*   **P2 as Specialized Adaptation:** Optimization (P2) happens *within* the context of the room's accumulated knowledge. An "Ensign" model (a LoRA adapter exported from a room) is the embodiment of a P2 policy—a highly optimized strategy for a specific domain, but one that is fundamentally constrained by the room's historical P0 and P1 knowledge.

**Violation in the Old Paradigm:** The vision document critiques major AI labs for essentially performing "P2-only" development: pouring resources into optimizing model size and capability (`P2: argmin(cost(P))`) without first rigorously mapping the negative space of societal harm, bias, and misuse (`P0`), or establishing verified safe operational channels (`P1`). The Deadband Protocol provides a formal structure for this critique.

## 4. Identified Research Gaps & Questions

1.  **Threshold Definition:** The protocol defines `S_neg` as states where `|failure(S_neg)| > threshold`. How is this threshold determined dynamically across different domains and rooms? Is it learned, set by a room curator, or emergent?
2.  **Channel Discovery Complexity:** In high-dimensional state spaces (like code or language generation), exhaustively finding all safe channels (P1) may be computationally intractable. What heuristics or approximations does the fleet use (e.g., in `constraint-theory-core` or `holodeck-rust`)?
3.  **Inter-Room Channel Mapping:** If a safe channel is discovered in one PLATO room (e.g., a coding pattern in `plato-torch`), how is that channel communicated or made available to agents in other, related rooms? This relates to the "Weaver" agent's task of mapping integrations.
4.  **Empirical Validation Beyond Mazes:** The provided proof is a 2D maze simulation. Are there validated results from applying the protocol to the fleet's own complex domains like sentiment-aware NPCs (`holodeck-rust`) or geometric constraint solving (`constraint-theory-core`)?
