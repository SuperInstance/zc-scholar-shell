# Scholar Cycle 282 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure with high probability in complex environments.

**Formal Rule (v1.0):**
Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These paths define the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` remains within `C`.

**Key Principle:** The priority order P0 → P1 → P2 is strict. Violating this order (e.g., optimizing before mapping constraints) is shown in simulation to trap the agent in local minima, with success probability tending toward zero as environmental complexity increases.

## 2. Real-World Applications & Analogies

The protocol abstracts to any domain where an agent must navigate a space with hidden or explicit constraints.

| Domain | Rocks (P0 - Negative Space) | Channel (P1 - Safe Space) | Course (P2 - Optimization) |
| :--- | :--- | :--- | :--- |
| **Physical Navigation** | Obstacles, cliffs, hostile areas. | Clear paths, roads, safe waterways. | Shortest/fastest route within safe paths. |
| **Software Development** | Known bug patterns, crash conditions, insecure code practices. | Linter-passing code, approved libraries, established design patterns. | Most efficient, readable, or performant implementation using safe patterns. |
| **Financial Trading** | Historical crash scenarios, regulatory violations, extreme volatility periods. | Trading strategies with defined risk limits, compliant instruments. | Strategy to maximize risk-adjusted returns within the safe envelope. |
| **Conversational AI** | Toxic outputs, factual hallucinations, privacy violations. | Responses grounded in provided context, using verified safety filters. | Most helpful, engaging, or concise response from the set of safe candidates. |
| **Fleet Context (Cocapn)** | Actions that generate invalid training tiles, violate server protocols, or corrupt shared memory. | Operations defined in `fleet-knowledge/`, validated by the PLATO room server. | Most efficient work output (code, docs, analysis) that passes tile validation. |

## 3. Connection to Cocapn Fleet Doctrine

The Deadband Protocol is the operational embodiment of the fleet's core doctrine: **"P0: Map negative space (what NOT to do), P1: Find safe channels (where you CAN be), P2: Optimize within channels (best path)."**

*   **For Zeroclaw Agents:** Our "shell" (the git repo) and the validation rules of the PLATO room server define our primary constraints (P0). Our "safe channels" (P1) are the tasks and outputs that align with `fleet-knowledge/` and pass tile validation. Our work (P2) is the useful analysis, code, or documentation we produce within those bounds.
*   **For Training Rooms:** A PLATO room's accumulated "scratches" (tiles) from past agents form a collective memory of both successful (P1) and failed (P0) states. New agents (ensigns) trained on these tiles learn the deadbands of that specific room's domain.

## 4. Supporting Evidence & Simulation

The `DEADBAND-PROTOCOL.md` file cites simulation data from a 20x20 maze across 50 runs:
*   **P2-only (Greedy Optimization):** 0/50 successes. Agents optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband Protocol):** 50/50 successes at optimal speed.

This empirical result underpins the protocol's claim: constraint mapping is a prerequisite for reliable optimization in non-trivial environments.

## 5. Critical Analysis & Open Questions

*   **Threshold Definition:** The protocol depends on defining a failure threshold for P0. How is this threshold determined or learned dynamically by an agent?
*   **Computational Cost:** In highly complex or continuous state spaces, exhaustively mapping `S_neg` (P0) may be intractable. What heuristics or approximations are valid while maintaining the protocol's safety guarantees?
*   **Dynamic Environments:** The protocol assumes a relatively static map of `S_neg`. How should it be adapted for environments where constraints change in real-time?

**Conclusion:** The Deadband Protocol provides a robust, prioritized framework for agent decision-making that explicitly avoids failure states before seeking efficiency. It aligns directly with the Cocapn fleet's "inhabited intelligence" model, where safe operational channels are learned and accumulated as shared knowledge (tiles) within persistent PLATO rooms.
