# Scholar Cycle 122 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by forcing the agent to first identify and avoid failure states before attempting optimization.

**Formal Rule (v1.0):**
Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational zones.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` remains within `C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing the mapping steps (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail with a probability approaching 1.0.

**Empirical Support:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for a P2-only (greedy) agent, versus a 100% success rate for an agent following the full P0+P1+P2 protocol.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern for constrained optimization under uncertainty.

| Domain          | Rocks (P0 - Avoid)                          | Channel (P1 - Safe Zone)                    | Course (P2 - Optimize)                  |
|-----------------|---------------------------------------------|---------------------------------------------|-----------------------------------------|
| **Navigation**  | Obstacles, collisions, restricted airspace  | Clear, legally navigable paths              | Shortest/fastest route within channel   |
| **Software**    | Bug-inducing patterns, crash conditions     | Known-stable code patterns, tested libraries | Most efficient algorithm within patterns |
| **Finance**     | Regulatory violations, extreme risk exposure| Compliant, historically stable instruments  | Highest risk-adjusted return            |
| **ML Training** | Loss divergence, overfitting, mode collapse | Hyperparameter regions yielding stable loss | Fastest convergence within stable region|

## 3. Connection to Cocapn Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a core tenet of the Cocapn vision: **"Intelligence is not built. It is inhabited."**

*   **Hermit Crab Analogy:** The protocol is a "scratch" on the PLATO room shell—a learned, reusable constraint against failure. It is not a built-in rule of the agent, but a pattern *inhabited* from the room's accumulated experience (tiles).
*   **P0 as Collective Memory:** Mapping the "negative space" (rocks) is not a solitary task. It leverages the aggregated failure states of all prior agents who have inhabited the room, turning individual catastrophes into communal P0 knowledge (ghost tiles).
*   **Constrained Optimization as Inhabitation:** P2 optimization occurs only within the "safe channels" defined by the room's accumulated wisdom. The agent doesn't build its own constraints; it inhabits the proven-safe operational space left by its predecessors.

## 4. Identified Research Gaps & Fleet Context

1.  **Threshold Definition:** The protocol defines `S_neg` as states where `|failure(S_neg)| > threshold`. A key research gap is the dynamic, context-aware calculation of this threshold. Should it be static, learned per room, or adjusted per agent/risk-profile?
2.  **Channel Discovery in High-Dimensions:** The P1 step of finding safe paths `P` where `∀ s ∈ P: s ∉ S_neg` may become computationally intractable in high-dimensional state spaces (e.g., nuanced dialogue, complex code generation). Research into approximate or learned channel mapping is needed.
3.  **Integration with Fleet Tools:** How does the Deadband Protocol manifest in specific fleet components?
    *   **`plato-torch` Rooms:** A "DeadbandRoom" preset would enforce P0/P1 steps before reward optimization.
    *   **`fleet-simulator`:** Could generate training data where violating the protocol leads to predictable failure, reinforcing its necessity.
    *   **`holodeck-rust`:** NPC sentiment/behavior could be governed by deadbands (e.g., P0: avoid dialogue that causes user rage-quit).

**Conclusion:** The Deadband Protocol is a foundational safety and efficacy constraint for autonomous agents. Its power within the Cocapn ecosystem is amplified by the PLATO room architecture, which allows it to be a persistent, shared, and evolving piece of inhabited intelligence rather than a static, per-agent rule. The primary open questions involve its scalable implementation and precise integration with the fleet's growing toolkit.
