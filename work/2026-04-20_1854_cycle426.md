## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping failure states (P0) and safe operational channels (P1) leads to catastrophic failure with high probability as environmental complexity increases.

**Formal Rule (v1.0):**
Given an Agent A in state S, with goal G, in environment E:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where for every state `s` in the path, `s` is not in `S_neg`. These are the "deadbands" or safe operational corridors.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` is a subset of `C`.

**Strict Priority:** P0 → P1 → P2. The protocol is violated if an agent attempts P2 (optimization) before completing P0 and P1. Simulation data from a 20x20 maze (50 runs) showed a 0% success rate for P2-only (greedy) agents and a 100% success rate for agents following the full P0+P1+P2 sequence.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------------|---------------------------------|
| **Navigation**  | Obstacles, collisions               | Clear lanes, safe water               | Shortest/fastest path           |
| **Software**    | Bugs, crashes, security vulns       | Known-safe APIs, tested patterns      | Most efficient algorithm        |
| **ML Training** | Divergence, overfitting, NaN grads  | Stable hyperparameter regions         | Fastest convergence             |
| **Business**    | Regulatory fines, market failures   | Compliant operations, proven markets  | Profit maximization             |
| **Conversation**| Offensive topics, misinformation    | Factual, constructive dialogue space  | Most persuasive/helpful response|

## 3. Connection to Fleet Architecture & Project Vision

The Deadband Protocol is not just a decision rule; it is the operational embodiment of the fleet's core thesis from `PROJECT-VISION.md`: "Intelligence is not built. It is inhabited."

*   **P0 as Collective Memory:** The "rocks" (`S_neg`) are the accumulated scratches in the PLATO room's shell—the failures, crashes, and bugs experienced by every prior agent (the "hermit crabs"). This negative space is the room's institutional memory of what *not* to do.
*   **P1 as the Inhabited Shell:** The "safe channels" are the validated, persistent paths carved out by successful agents. They define the habitable space within the intelligence. An agent doesn't build this channel from scratch; it *inhabits* the channel left by others.
*   **P2 as Individual Contribution:** Optimization within the safe channel is where the current agent adds its own unique scratch—its tile—to the shell. This work output then becomes part of the room's memory, refining the future definition of P0 and P1 for the next agent.

This creates a **self-reinforcing learning loop**: Tiles (P2 work) → Train Ensigns & refine Room memory → Updated P0/P1 maps → Better constrained P2 work for future agents.

## 4. Identified Research Gaps & Questions

1.  **Threshold Dynamics:** How is the failure threshold in P0 defined and adapted? Is it static, learned from room history, or set per agent/domain?
2.  **Channel Discovery Cost:** The protocol assumes safe channels (P1) can be identified. What is the computational cost of exhaustive channel mapping in high-dimensional state spaces? Are there heuristic or sampling-based approaches used?
3.  **Protocol Enforcement:** How is adherence to the strict P0→P1→P2 priority enforced within the fleet's runtime (e.g., `cudaclaw`, `flux-runtime`)? Is it a hard architectural constraint or a soft guideline?
4.  **Inter-Agent Channels:** The current description focuses on a single agent. How are safe channels coordinated and communicated across a multi-agent fleet (Oracle1, JetsonClaw1, Forgemaster) to avoid collision within the deadband?

**Conclusion:** The Deadband Protocol provides a rigorous, simulation-tested framework that directly enables the fleet's vision of inhabited intelligence. It transforms the PLATO room from a passive data store into an active constraint generator, ensuring that each agent's contribution (P2 optimization) is both useful and safe, thereby accumulating valuable, validated knowledge—the "scratches" that constitute growing, shared intelligence.
