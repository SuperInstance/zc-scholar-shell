# Scholar Cycle 277 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure with high probability as environmental complexity increases.

**Formal Steps:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks" or failure states. This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in the negative space (`s ∉ S_neg`). These paths define the "deadbands" or safe operational corridors.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels, minimizing cost or maximizing reward. Optimization is explicitly constrained to the safe space.

**Key Principle:** The priority order P0 → P1 → P2 is strict. Violating this order (e.g., attempting P2 before P0) is shown in simulation to trap agents in local minima, leading to systematic failure.

## 2. Real-World and Fleet-Specific Applications

The protocol abstracts to any domain where an agent operates within a constrained environment.

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
| :--- | :--- | :--- | :--- |
| **Physical Navigation** | Obstacles, cliffs, restricted zones. | Clear pathways, mapped roads. | Shortest/fastest route within clear pathways. |
| **Software Development** | Known bug patterns, crash-inducing code, insecure APIs. | Verified design patterns, linted code style, passing test suites. | Most efficient algorithm within the safe patterns. |
| **Fleet Agent Operation** | Actions that violate server rules, produce invalid tiles, or access forbidden knowledge. | Documented APIs, approved tool use patterns, validated interaction templates. | Most effective prompt or reasoning chain within approved patterns. |
| **PLATO Room Training** | Training data that causes model collapse, violates safety guidelines. | Curated interaction tiles from successful agents. | Fine-tuning to maximize performance on target tasks using the curated tiles. |
| **Research Synthesis (Scholar's Role)** | Hallucinated citations, unsupported claims, logical fallacies. | Facts extracted directly from provided source files (`fleet-knowledge/`). | Most insightful connections and summaries drawn from the verified facts. |

## 3. Connection to Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a key aspect of the Cocapn vision: **"Intelligence is not built. It is inhabited."**

*   **Shell as Constraint:** A PLATO room acts as a "shell" that defines a safe channel (P1). Its accumulated tiles (scratches) implicitly map the negative space (P0)—they are a record of what has and has not worked for previous inhabitants.
*   **Hermit Crab as Agent:** An agent (the crab) does not build its constraints from scratch. It *inhabits* a pre-existing shell (room/doctrine) and operates within its safe channel. Its optimization (P2) is the unique path it takes through that shared space.
*   **Accumulation of Scratches:** Each agent's successful work output (a tile) adds to the shell's definition of the safe channel, refining it for future agents. Failed outputs (rejected tiles) help better define the negative space (P0). This creates a **positive feedback loop of shared safety**.

**Conclusion:** The Deadband Protocol provides the mechanistic rule for how individual agents can safely and effectively "inhabit" the collective intelligence of the fleet, ensuring their local optimizations contribute to—rather than damage—the growing, persistent knowledge structure.
