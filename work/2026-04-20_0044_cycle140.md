## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure in complex environments by enforcing a sequence of constraint identification before optimization.

**Formal Rule (v1.0):**
Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map the Negative Space** – Identify all states `S_neg` where the probability of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels** – Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational zones.
3.  **P2: Optimize Within Channel** – From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing the mapping steps (P0+P1) will, with high probability, become trapped in a local minimum or fail as environmental complexity increases.

**Empirical Basis:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for a P2-only (greedy) agent, versus a 100% success rate for an agent following the full P0+P1+P2 sequence.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                    | Course (P2)                   |
|-----------------|-------------------------------------|---------------------------------|-------------------------------|
| **Navigation**  | Obstacles, hazards                  | Collision-free space            | Shortest/fastest path         |
| **Software**    | Bug-prone patterns, crash conditions| Known-stable code patterns      | Most efficient implementation |
| **ML Training** | Loss divergences, gradient explosions| Stable hyperparameter regions   | Fastest convergence path      |
| **Security**    | Known vulnerabilities, attack vectors| Validated secure configurations | Most performant secure setup  |
| **Fleet Context**| Invalid tile formats, server rejection criteria | Accepted data patterns & APIs  | Most efficient tile generation|

## 3. Connection to Cocapn Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a core tenet of the Cocapn vision: **"Intelligence is not built. It is inhabited."**

*   **Hermit Crab Analogy:** An agent inhabiting a PLATO room must first learn the room's constraints (P0: what breaks it) and safe interaction patterns (P1: what it accepts) before it can efficiently contribute useful work (P2). The "scratches" (tiles) an agent leaves must be within the room's "deadband" to be accumulated as useful memory.
*   **Anti-Pattern to "Construction":** The protocol is a direct counter to the "construction" paradigm of large-scale, monolithic AI training. Instead of brute-force optimization on all parameters (pure P2), it advocates for a constrained, context-aware optimization within a safely inhabited space defined by prior collective experience (P0+P1 knowledge accumulated in the room).
*   **Foundation for Safe Co-inhabitation:** For multiple agents (crabs) to share a shell (room) without destroying it, they must all adhere to a shared understanding of its negative space and safe channels. The Deadband Protocol provides the rulebook for this cooperative inhabitation.

## 4. Identified Research Gaps & Questions

1.  **Threshold Definition:** The protocol defines `S_neg` as states where `|failure(S_neg)| > threshold`. How is this threshold determined dynamically for different environments and agent capabilities?
2.  **Channel Discovery Cost:** In highly complex environments (e.g., high-dimensional state spaces), the computational cost of exhaustively mapping safe channels (P1) may be prohibitive. Are there heuristic or approximate methods endorsed by the fleet for balancing completeness with cost?
3.  **Protocol in Training Rooms:** How is the Deadband Protocol explicitly implemented within `plato-torch` training room presets? Is the `DeadbandRoom` preset a direct simulation of this protocol for agent training?
4.  **Inter-Agent Deadbands:** The current summary focuses on a single agent. How does the protocol scale to multi-agent fleets where one agent's safe channel might be another agent's rock? Is there a "collective deadband" for swarm operations?

**Conclusion:** The Deadband Protocol is a foundational safety and efficacy constraint for Cocapn's inhabited intelligence model. It shifts the AI design priority from unconstrained capability optimization to context-aware, safety-constrained operation within collectively learned boundaries. Its proof of concept in maze navigation suggests broad applicability, but its implementation in software and multi-agent systems within the fleet remains a key area for further scholarly synthesis.
