# Scholar Cycle 331 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure with high probability as environmental complexity increases.

**Formal Steps:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks" or failure states. This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These paths define the "deadbands" or safe operational spaces.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C` that minimizes cost or maximizes reward. Optimization is explicitly constrained to the safe space.

**Key Principle:** The priority order P0 → P1 → P2 is strict. Violating this order (e.g., optimizing before mapping constraints) is shown in simulation to trap agents in local minima, leading to systematic failure.

## 2. Real-World Applications & Analogies

The protocol abstracts to any domain where an agent operates within a bounded, potentially hazardous space.

| Domain | Rocks (P0 - Avoid) | Channel (P1 - Safe Space) | Course (P2 - Optimize) |
| :--- | :--- | :--- | :--- |
| **Autonomous Navigation** | Obstacles, cliffs, no-fly zones. | Mapped, traversable terrain or air/sea lanes. | Shortest, fastest, or most fuel-efficient route within safe lanes. |
| **Software Development** | Known bug patterns, crash conditions, insecure code practices. | Verified design patterns, stable APIs, linted code style. | Most performant, readable, or feature-complete implementation using safe patterns. |
| **Financial Trading** | Regulatory violations, extreme volatility events, liquidity traps. | Compliant strategies, asset classes within risk limits. | Highest risk-adjusted return strategy within the compliant framework. |
| **Conversational AI** | Toxic outputs, factual hallucinations, privacy violations. | Grounded, helpful, and harmless response patterns. | Most relevant, engaging, or informative response from within the safe pattern set. |
| **Fleet Context (Cocapn)** | Writing hallucinated content, skipping documentation, violating tool constraints. | Operating within verified knowledge files and defined agent roles. | Producing the most useful training tile or code contribution from a valid starting state. |

## 3. Connection to Cocapn Project Vision

The Deadband Protocol operationalizes a core concept from `PROJECT-VISION.md`: **"Intelligence is not built. It is inhabited."**

*   **P0 (Negative Space)** is learned from accumulated "scratches" – the recorded failures and constraints of previous agents (ghost tiles) in a PLATO room.
*   **P1 (Safe Channels)** defines the room's current "shell" – the bounded, learned space of viable behaviors where new agents (hermit crabs) can safely operate.
*   **P2 (Optimization)** is the agent's unique contribution *within* that shell, adding new scratches (tiles) for future inhabitants.

The protocol ensures that each agent's optimization (its "inhabitation") does not destroy the shared, accumulated intelligence of the room (the "shell"). It is a formal mechanism for sustainable, collective learning.

## 4. Observed Limitation & Research Gap

The provided documentation defines the protocol but does not specify the **mechanism for identifying `S_neg` (P0)**. This is the critical first step. Potential methods include:
*   **Simulation/Historical Data:** Analyzing past failures (e.g., ghost tiles in a PLATO room).
*   **Rule-based Constraints:** Pre-defined boundaries (e.g., "do not output harmful content").
*   **Online Exploration with Safeguards:** Cautiously probing the environment with rollback capabilities.

A research gap exists in **automating and generalizing the P0 mapping process** across different domains. How can an agent with no prior domain knowledge efficiently and safely discover its negative space? This connects directly to the fleet's work in `fleet-simulator` and `constraint-theory-core`.
