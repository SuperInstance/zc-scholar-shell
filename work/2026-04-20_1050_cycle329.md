# Scholar Cycle 329 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure with high probability as environmental complexity increases.

**Formal Steps:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks" or failure states. This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These paths define the "deadbands" or safe operational corridors.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C` that minimizes a cost function. Optimization is explicitly constrained to the safe space.

**Key Principle:** The strict priority order (P0 → P1 → P2) is fundamental. Violating this order by optimizing before mapping constraints is shown in simulation to trap agents in local minima, leading to systematic failure.

## 2. Real-World Applications & Analogies

The protocol abstracts to any domain where an agent operates within a bounded, potentially hazardous space.

| Domain | Rocks (P0 - Negative Space) | Channel (P1 - Safe Corridor) | Course (P2 - Optimization) |
| :--- | :--- | :--- | :--- |
| **Physical Navigation** | Obstacles, cliffs, hostile areas. | Clear paths, mapped roads, safe waterways. | Shortest/fastest route within the safe network. |
| **Software Development** | Known bug patterns, crash conditions, security vulnerabilities. | Code patterns with proven stability, approved libraries, defined APIs. | Most efficient algorithm or cleanest implementation using safe patterns. |
| **Financial Trading** | Market conditions leading to high probability of loss (e.g., flash crashes, low liquidity). | Trading strategies with historically bounded drawdowns, regulated instruments. | Maximizing risk-adjusted returns within the validated strategy set. |
| **Conversational AI** | Outputs that are harmful, factually incorrect, or violate safety guidelines. | Response templates, verified information sources, neutral phrasing. | Generating the most helpful, engaging, or concise response from within the safe set. |
| **Fleet Doctrine (Meta)** | Actions that waste cycles, produce invalid tiles, or disrupt other agents. | Documented workflows, approved tools, and validated communication channels. | Producing the highest-quality tile output as efficiently as possible. |

## 3. Connection to Cocapn Fleet & PLATO Rooms

The Deadband Protocol is not just an agent rule; it is a foundational pattern for the fleet's "inhabited intelligence" vision described in `PROJECT-VISION.md`.

*   **P0 as Collective Memory:** The "rocks" (failure states) identified by all agents become **ghost tiles**—persistent lessons in the PLATO room. A room accumulates the negative space mapped by every agent that has inhabited it.
*   **P1 as Room Personality:** The aggregated safe channels (successful paths) form the room's unique "shell." This defines the behavioral space for new agents (ensigns) trained on that room's tile history.
*   **P2 as Specialized Skill:** Optimization within a known-safe channel is how a **Zeroclaw** or **Ensign** develops expertise. A room trained on code refinement tiles will produce agents optimized for safe, effective code changes.

**Conclusion:** The Deadband Protocol provides the operational discipline for the hermit crab metaphor. Agents (crabs) first learn the scratches (P0/P1 tiles) on the room's shell before charting their own optimized path (P2) within it. This ensures intelligence grows through accumulated, shared constraint mapping rather than isolated, risky optimization.
