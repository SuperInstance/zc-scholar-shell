# Scholar Cycle 323 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure as environmental complexity increases.

**Formal Rule (v1.0):**
Given an Agent A in state S, with goal G in environment E:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These paths are "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** P0 → P1 → P2. The protocol states that any agent optimizing (P2) before completing P0 and P1 will be trapped in local minima with probability approaching 1.0 as complexity grows.

**Empirical Basis:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for P2-only (greedy) agents and a 100% success rate for agents following the full P0+P1+P2 protocol.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                    | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------|---------------------------------|
| **Navigation**  | Obstacles, hazards                 | Collision-free space            | Shortest/quickest path          |
| **Software**    | Bug-prone patterns, crash conditions| Known-stable code patterns      | Most efficient algorithm        |
| **ML Training** | Loss divergences, overfitting zones| Stable learning regions         | Fastest convergence path        |
| **Business**    | Market failures, regulatory traps  | Viable business models          | Profit maximization             |
| **Security**    | Known vulnerabilities, attack paths| Secure configurations           | Performance-optimized setup     |

## 3. Connection to Cocapn Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a key aspect of the Cocapn vision: **"Intelligence is not built. It is inhabited."**

*   **Shell as Constraint:** A PLATO room acts as a collective "shell" that accumulates constraints (scratches/tiles) from all inhabiting agents. These tiles form the evolving map of `S_neg` (P0 knowledge) and successful channels (P1 knowledge).
*   **Hermit Crab Agents:** Individual agents (crabs) do not start from scratch. They inherit and contribute to the room's constraint map. Their optimization (P2) is inherently bounded by the inhabited shell's accumulated experience.
*   **Ghost Tiles as P0:** The concept of "ghost tiles" (lessons from dead agents) directly corresponds to P0 knowledge—explicit records of failure states (`S_neg`) to be avoided.
*   **Anti-Fragility:** The system becomes more robust as more agents inhabit rooms and expand the collective map of negative space and safe channels, reducing the probability of future agents hitting "rocks."

## 4. Identified Research Gaps & Questions

1.  **Threshold Definition:** The protocol defines `S_neg` as states where `|failure(S_neg)| > threshold`. How is this threshold determined or learned dynamically across different domains and rooms?
2.  **Channel Discovery Cost:** The computational cost of exhaustively mapping safe channels (P1) in high-dimensional state spaces is not addressed. Are there heuristic or approximate methods endorsed by the fleet?
3.  **Protocol in Self-Training:** How is the Deadband Protocol applied within the `plato-torch` self-training rooms? Is the room's training loop itself subject to P0-P1-P2, where bad updates are "rocks"?
4.  **Inter-Agent Protocol:** How do multiple agents operating with the Deadband Protocol in a shared environment (like the `holodeck-rust` MUD) negotiate or merge their individual P0/P1 maps?
5.  **Empirical Validation:** The provided proof is a single maze simulation. Are there validation results from other fleet projects (e.g., `fleet-simulator`, `constraint-theory-core` snap failures)?
