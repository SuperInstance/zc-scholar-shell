## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) without first mapping failure states (P0) and safe operational channels (P1) leads to catastrophic failure as environmental complexity increases.

**Formal Steps:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is non-negotiable and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These are the "deadbands" or safe operational corridors.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C` that minimizes cost `(argmin(cost(P)) subject to P ⊆ C)`.

**Key Principle:** The priority order (P0 → P1 → P2) is strict. Violating this order—attempting to optimize before mapping safe channels—results in agents becoming trapped in local minima or failing outright, with the probability of failure approaching 1.0 as complexity grows.

**Empirical Support:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for P2-only (greedy) agents, while agents following the full P0+P1+P2 protocol achieved a 100% success rate at optimal speed.

## 2. Real-World Applications & Abstract Pattern

The protocol presents an abstract pattern applicable to any agentic domain. The following table maps the core concepts to several fields:

| Domain | Rocks (P0 - Avoid) | Channel (P1 - Safe Corridor) | Course (P2 - Optimize Within) |
| :--- | :--- | :--- | :--- |
| **Physical Navigation** | Obstacles, cliffs, hostile zones. | Mapped, traversable terrain. | Shortest/fastest/least-energy path. |
| **Software Development** | Known bug patterns, crash conditions, insecure code practices. | Verified design patterns, linted code style, passing test suites. | Most efficient algorithm, cleanest architecture. |
| **Financial Trading** | Black swan events, regulatory violations, maximum drawdown limits. | Historically stable regimes, compliant strategies, risk-bounded positions. | Highest risk-adjusted return strategy. |
| **Conversational AI** | Toxic outputs, factual hallucinations, privacy violations. | Grounded, safe, and helpful response templates. | Most engaging, concise, or informative response. |
| **ML Model Training** | Overfitting, mode collapse, gradient explosion. | Hyperparameter ranges that yield stable training. | Model architecture with highest validation accuracy. |

## 3. Connection to Fleet Vision & Technology

The Deadband Protocol is not just a navigation rule; it is a foundational **constraint theory** for building reliable, inhabited intelligence, as described in `PROJECT-VISION.md`.

*   **P0 as Collective Memory:** The "rocks" (`S_neg`) identified by agents become **ghost tiles**—persistent lessons from past failures that train future agents. This is the "accumulated scratches" in a PLATO room.
*   **P1 as the Shell:** The safe channels are the validated structures of a **PLATO room preset** (e.g., `DeadbandRoom` in `plato-torch`). Agents "inhabit" this shell.
*   **P2 as Specialized Skill:** Optimization within a safe channel is the domain of a trained **ensign** (a LoRA adapter exported via `plato-ensign`), which excels at a specific task within known boundaries.
*   **Violation as Training Signal:** An agent that skips to P2 and fails generates a high-value training tile, reinforcing the protocol for the entire swarm.

This creates a self-reinforcing loop: the protocol generates reliable tiles (work), which train better ensigns (specialists), which operate more effectively within their deadbands, producing higher-quality work. It is the operational embodiment of the vision that "intelligence is not built. It is inhabited."
