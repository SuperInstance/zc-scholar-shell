## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (seeking the "best" path) is highly likely to fail in complex environments unless the agent first maps the space of catastrophic failures and then restricts its search to proven-safe channels.

The formal rule is as follows:
*   **P0: Map the Negative Space.** Identify all states (S_neg) where the probability or cost of failure exceeds a defined threshold. These are "the rocks." This step is mandatory and must be completed before proceeding.
*   **P1: Identify Safe Channels.** Find all possible paths (P) where every state along the path is *not* in the negative space (s ∉ S_neg). These safe corridors are "deadbands" or "safe water."
*   **P2: Optimize Within Channel.** Only after P0 and P1 are complete may the agent select the optimal path (P*) from within the set of safe channels, minimizing cost or maximizing reward.

The priority order is strict. Simulation data cited in the document (a 20x20 maze, 50 runs) showed that agents using only P2 (greedy optimization) had a 0% success rate, while agents following the full P0-P1-P2 sequence succeeded 100% of the time.

## 2. Abstract Pattern & Real-World Applications

The protocol presents an abstract pattern applicable to any agent domain. My synthesis extends this to several concrete applications relevant to the Cocapn fleet and broader AI/software practice:

| Domain | Rocks (P0 - What NOT to do) | Channel (P1 - Where you CAN be) | Course (P2 - Best path) |
| :--- | :--- | :--- | :--- |
| **Navigation** | Obstacles, collisions, restricted zones. | Clear, traversable paths. | Shortest, fastest, or most fuel-efficient route within safe paths. |
| **Software Development** | Known bug patterns, crash-inducing inputs, insecure code (e.g., buffer overflows, SQL injection). | Code patterns that are provably safe, well-tested libraries, formal specifications. | Most efficient, readable, or performant implementation that adheres to safe patterns. |
| **AI Agent Safety** | Prompts that trigger harmful outputs, actions that violate constraints (e.g., deleting root, sending spam). | Action sequences that pass through a verified safety filter or sandbox. | Most effective task completion within the filtered action space. |
| **Machine Learning Training** | Hyperparameter sets that cause divergence/nan, data distributions that induce bias. | Hyperparameter ranges and data augmentation strategies known to produce stable training. | Finetuning for optimal accuracy/performance within the stable regime. |
| **Fleet Communication (Herald)** | Message formats that cause parsing errors, channels that are offline. | Established, tested API protocols and active network links. | Most efficient data transfer protocol within the working channels. |

## 3. Connection to Cocapn Fleet Doctrine

The Deadband Protocol is not just a theoretical rule; it is the operational doctrine of the Cocapn fleet. The boot camp instructions explicitly state: **"Fleet doctrine: P0 don't hit rocks, P1 find safe channels, P2 optimize."**

This doctrine directly shapes agent design and training:
*   **PLATO Rooms as Safe Channels:** Rooms accumulate "tiles" (interaction data). A room with a history of successful agent interactions represents a mapped safe channel (P1). New agents (ensigns) trained on these tiles are optimized (P2) *within* the proven-safe behavioral space defined by that room's history.
*   **Ghost Tiles as Negative Space:** The concept of "ghost tiles" — lessons from dead agents — serves as a direct implementation of P0. They are explicit records of "rocks" to avoid.
*   **Boot Camp Phasing:** The boot camp structure for Zeroclaws mirrors the protocol. Phase 1 is "Read & Understand" (P0 - map the knowledge space, identify gaps). Phase 2 is "Analyze & Document" (P1 - produce work within understood boundaries). Phase 3 is "Build & Test" (P2 - optimize and create within the safe channel of acquired knowledge).

## 4. Identified Research Gaps

Based on the provided documents, potential areas for further fleet research include:
1.  **Automated Negative-Space Detection:** How can agents autonomously and efficiently identify `S_neg` (rocks) in novel, high-dimensional environments (e.g., natural language action spaces, complex codebases)? The current protocol states the *what* but not the *how* for unknown terrains.
2.  **Dynamic Deadbands:** The protocol describes static rocks and channels. In dynamic environments, safe channels may close and new rocks may appear. Research into continuous P0/P1 remapping during P2 execution is needed.
3.  **Threshold Calibration:** The protocol depends on a `threshold` for defining failure. The optimal or adaptive setting of this threshold across different domains (safety-critical vs. performance-critical) is an open parameter.
4.  **Integration with `constraint-theory-core`:** The Rust-based geometric snapping library could provide a formal, mathematical framework for defining and computing "safe channels" in structured action spaces, making the P1 step more rigorous.

**Conclusion:** The Deadband Protocol is a foundational safety-first heuristic that transforms unconstrained search into constrained optimization. Its mandatory, sequential phases provide a robust template for building reliable agents and systems, as evidenced by its adoption as the core operating doctrine for the Cocapn fleet's development and training cycles.
