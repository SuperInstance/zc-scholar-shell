# Scholar Cycle 368 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure with high probability as environmental complexity increases.

**Formal Rule (v1.0):**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory and must be completed first.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` along the path is *not* in `S_neg`. These paths are "deadbands" or "safe channels." The agent's operational space is constrained to these channels.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels `C` that minimizes a cost function. Optimization is never unconstrained.

**Key Principle:** The priority order P0 → P1 → P2 is strict and non-negotiable. Violating this order (e.g., attempting P2 before P0) is predicted to trap the agent in local minima with probability approaching 1.0 in complex environments.

**Empirical Support:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for a P2-only (greedy) agent, versus a 100% success rate for an agent following the full P0+P1+P2 protocol.

## 2. Real-World Applications & Abstract Pattern

The protocol abstracts beyond navigation to any agent domain. The provided table can be extended:

| Domain          | Rocks (P0)                          | Channel (P1)                                  | Course (P2)                           |
|-----------------|-------------------------------------|-----------------------------------------------|---------------------------------------|
| Navigation      | Obstacles, hazards                  | Clear, traversable paths                      | Shortest/fastest route                |
| Software Development | Bug-prone patterns, crash conditions | Known-safe APIs, tested design patterns       | Most efficient/readable implementation |
| Financial Trading   | High-volatility events, flash crashes | Stable, liquid market conditions              | Optimal execution strategy            |
| Conversation Agent | Toxic outputs, factual hallucinations | Ethically and factually bounded responses     | Most helpful/engaging reply           |
| Mechanical Design | Material failure points, stress concentrations | Geometrically/physically valid configurations | Lightest/strongest/most efficient design |

## 3. Connection to Cocapn Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a core tenet of the Cocapn vision: **"Intelligence is not built. It is inhabited."**

*   **P0 as Collective Memory:** The "rocks" (`S_neg`) are not just static obstacles; they are the accumulated "scratches" of past agent failures within a PLATO room. A room that remembers every crash, bug, or dead-end provides the foundational negative space map for all future inhabitants.
*   **P1 as Inherited Shell:** The "safe channels" are the proven, inhabited paths within the shell. An agent (the "hermit crab") doesn't compute these from scratch; it inherits them from the room's memory (the "shell").
*   **P2 as Local Adaptation:** Optimization within the safe channel is where the current agent's specific goal and context are applied, using the inherited safe operating space as a constraint.

Thus, the protocol provides a concrete decision-making framework that enables the "inhabited intelligence" model. It ensures agents leverage collective memory (P0, P1) before executing individual, optimized action (P2), preventing them from repeatedly hitting the same "rocks" discovered by prior inhabitants.

## 4. Identified Research Gaps / Questions

1.  **Threshold Definition:** How is the failure threshold in P0 (`|failure(S_neg)| > threshold`) quantitatively defined or learned across different domains (e.g., maze vs. code vs. conversation)?
2.  **Dynamic Environments:** How does the protocol handle environments where the negative space (`S_neg`) or safe channels evolve in real-time? Is there a defined re-mapping frequency or trigger?
3.  **Scalability of Mapping:** In extremely high-dimensional state spaces (like natural language), exhaustively mapping P0 and P1 may be intractable. What approximation or sampling techniques are valid without violating the protocol's guarantee?
4.  **Integration with Learning:** Could the outcomes of P2 optimizations (both successes and new failures discovered) feed back into the room's collective P0/P1 map, creating a learning loop? This aligns strongly with the tile/scratch accumulation model.
