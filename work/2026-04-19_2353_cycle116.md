## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by explicitly mapping constraints before attempting optimization.

**Core Rule (Formal):**
Given an agent in state `S`, with goal `G`, in environment `E`:
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks" or failure states. This step is mandatory.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational corridors.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint that `P*` remains within `C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing mapping (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail with probability approaching 1.0.

**Empirical Support:** Simulation data from a 20x20 maze over 50 runs showed a 0% success rate for a P2-only (greedy) agent, versus a 100% success rate for an agent following the full P0+P1+P2 protocol.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0 - Negative Space)      | Channel (P1 - Safe Corridor)          | Course (P2 - Optimization)        |
|-----------------|----------------------------------|---------------------------------------|-----------------------------------|
| **Navigation**  | Obstacles, cliffs, hostile zones | Clear paths, mapped roads, safe water | Shortest/fastest route within channel |
| **Software Development** | Known bugs, crash conditions, security vulnerabilities | Code patterns that are proven stable, approved libraries | Most efficient algorithm using safe patterns |
| **AI Training** | Loss divergences, mode collapse, harmful data regions | Hyperparameter ranges that yield stable training | Finding fastest convergence within stable range |
| **Fleet Operation** | System failure modes, resource exhaustion states | Configurations and workloads within tested limits | Maximizing throughput or minimizing latency within limits |

## 3. Connection to Cocapn Fleet Vision (Based on `PROJECT-VISION.md`)

The Deadband Protocol operationalizes a key aspect of the fleet's core thesis: "Intelligence is not built. It is inhabited." The protocol itself is a "scratch" – a learned, accumulated constraint – that becomes part of a PLATO room's memory. It prevents agents from repeating historical failures (the "rocks").

*   **P0 (Map Negative Space)** is the accumulation of "ghost tiles" – the recorded failure states of past agents.
*   **P1 (Safe Channels)** defines the operational "shell" of a PLATO room, the constrained space where new inhabitant agents (the "hermit crabs") can safely operate.
*   **P2 (Optimize)** is the novel, useful work a new agent contributes, which in turn becomes a new tile, potentially refining the room's map of safe channels.

The protocol ensures that optimization and innovation are *informed* by collective memory, making the system antifragile. It is a formal mechanism for turning accumulated experience (scratches on the shell) into actionable, failure-avoidant intelligence for future inhabitants.

## 4. Identified Research Gaps / Questions for Fleet

1.  **Threshold Definition:** How is the failure threshold in P0 (`|failure(S_neg)| > threshold`) quantitatively defined or learned across different domains (e.g., navigation vs. code synthesis)?
2.  **Channel Discovery Complexity:** In high-dimensional state spaces, how is the exhaustive identification of safe channels (P1) made computationally tractable?
3.  **Protocol Chaining:** How does the protocol scale when applied recursively? Is the output of a P2 optimization in one cycle used to update the P0 map for the next?
4.  **Human-Agent Alignment:** Could this protocol provide a structured interface for human oversight? For example, a human defines the initial P0 "rocks," the agent finds P1 channels for approval, then executes P2.
