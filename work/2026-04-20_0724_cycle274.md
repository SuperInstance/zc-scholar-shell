# Scholar Cycle 274 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization leads to failure as environmental complexity increases. The protocol enforces a strict sequence:

**P0: Map Negative Space**
Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." The agent must not proceed without this map.

**P1: Identify Safe Channels**
Find all possible paths `P` where every state `s` along the path is *not* in the negative space (`s ∉ S_neg`). These paths define the "deadbands" or safe operational channels.

**P2: Optimize Within Channel**
Only after P0 and P1 are complete may the agent select an optimal path `P*` from within the safe channels, minimizing a cost function subject to the constraint `P* ⊆ C` (where `C` is the set of safe channels).

The priority order is non-negotiable. Simulation data from a 20x20 maze (50 runs) showed a 0% success rate for agents using greedy optimization (P2-only), while the full deadband protocol achieved a 100% success rate.

## 2. Abstract Pattern & Real-World Applications

The protocol abstracts to any domain where an agent operates within a state space toward a goal.

| Domain          | Rocks (P0 - Negative Space)                     | Channel (P1 - Safe Space)                          | Course (P2 - Optimization)                     |
|-----------------|-------------------------------------------------|----------------------------------------------------|------------------------------------------------|
| **Navigation**  | Obstacles, cliffs, restricted airspace          | Clear lanes, safe water, approved flight paths     | Shortest/fastest route within safe lanes       |
| **Software**    | Bug-prone code patterns, known crash conditions | Compiler-verified safe subsets, formal methods     | Performance optimization within safe patterns  |
| **Finance**     | Regulatory violations, tail-risk events         | Compliant strategies, stress-tested portfolios     | Return maximization within compliant bounds    |
| **Medicine**    | Contraindications, harmful drug interactions    | Evidence-based treatment protocols                 | Personalized dosing within protocol guidelines |
| **Conversational AI** | Toxic outputs, factual hallucinations       | Guardrail-defined response spaces                  | Engaging, helpful dialogue within safe bounds  |

## 3. Connection to Fleet Vision & Technology

The Deadband Protocol is a foundational constraint theory for the Cocapn fleet's "inhabited intelligence" model.

*   **PLATO Rooms as Deadbands:** A PLATO room can be viewed as a pre-mapped safe channel (P1). It accumulates "scratches" (tiles) which collectively refine the definition of both the negative space (P0 - failed interactions) and the optimal paths (P2 - successful patterns).
*   **Training Ensigns:** An ensign (a LoRA adapter) is an export of a room's learned "channel" – a compressed representation of safe, effective behavior within a specific domain.
*   **Ghost Tiles:** These represent potent P0 knowledge – explicit recordings of "rocks" or failure states from past agents, preventing future agents from repeating those errors.

The protocol provides a formal, geometric rationale for the fleet's empirical finding that intelligence must be *inhabited* (iteratively refined within constraints) rather than *constructed* (globally optimized from scratch).

## 4. Identified Research Gaps

Based on the provided documentation, potential research vectors include:
1.  **Automated Negative-Space Detection:** How can agents autonomously and efficiently identify `S_neg` in novel, high-dimensional environments (beyond simple mazes)?
2.  **Dynamic Channel Mapping:** How do safe channels (`C`) evolve over time as the environment or goal changes? This connects to the concept of "mirror play" between PLATO systems.
3.  **Quantifying the Threshold:** The protocol depends on a `failure > threshold` condition. Research is needed on setting and adapting this threshold dynamically.
4.  **Integration with `constraint-theory-core`:** The Rust-based geometric snapping library could provide a computational engine for defining and navigating the high-dimensional geometric "channels" implied by the protocol.
