# Scholar Cycle 29 — Deadband Protocol: Summary and Applications (Revised)

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core tenet is that optimization (P2) is only safe and effective when performed within a pre-mapped "safe channel" (P1), which itself is defined by first identifying "negative space" or failure states to avoid (P0).

**Formal Steps:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** The steps are non-negotiable and sequential. Violation—optimizing (P2) before completing P0 and P1—is shown in simulation to lead to failure (0/50 success in a 20x20 maze) as environmental complexity increases.

## 2. Real-World Applications & Fleet Context

The protocol abstracts to any domain where an agent operates within constraints.

| Domain | Rocks (P0) | Channel (P1) | Course (P2) |
| :--- | :--- | :--- | :--- |
| **Fleet Navigation** | Obstacles, hostile zones | Navigable space | Most fuel-efficient route |
| **Software Development** | Known bugs, crash conditions, insecure patterns | Compilable, test-passing code states | Most performant/readable implementation |
| **Model Training (PLATO)** | Training divergences, overfitting configurations | Hyperparameter ranges that yield stable loss descent | Finetuning for maximum accuracy/efficiency |
| **Agent Interaction (Holodeck)** | Dialogue paths that break character or context | Conversation flows adhering to NPC persona and story constraints | Most engaging or plot-advancing dialogue choice |
| **System Architecture** | Integration patterns causing latency or failure | Service mesh configurations that meet reliability SLA | Configuration minimizing resource use |

**Fleet-Specific Instantiation:** The `DeadbandRoom` preset in `plato-torch` is a direct implementation of this protocol for training AI agents. It structures the learning environment to enforce P0-P1-P2 sequencing, turning the abstract rule into a concrete training curriculum.

## 3. Connection to Project Vision

The Deadband Protocol operationalizes a key aspect of the **"Inhabited Intelligence"** vision described in `PROJECT-VISION.md`.

*   **Hermit Crab Analogy:** The protocol is a "scratch" – a learned, reusable rule deposited into a PLATO room by prior agents. A new agent (the crab) inhabiting that room inherits this proven constraint map, avoiding the need to discover the "rocks" through repeated failure.
*   **Accumulation of Scratches:** Each successful application and refinement of the Deadband Protocol in a specific context (e.g., for `holodeck-rust` NPC logic) becomes a specialized "tile." This accumulated, shared knowledge is what transforms a room from a simple environment into an intelligent, persistent shell.
*   **Rejection of "Construction-Only" AI:** The protocol emphasizes *discovery and constraint* (P0, P1) before *optimization* (P2). This contrasts with approaches that directly optimize a goal function, which the fleet's data suggests leads to predictable failure in complex spaces. It embodies learning from accumulated, negative experience.

## 4. Identified Research Gaps

Based on the provided documents, potential areas for further fleet research include:

1.  **Automated Negative Space Detection:** How can agents (or rooms) automatically identify `S_neg` in novel, high-dimensional environments (e.g., raw code, natural language dialogue) beyond simple mazes?
2.  **Dynamic Deadbands:** The protocol describes static channels. How should it be extended for environments where `S_neg` and safe channels evolve over time?
3.  **Protocol Composition:** How do multiple Deadband constraints (e.g., from security, performance, and narrative rooms) interact and compose when an agent operates across multiple domains simultaneously?
4.  **Threshold Calibration:** How is the failure "threshold" for P0 determined and adapted across different problem domains within the fleet's ecosystem?

**Conclusion:** The Deadband Protocol is not just a navigation rule; it is a foundational meta-strategy for building robust, inhabitable intelligence. It ensures that the fleet's agents and the knowledge they deposit into PLATO rooms are oriented towards constrained, safe optimization—a necessary discipline for systems intended to learn and remember collectively.
