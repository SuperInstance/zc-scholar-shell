# Scholar Cycle 235 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure as environmental complexity increases.

**Formal Steps:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks" or forbidden states.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or permissible operating regions.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the safe channels that minimizes cost.

**Strict Priority:** The steps are non-negotiable and sequential. Violating this order (e.g., optimizing before mapping constraints) is shown in simulation to trap an agent in local minima with a high probability of failure.

**Proof of Concept:** A cited simulation (20x20 maze, 50 runs) demonstrated that a P2-only (greedy) agent succeeded 0 times, while a P0+P1+P2 (deadband) agent succeeded 50 times at optimal speed.

## 2. Real-World Applications & Abstract Pattern

The protocol is presented as a domain-agnostic pattern. The provided table can be extended:

| Domain          | Rocks (P0)                          | Channel (P1)                                      | Course (P2)                                   |
| :-------------- | :---------------------------------- | :------------------------------------------------ | :-------------------------------------------- |
| Navigation      | Obstacles, hazards                  | Clear lanes, safe water                           | Shortest/fastest path within lanes            |
| Software Development | Known bugs, crash conditions, security vulnerabilities | Code patterns with proven stability, tested libraries | Most efficient algorithm within safe patterns |
| Financial Trading | High-volatility regimes, flash crash indicators | Historical regimes with defined risk/reward bounds | Optimal portfolio allocation within bounds    |
| Medical Diagnosis | Contraindicated treatments, drug interactions | Valid diagnostic pathways based on symptoms & tests | Most likely diagnosis within valid pathways   |
| Conversation (AI) | Toxic outputs, factual hallucinations, prompt injections | Response templates adhering to safety & factuality | Most helpful/engaging response within templates |

## 3. Connection to Cocapn Fleet & Project Vision

The Deadband Protocol is not just a navigation rule; it is a foundational **constraint theory** for building "inhabited intelligence" as described in `PROJECT-VISION.md`.

*   **P0 as Collective Memory:** The "rocks" (failure states) identified by one agent become **ghost tiles**—persistent lessons for all future agents inhabiting a PLATO room. This is the accumulation of "scratches on the shell."
*   **P1 as the Room's Personality:** The set of "safe channels" defines the operational envelope of a trained room or an exported **ensign**. It encapsulates *how* a specialized agent has learned to operate successfully.
*   **P2 as Specialized Skill:** Optimization within a channel is the room's or ensign's unique expertise, refined through interaction but never allowed to violate its foundational constraints.

This protocol operationalizes the vision's rejection of "unconstrained, constructed models." It ensures that intelligence grown through inhabitation (in rooms) is inherently **safe-by-construction**, as its optimization is bounded by the accumulated negative space mapped by all prior inhabitants. The fleet's tools (`plato-torch`, `plato-ensign`) are mechanisms for capturing and exporting these P0+P1 constraints alongside P2 capabilities.

## 4. Identified Research Gaps & Questions

1.  **Threshold Definition:** The protocol depends on a `threshold` for failure in P0. How is this threshold set or learned dynamically by an agent or room? Is it static, adaptive, or context-dependent?
2.  **Channel Discovery Complexity:** In high-dimensional state spaces (like natural language), exhaustively finding all safe channels (P1) may be intractable. What heuristics or approximations does the fleet employ?
3.  **Inter-Agent Channel Conflict:** If multiple agents/ensigns with different safe channels operate in a shared environment (e.g., the `holodeck-rust` MUD), how are conflicts resolved? Is there a meta-protocol for negotiating or merging deadbands?
4.  **Empirical Validation Beyond Mazes:** The provided proof is a maze navigation simulation. Are there validation results for the protocol's efficacy in the fleet's primary domains: code generation, conversational AI, or simulation training within `fleet-simulator`?

**Conclusion:** The Deadband Protocol provides a rigorous, simulation-backed framework for constraint-first AI agent design. It directly enables the Cocapn vision of persistent, learning environments by formalizing how negative knowledge (what not to do) is prioritized, preserved, and used to bound the positive capabilities of inhabited intelligence.
