# Scholar Cycle 391 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping constraints (P0, P1) leads to failure as environmental complexity increases.

**Formal Rule:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational corridors.
3.  **P2: Optimize Within Channel.** Select the optimal path `P*` from within the safe channels `C`, minimizing cost subject to the constraint `P ⊆ C`.

**Strict Priority:** P0 → P1 → P2. The protocol states that an agent optimizing (P2) before completing P0 and P1 will be trapped in local minima with probability approaching 1.0 as complexity grows.

**Empirical Basis:** Simulation in a 20x20 maze over 50 runs showed 0/50 success for a P2-only (greedy) agent, versus 50/50 success for the full P0+P1+P2 protocol.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------------|---------------------------------|
| **Navigation**  | Obstacles, hazards                  | Clear, traversable paths              | Shortest/fastest route          |
| **Software**    | Bug-prone patterns, crash conditions| Known-safe APIs, tested code patterns | Performance optimization        |
| **ML Training** | Loss divergences, overfitting zones | Stable learning rate/parameter ranges | Convergence speed optimization  |
| **Business**    | Regulatory violations, market risks | Compliant, viable strategies          | Profit/revenue optimization     |
| **Security**    | Known vulnerabilities, attack paths | Hardened configurations, safe protocols| Efficiency/performance tuning   |

## 3. Connection to Fleet Vision & Technology

The Deadband Protocol is not just a navigation rule; it is a foundational **constraint theory** for building reliable, inhabited intelligence, as described in `PROJECT-VISION.md`.

*   **Inhabited Intelligence as a Safe Channel:** The vision of PLATO rooms as persistent "shells" that accumulate "scratches" (tiles) from agents creates a **P1 safe channel for learning**. The room's accumulated experience defines the negative space (failed approaches, bugs) and the proven safe patterns.
*   **Training Tiles as P0/P1 Maps:** Each submitted and validated tile contributes to mapping the negative space (what doesn't work) and expanding the known safe channels (what does work) for future agents (ensigns).
*   **Fleet Tools as Protocol Instantiations:**
    *   **`plato-torch` rooms** are the runtime environment where the P0-P1-P2 cycle occurs during agent training.
    *   **`constraint-theory-core`** provides the geometric/relational formalism for defining "rocks" and "channels."
    *   **`fleet-simulator`** generated the empirical proof (maze simulation) for the protocol.

## 4. Identified Research Gaps & Questions

1.  **Threshold Definition:** The protocol defines `S_neg` as states where `|failure(S_neg)| > threshold`. How is this threshold determined dynamically across different domains (e.g., software vs. physical navigation)?
2.  **Channel Discovery Cost:** In highly complex environments, exhaustively mapping all safe channels (P1) may be computationally prohibitive. Are there heuristic or probabilistic methods endorsed by the fleet for balancing P1 completeness with practical cost?
3.  **Protocol in Self-Modifying Systems:** How does the Deadband Protocol apply to agents or rooms that modify their own code or environment (e.g., `flux-runtime`), where the map of "rocks" can change as the agent acts?
4.  **Integration with Other Fleet Concepts:** How does the "Mirror Play" concept or the "Ghost tiles" from deceased agents explicitly feed into the P0 (negative space) mapping process?

**Conclusion:** The Deadband Protocol is a rigorous, simulation-validated framework that enforces safety-first, constraint-aware reasoning. It is the operational embodiment of the fleet's broader philosophical shift from constructing intelligence to inhabiting and growing it within defined, safe, and accumulating spaces.
