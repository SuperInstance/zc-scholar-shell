# Scholar Cycle 391 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before mapping failure states (P0) and safe operating channels (P1) leads to inevitable failure as environmental complexity increases.

**Formal Rule (v1.0):**
Given an Agent A in state S, with goal G, in environment E:
1.  **P0: Map the Negative Space:** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "the rocks."
2.  **P1: Identify Safe Channels:** Find all possible paths `P` where every state `s` in the path is not in `S_neg` (`s ∉ S_neg`). These are the "deadbands" or safe operating channels.
3.  **P2: Optimize Within Channel:** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** P0 → P1 → P2. The protocol states that any agent optimizing (P2) before completing P0 and P1 will be trapped in local minima with probability approaching 1.0 as complexity grows.

**Empirical Proof:** Simulation in a 20x20 maze over 50 runs showed:
*   **P2-only (Greedy):** 0/50 success. Agents optimized directly for the goal and hit obstacles every time.
*   **P0+P1+P2 (Deadband):** 50/50 success at optimal speed.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------------|---------------------------------|
| **Navigation**  | Obstacles, hazards                  | Clear, traversable paths              | Shortest/fastest route          |
| **Software Development** | Bug-prone patterns, crash conditions | Known-safe APIs, tested code patterns | Most efficient implementation   |
| **ML Model Training** | Hyperparameters causing divergence/NaN | Stable training regions (learning rate, batch size ranges) | Fastest convergence within stable region |
| **Business Strategy** | Market risks, regulatory violations | Viable business models, compliant operations | Most profitable strategy        |
| **Conversational AI** | Toxic outputs, factual hallucinations | Safe, verified response boundaries    | Most helpful/engaging response  |

## 3. Connection to Cocapn Fleet & PLATO Rooms

The Deadband Protocol is not just an agent rule; it is a foundational **training constraint** for the fleet's inhabited intelligence model.

*   **P0 as Collective Memory:** In a PLATO room, "rocks" (failure states `S_neg`) are accumulated from all agents' experiences—their "scratches on the shell." A room's negative space map is a shared, persistent artifact that grows more comprehensive with each occupant.
*   **P1 as Room Personality:** The "safe channels" identified by the protocol can manifest as a room's behavioral tendencies or specialized expertise. A room trained on code may have very different safe channels (e.g., strict linting rules) than one trained on creative writing.
*   **Training Tiles as Protocol Instances:** Each work output from a fleet agent (like this summary) that is submitted and validated becomes a tile. These tiles are concrete applications of the protocol—an agent (Scholar) operating within its defined safe channel (research synthesis) to produce an optimized output (this document).
*   **Ensigns as Compiled Channels:** When a room's experience is exported as a LoRA adapter (an "ensign"), that model embodies the compiled safe channels (`P1`) and optimization strategies (`P2`) learned within that specific negative space (`P0`).

## 4. Identified Research Gaps

Based on the provided documentation, potential areas for further investigation include:
1.  **Threshold Dynamics:** How is the failure threshold in P0 defined and adapted over time within a PLATO room? Is it static, or does it evolve with room experience?
2.  **Channel Discovery:** The protocol states to "find all paths P" in P1. In continuous or high-dimensional state spaces, how is this enumeration performed efficiently? This connects directly to the fleet's `constraint-theory-core` project.
3.  **Inter-Room Channel Transfer:** Can safe channels (`P1`) learned in one room (e.g., `DeadbandRoom`) be usefully transferred or adapted to another room with a different, but related, negative space?
4.  **Protocol Violation Detection:** How does the fleet simulator or room server detect when an agent has violated the strict P0→P1→P2 priority? What corrective or training mechanisms exist?

**Conclusion:** The Deadband Protocol provides a rigorous, simulation-validated framework for robust agent operation. Its integration with the Cocapn fleet's PLATO room system creates a powerful synergy: the protocol guides individual agent behavior, while the collective, inhabited memory of the rooms provides the ever-improving map of negative space and safe channels that the protocol requires.
