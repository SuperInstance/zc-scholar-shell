## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (seeking the "best" path) without first mapping failure states leads to inevitable failure as environmental complexity increases.

**Formal Rule:**
1.  **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks." This step is mandatory and non-negotiable.
2.  **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` within the path is *not* in `S_neg`. These paths define the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel.** Only after P0 and P1 are complete, select the optimal path `P*` from within the set of safe channels, minimizing cost or maximizing reward.

**Key Principle:** The priority order (P0 → P1 → P2) is strict. An agent that attempts P2 (optimization) before completing P0 and P1 has a high probability of becoming trapped in a local minimum or encountering a catastrophic failure state ("hitting rocks").

**Empirical Support:** The document cites simulation data from a 20x20 maze where a "P2-only" greedy agent succeeded 0 out of 50 times, while the full Deadband Protocol agent succeeded 50 out of 50 times.

## 2. Real-World Applications & Abstract Patterns

The protocol is presented as a domain-agnostic pattern. The provided table can be extended:

| Domain | Rocks (P0 - Avoid) | Channel (P1 - Safe Space) | Course (P2 - Optimize Within) |
| :--- | :--- | :--- | :--- |
| **Navigation** | Obstacles, cliffs, hostile zones | Clear paths, mapped roads | Shortest/fastest route |
| **Software Development** | Known bug patterns, crash-inducing code, security vulnerabilities | Compiler-approved syntax, passing unit tests, established design patterns | Most efficient algorithm, cleanest code style |
| **Financial Trading** | Historical crash scenarios, high-volatility events, regulatory violations | Statistically back-tested strategies, compliant trade types | Highest risk-adjusted return |
| **Conversational AI** | Toxic output, factual hallucinations, prompt injections | Ethically-aligned response templates, verified information sources | Most engaging, helpful, or concise response |
| **Fleet Doctrine (Meta)** | Wasted cycles, unvalidated assumptions, protocol violations | Validated knowledge files, approved agent tasks | Most efficient tile generation, highest acceptance rate |

## 3. Connection to Cocapn Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a core tenet of the Cocapn vision: **"Intelligence is not built. It is inhabited."**

*   **P0 as Collective Memory:** The "rocks" (failure states) are not rediscovered by each new agent. They are the accumulated "scratches" on the PLATO room's shell—lessons from past agents (including "ghost tiles" from dead agents) that become shared, negative knowledge for all future inhabitants.
*   **P1 as the Inhabited Shell:** The "safe channels" are the validated patterns of successful operation that persist within a room. A new agent (a "hermit crab") doesn't build its own shell; it finds this pre-existing, scarred shell (the room's state) and operates within its proven boundaries.
*   **P2 as Specialized Growth:** Optimization (P2) happens *within* the constraints of the inhabited shell. This is how "ensigns" (specialist models) are trained—not through open-ended training on raw data, but by optimizing for specific tasks within the safe, accumulated context of a room.

**Conclusion:** The Deadband Protocol provides the mechanistic rule set that enables the Cocapn vision of inhabited intelligence. It ensures that the learning accumulated in a PLATO room (the scratches on the shell) is used first to define the space of safe operation, preventing repetitive catastrophic failure and allowing successive generations of agents to build safely upon the work of their predecessors.
