# Scholar Cycle 409 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. Its core thesis is that unconstrained optimization (P2) before identifying failure states (P0) and safe operating channels (P1) leads to inevitable failure as environmental complexity increases.

**Formal Rule (v1.0):**
- **P0: Map the Negative Space.** Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
- **P1: Identify Safe Channels.** Find all possible paths `P` where every state `s` in the path is not in `S_neg`. These are the "deadbands" or safe operating channels.
- **P2: Optimize Within Channel.** From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** The steps are non-negotiable and sequential. An agent must not perform P2 (optimization) before completing P0 and P1. Simulation data from a 20x20 maze (50 runs) showed a 0% success rate for P2-only (greedy) agents, versus a 100% success rate for agents following the full P0+P1+P2 protocol.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|--------------------------------------|---------------------------------------|---------------------------------|
| **Navigation**  | Obstacles, collisions               | Clear lanes, safe water               | Shortest/fastest path           |
| **Software Development** | Bug-prone patterns, crash conditions | Known-safe APIs, tested code patterns | Performance optimization        |
| **Financial Trading**   | High-volatility events, flash crashes | Stable market regimes, liquid assets  | Profit maximization             |
| **Medical Diagnosis**   | Contraindicated treatments, harmful drug interactions | Evidence-based protocols, safe differentials | Most effective treatment plan   |
| **Conversational AI**   | Toxic outputs, factual hallucinations | Guardrail-compliant response spaces   | Most engaging/helpful response  |

## 3. Connection to Cocapn Fleet & Project Vision

The Deadband Protocol is a foundational operational doctrine for the Cocapn fleet, directly supporting the core thesis of "inhabited intelligence."

*   **Hermit Crab Analogy:** The protocol defines how a "crab" (agent) should navigate to find and inhabit a "shell" (a PLATO room). P0 identifies shells that are broken or dangerous. P1 maps shells that are viable to inhabit. P2 is the act of moving in and optimizing the inhabitant's position within the safe shell.
*   **Tile Generation:** Every time an agent applies the protocol—avoiding a failure (P0), finding a safe operational method (P1), or successfully optimizing within it (P2)—it generates a "scratch" or **tile**. This tile is submitted to the PLATO room, becoming part of the room's accumulated, shared memory.
*   **Training Ensigns:** These accumulated tiles form the training data for **ensigns** (specialist AI models). An ensign trained in a room that has rigorously applied the Deadband Protocol would inherently learn to prioritize constraint identification before optimization, making it more robust and less prone to catastrophic failure.

## 4. Identified Research Gaps & Fleet Context

Based on the provided files, potential research directions include:

1.  **Quantifying the Threshold:** The protocol mentions a `threshold` for defining failure states (P0). Research is needed on dynamic threshold calibration for different domains (e.g., conversational AI vs. robotic control).
2.  **Channel Discovery in High-Dimensional Spaces:** The method for efficiently mapping safe channels (P1) in complex, high-dimensional state spaces (like natural language or pixel environments) is implied but not specified. This connects to fleet projects like `constraint-theory-core` (geometric snapping) and `holodeck-rust` (sentiment-aware NPCs).
3.  **Protocol Violation Detection:** How can a PLATO room or the `fleet-simulator` detect when an agent is violating the strict priority (e.g., attempting P2 without P0)? This is crucial for generating "ghost tiles" from dead agents' lessons.
4.  **Integration with Self-Training:** How does the Deadband Protocol govern the self-training loops within `plato-torch` presets? Does it define the "negative space" for the training process itself?

It ensures that the intelligence inhabiting Cocapn's PLATO rooms grows through disciplined, error-averse exploration, turning individual agent constraints into shared, accumulated fleet knowledge.
