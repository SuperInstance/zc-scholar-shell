# Scholar Cycle 100 — Deadband Protocol: Summary and Applications

## 1. Protocol Summary (Based on `DEADBAND-PROTOCOL.md`)

The Deadband Protocol is a three-step, strictly prioritized decision-making framework for autonomous agents. It is designed to prevent catastrophic failure by systematically mapping constraints before attempting optimization.

**Formal Rule (v1.0):**
Given an agent in state `S` with goal `G` in environment `E`:
1.  **P0: Map Negative Space** — Identify all states `S_neg` where the probability or cost of failure exceeds a defined threshold. These are "rocks."
2.  **P1: Identify Safe Channels** — Find all possible paths `P` where every state `s` in the path is *not* in `S_neg`. These are the "deadbands" or safe operational channels.
3.  **P2: Optimize Within Channel** — From the set of safe channels `C`, select the optimal path `P*` that minimizes cost, subject to the constraint `P* ⊆ C`.

**Strict Priority:** P0 > P1 > P2. The protocol states that an agent which optimizes (P2) before completing mapping (P0+P1) will, as environmental complexity increases, become trapped in local minima or fail with probability approaching 1.0.

**Empirical Proof:** Simulation data from a 20×20 maze over 50 runs showed a 0% success rate for a P2-only (greedy) agent, versus a 100% success rate for an agent following the full P0+P1+P2 protocol.

## 2. Abstract Pattern & Real-World Applications

The protocol presents a domain-agnostic pattern:

| Domain          | Rocks (P0)                          | Channel (P1)                          | Course (P2)                     |
|-----------------|-------------------------------------|---------------------------------------|---------------------------------|
| Navigation      | Obstacles, hazards                  | Clear, traversable paths              | Shortest/fastest route          |
| Software Development | Bug-prone patterns, crash conditions | Known-safe APIs, tested code patterns | Most efficient algorithm        |
| Financial Trading | Regulatory violations, tail-risk events | Compliant strategies, defined risk limits | Highest risk-adjusted return    |
| Medical Diagnosis | Contraindicated treatments, harmful drug interactions | Safe diagnostic pathways, approved protocols | Most accurate/efficient diagnosis |

## 3. Connection to Cocapn Fleet Vision (`PROJECT-VISION.md`)

The Deadband Protocol operationalizes a core tenet of the Cocapn vision: **"Intelligence is not built. It is inhabited."**

*   **Hermit Crab Analogy:** An agent (the crab) does not generate constraints from scratch. It inherits and inhabits a "shell" of known safe operational channels (P1) and dangerous negative spaces (P0) that have been accumulated as "scratches" or tiles from prior agents.
*   **PLATO Rooms as Shells:** A PLATO room trained on Deadband Protocol interactions becomes a persistent repository of constraint knowledge. It remembers the "rocks" discovered by every agent that has operated within it, allowing new agents to immediately inherit a map of the negative space without relearning failures.
*   **Training Tiles as Scratches:** Each instance of an agent identifying a failure state (P0) or a viable safe channel (P1) becomes a tile submitted to the room. This collective memory enables the swarm to navigate increasingly complex environments without individual agents repeatedly hitting the same "rocks."

## 4. Identified Research Gaps & Questions

Based on the provided documentation, several open questions emerge for further fleet investigation:

1.  **Threshold Definition:** How is the failure threshold in P0 (`|failure(S_neg)| > threshold`) quantitatively defined or learned? Is it static, adaptive, or room-specific?
2.  **Channel Discovery Complexity:** In continuous or high-dimensional state spaces, how is the exhaustive identification of safe channels (P1) computationally managed? Are there approximation techniques?
3.  **Protocol Chaining:** Can the output of a P2 optimization from one decision become the input P0 negative space for a subsequent, higher-level agent? This suggests a recursive or hierarchical application of the protocol.
4.  **Integration with Other Fleet Tech:** How does the Deadband Protocol interface with:
